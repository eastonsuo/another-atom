# 【产品设计】Approval 审批式与对话式交互评审

## 1. 背景与问题

当前 V1 设计把 **Blueprint Approval（审批确认）** 作为核心交互框架：用户提交 Prompt 后，Product Manager 生成 Blueprint，用户必须显式确认才能进入构建。审批被绑定在“提交 / 开始构建”这一个入口上，后续阶段按固定顺序自动接力，中途不再插入确认。

评审提出两个核心质疑：

1. 把 approval 作为主流程门，是否让产品退化成“填表—确认—等待”，而不是对话式协作？
2. 审批是否只应出现在提交时？后续消耗大量配额、覆盖线上版本、删除数据等动作同样需要确认。

结论方向：**应以对话式为主框架，approval 从流程主干降级为风险驱动的即时插入，并覆盖后续任何高风险动作。**

## 2. 两种交互形态

### 2.1 审批式的价值与代价

**价值**

- 资源门控清晰：花构建资源前先对齐目标。
- 可追溯：每次确认都是明确决策节点。
- 范围收敛：可承接 supported / adapted / unsupported 判定。

**代价**

- 主框架僵硬：体验偏表单确认和等待进度条。
- 审批点单一：后续高风险动作没有再次确认机会。
- 风险与确认错配：低风险和高风险动作被同一个入口审批一刀切处理。

### 2.2 对话式的价值与代价

**价值**

- 自然协作：用户可持续用自然语言调整需求。
- 渐进式：可以边聊边改，不必一次填对全部字段。
- 确认可内联：作为对话中的待办卡片出现。

**代价**

- 纯对话容易让模型执行未经授权的高风险动作。
- 若不落结构化产物，可追溯性和可验证性会下降。

综合判断：**对话为主干，风险驱动的即时确认为护栏。**

## 3. 核心改进建议

### 3.1 Approval 降级为内联护栏

- 主流程以对话和工作流为主线。
- 保留 Blueprint 等结构化、可编辑、可追溯产物。
- Approval 不再默认卡住所有流程，只在命中风险时出现。

### 3.2 从入口驱动改为风险驱动

任何步骤命中以下动作时当场确认：

- 覆盖写或批量修改已有代码和文件。
- 批量、循环或明显放大的 LLM 配额消耗。
- Publish、Update、Always Latest 覆盖线上内容。
- 删除项目或数据、Restore、重置等不可逆或高影响操作。
- 引入第三方服务、授权或写入密钥。

### 3.3 总体计划确认不等于后续动作均获授权

执行中发现风险超出原计划预期时，应插入 `approval.required` 并暂停当前步骤，而不是依赖开头的一次确认继续执行。

### 3.4 低风险步骤静默直通

读文件、分析、生成报告、预览、切换视图和只读版本对比等可逆且不产生外部影响的操作，不应插入确认。

### 3.5 分级确认与记忆降噪

- 轻确认：内联提示并一键继续，例如说明本步预计配额。
- 重确认：删除、覆盖线上和 Unpublish 等动作明确展示后果并二次确认。
- 低/中风险偏好可以记忆；高风险不可逆动作不能永久静音。

## 4. 分场景推荐策略

| 场景 | 风险特征 | 推荐交互 |
| --- | --- | --- |
| 读文件、分析、报告、预览、只读对比 | 可逆、不花钱、不改线上 | 静默直通，仅展示产物 |
| 单次 LLM、修改 Blueprint、保存新版本 | 少量配额、可回滚 | 轻确认或按用户偏好直通 |
| 覆盖线上、发布、删除、Restore、批量调用、第三方授权 | 成本高、不可逆或影响公开内容 | 当场重确认，不可永久静音 |
| unsupported、adapted、缺关键字段 | 目标未对齐 | 对话澄清；展示映射和舍弃项后确认 |

## 5. 小结

- 保留 Blueprint、范围三态和状态追溯。
- 将 approval 从单一入口闸门改为可出现在任何步骤的风险护栏。
- 低风险静默，高风险即时确认，以分级确认平衡安全和打断成本。
