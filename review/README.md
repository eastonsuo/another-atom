# Review 文件夹规范

## 作用

`review/` 用于记录围绕 Another Atom 的产品设计、工程设计、阶段总结和缺陷排查。它不是正式需求或架构的替代品：评审负责提出判断、证据、取舍和待办；已经确认并稳定的结论再同步到 PRD、架构设计、Agent 设计或 README。

## 基本规则

1. **只用中文。** 必要的技术名词可保留英文括号，例如 Project、Git commit、Sandbox。
2. **标题必须带类型标签。** 格式为 `# 【类型】主题`；有两个同等重要的维度时可写 `# 【类型】【类型】主题`，不超过两个标签。
3. **文件名使用日期加英文短横线主题。** 格式为 `YYYY-MM-DD-主题.md`。日期表示首次创建日；后续状态变化写入文内 Update，不通过改文件名表达。
4. **首屏先给状态。** 标题下紧接一行 `> 类型：…｜状态：…｜日期：…｜范围：…`，让读者不必通读全文才知道这是历史评审、进行中讨论还是已关闭问题。
5. **Update 放在正文前。** 旧评审被后续实现改变时，在元信息后、原正文前新增 `## Update（YYYY-MM-DD，基于…）`。Update 必须逐项说明“已修复 / 部分修复 / 未修复”，并给出代码、测试或部署证据；不改写历史发现。

## 类型标签

| 标签 | 适用内容 |
| --- | --- |
| `【产品设计】` | 用户流程、交互、审批、体验、产品边界与价值判断。 |
| `【工程设计】` | 架构、数据模型、运行时、安全、部署、可靠性与技术取舍。 |
| `【Bug】` | 可复现缺陷、风险、回归、验证缺口及其修复状态。 |
| `【总结】` | 阶段实现盘点、验收结果、里程碑结论与后续优先级。 |

## 新文档模板

```markdown
# 【产品设计】主题

> 类型：产品设计｜状态：讨论中｜日期：YYYY-MM-DD｜范围：一句话说明
> 关联：PRD / 架构 / 代码路径（如有）

## 结论

先给当前判断和适用边界。

## 证据与分析

只列支撑结论的产品事实、代码事实或测试事实。

## 建议与优先级

按 P0 / P1 / 后置说明行动、收益和代价。
```
