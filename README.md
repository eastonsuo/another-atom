# Another Atom

[English](./README.md) | [简体中文](./README.zh-CN.md)

> A cloud-hosted AI application builder that converts product intent into a validated Blueprint, a controlled AppSpec, and a runnable web application.

Another Atom is an independent, Atoms-inspired product demo. It combines real LLM reasoning with a deterministic React renderer so users can review what will be built before the platform generates, validates, versions, and publishes the result.

The project is a standalone public product and engineering case study. It does not depend on Atoms source code or internal infrastructure.

## V1 Workflow

```text
Prompt -> Editable Blueprint -> User Approval -> Validated AppSpec
       -> Async Build -> Interactive Preview -> Revision -> Version
       -> Explicit Publish -> Public URL
```

V1 has one delivery and execution path: a Railway-hosted Cloud Demo accessed through React Visual Studio.

```text
React Visual Studio
        |
        | REST + SSE
        v
FastAPI / LLM Orchestrator
        |
        +---- PostgreSQL: users, sessions, quota, jobs, versions
        |
        `---- Build Worker -> Fixed React Template -> Preview/Publish
```

## Key Decisions

- **Real LLM:** Blueprint and AppSpec are generated by a real model and validated with Pydantic.
- **Controlled execution:** the model cannot install dependencies, change build commands, or execute arbitrary shell input.
- **Deterministic renderer:** AppSpec is materialized through a fixed React template with dependencies preinstalled in the Docker image.
- **Asynchronous builds:** persistent build jobs run outside the HTTP request lifecycle with an initial concurrency limit of one.
- **Transactional quota:** every model call reserves quota before execution and settles actual usage afterward.
- **Explicit publishing:** the agent produces a validated preview version; only the user can publish a selected version.

## Innovation

The primary design contribution is not the use of an LLM itself. Another Atom turns an opaque chat-generation process into an inspectable production contract:

```text
User intent
    -> Blueprint: editable product contract
    -> AppSpec: machine-validated application contract
    -> Renderer: deterministic execution boundary
    -> ProjectVersion: inspectable and recoverable result
```

Planner, Designer, Engineer, and QA are visible stages of one controlled workflow, not a claim that multiple agents are running in parallel. Each stage must produce an artifact that reviewers can inspect.

## V1 Scope

Included:

- React Visual Studio and Prompt Composer.
- Real LLM calls and structured output validation.
- Blueprint approval, AppSpec generation, and controlled React rendering.
- Persistent projects, multiple sessions, attachments, runs, events, and versions.
- Plans, quota reservation, usage settlement, and account-level limits.
- Asynchronous build, preview, revision, Resolve, Restore, and Publish.
- Docker deployment on Railway with PostgreSQL and a persistent volume.

Not included:

- Terminal CLI or local repository execution.
- SQLite and local-to-cloud project synchronization.
- Runtime dependency installation or arbitrary code execution.
- A model selector or true multi-agent parallelism.
- Generated-app authentication, database, commerce, or payment systems.
- Stripe billing, wallet, top-up, or invoicing.

## Local Agent Direction

A Claude Code-like local runtime remains the highest-priority extension after the Cloud Demo:

```text
Terminal CLI -> Local Agent -> Local files / Git / shell / npm
                         -> localhost Visual Studio
                         -> Cloud auth / quota / LLM gateway
```

This capability is not implemented in V1 and is not part of the current acceptance criteria.
