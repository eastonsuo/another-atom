## 2026-07-15 受控动态源码 Context 设计 Update

Patch（代码补丁）实现前，V1 目标设计由“单次静态源码 Context（源码上下文）后直接生成 Patch”调整为“完整基线源码清单 + 简化仓库地图 + 有界动态读取 + 最终 Patch”。Runtime（运行系统）仍掌握基线 Git commit（Git 提交）下的完整受控源码；Engineer（工程师智能体）先接收初始源码，信息不足时通过 `NeedContext`（需要更多上下文）申请列出、搜索或读取源码，最多补充两轮，随后返回 `SourcePatchSet`（源码补丁集）或明确无法继续。Runtime 在隔离候选工作区应用 Patch，并根据真实候选源码重新计算 `SourceDiff`（源码差异）。

本 Update 替代下方 2026-07-14“目标不引入检索或多轮 Patch”和“Engineer 必须一次接收全量受控源码”的容量决定，但不改写当日实现事实：固定 `MAX_SOURCE_CHARS` 的静态装箱已经落地，继续作为迁移起点。新目标 Contract、读取权限、轮次、累计预算、恢复和验收要求以[受控动态源码 Context 与 Patch 执行](../../design/V1/技术设计/09-[Agent][TODO]-受控动态源码Context与Patch执行.md)为准；相关代码与部署证据尚未完成，因此本 Review 继续留在`待办`。

## 2026-07-14 Context 容量决定更新

Engineer Context 的容量处理收敛为固定 `MAX_SOURCE_CHARS`：完整有效文档链不裁剪；源码未超预算时全量发送，超出时按“选中文件、消息中的准确相对路径、其余路径升序”装入完整文件，并记录字符预算、实际字符数以及包含/省略清单。不计算 tokenizer，也不引入 Planner、检索、依赖分析、分包或多轮 Patch。本更新替代下文“超限明确停止”的原处理要求；其他 Review 发现不变。

## 2026-07-14 源码 Context 实现 Update

`MAX_SOURCE_CHARS` 最小纵切已经落地：Runtime 从基线 commit 枚举受控源码，按完整文件确定性装箱，持久化 `source_context` Artifact 和事件，并将完整 ProductSpec、有效 Contract 与实际包含源码传给 Mock/Ollama/DeepSeek Engineer Provider。后续 `ai_edit` 版本会沿 `base_version_id` 读取并复制有效 ProductSpec；消息 API 已接受 `selected_files`，Studio 尚未提交该字段。

验证证据：源码装箱单元测试覆盖“选中文件 → 消息准确路径 → 其余路径升序”、大文件跳过后继续装入小文件；Project Chat 集成测试验证 SourceContext Artifact、全量未裁剪路径和连续多轮修改；`.venv/bin/ruff check another_atom tests` 与 `.venv/bin/pytest -q` 通过。当前 Engineer 仍返回完整 AppSpec，SourcePatchSet 与隔离 apply 未实现；消息先建 Run、Lead Project Context 和执行前 Approval 也未修复，因此本 Review 继续留在`待办`。

## 2026-07-14 Project 对话路由与修改授权实现 Update

本次实现已关闭“普通对话先建修改 Run”和“Lead 没有 Project Context”两个 P0 问题，并完成可用的执行前授权纵切：

- `POST /api/projects/{project_id}/messages` 现在只持久化消息、组装 Context 并调用独立 `ProjectLeadDecision(answer|clarify|propose_change)`；answer/clarify 不创建 Run、BuildJob、ProjectVersion 或写占用；
- Project Lead 收到当前完整 ProductSpec、Blueprint、ArchitectureSpec、AppSpec 元数据、当前 Project 对话、最近失败，以及按既有 `MAX_SOURCE_CHARS` 规则从基线 commit 装箱的实际源码；消息表只保存 Context hash、有效文档类型、对话条数和源码包含/省略清单，不复制整包源码；
- propose_change 持久化为 `ProjectMessage(message_type=change_proposal, payload.status=pending)`，Studio 默认按钮改为“发送”，仅在任务卡上显示独立“修改代码”按钮；
- 新批准端点在点击后重新检查 base version，原子创建 Run/BuildJob/写占用；重复批准返回同一 Run，基线变化将 proposal 标为 stale，并发对话仍允许但第二个代码执行不能取得写锁；
- `_execute_change_run` 删除了第二次 Lead 路由，避免批准后重复判断或把已授权修改重新走成 direct。

这里没有把专用 proposal 冒充为完整通用 Approval：现有 HumanTask 强制绑定 `run_id`，无法表达“批准前不建 Run”的对象，因此本纵切先把 pending/approved/stale 放在 ProjectMessage payload，并使用专用 approve API。通用 Approval subject、reject/cancel、ConversationJob/lease、消息幂等键、批准前完整 ChangeBrief、SourcePatchSet 和隔离 apply 仍未实现，本 Review 保持`待办`。

验证覆盖：Project 问答引用当前颜色和源码 Context 且写入计数不变；澄清后再提案；批准前无 Run/Job/版本/锁；批准幂等；旧提案 stale；对话可并发但执行单写；批准后 PM 澄清与 Worker 恢复；失败保留基线并可继续；双用户隔离；Ollama 请求实际包含批准文档与源码。`.venv/bin/ruff check another_atom tests`、`.venv/bin/pytest -q` 与 `cd studio && npm run build` 均通过；前端构建只有既有的大 chunk 提示，没有编译失败。

## 背景

用户在已完成 Project 中继续与 Lead 对话时，普通讨论仍创建 `ai_edit` Run 并投影为整条团队流水线；Lead 的直接回答没有取得当前 Project 与历史对话 Context；明确修改也缺少“查看修改任务并确认修改代码”的 Human-in-the-loop 门禁。结果是问答与执行边界不可信：用户不知道一次输入是在聊天、准备修改，还是已经开始改代码。

本 Review 记录实现与设计的差异，并确认本轮新的产品决定：Project Chat 默认只对话；Lead 判断需要修改代码后先形成可检查的修改任务，展示“修改代码”按钮；用户确认后才创建 `ai_edit` Run、取得写占用并启动固定团队。

## 摘要

- **[P0｜消息先执行后路由]** 当前 Project 消息先创建 `ai_edit` Run、BuildJob 和写占用，再由 Worker 判断 `direct/team`；普通问答因此也表现为一次流水线。
- **[P0｜Project Context 缺失]** Lead 路由和直接回答只接收当前消息；Engineer 主要接收数据库 AppSpec，完整已批准文档链和实际源码没有作为修改输入。
- **[P1｜修改授权缺失]** 当前 `team` 路由直接进入完整代码修改流水线，没有“修改任务待确认”Human Task，也没有“修改代码”按钮。
- **[P1｜状态投影失真]** direct Run 被标为 `completed` 后，前端把全部角色显示为已完成，掩盖实际只执行了 Lead。
- **[设计处理]** direct 不建 Run 的设计原本正确；“普通修改自动继续”的旧结论与本轮决定冲突，已同步修订相关产品和技术设计。

## 1. 检查结论

当前问题不是一句 Prompt 能修正，而是消息、路由、授权和执行四层顺序错误：

```text
当前实现
Project 消息
  -> 立即创建 ai_edit Run / BuildJob / Project 写占用
  -> Worker 调用 Lead 路由
       |-- direct -> 回答并把 Run 标记 completed
       `-- team   -> 直接执行完整修改流水线
```

目标顺序应为：

```text
Project 消息
  -> 持久化对话消息
  -> 组装 ProjectContextSnapshot
  -> Lead：answer | clarify | propose_change
       |-- answer / clarify -> 写回 Chat，不创建 Run
       `-- propose_change   -> 展示修改任务 + “修改代码”按钮
                                  |
                              用户确认
                                  |
                     创建 ai_edit Run / BuildJob / 写占用
                                  |
                              固定团队执行
```

## 2. P0 发现

### 2.1 Project Chat 在意图判断前创建执行 Run

`ProjectChatPanel.submitChange` 对非补充输入统一调用 `createProjectChange`。`POST /api/projects/{project_id}/messages` 在 Lead 判断前创建 `trigger=ai_edit` 的 Run、BuildJob，并尝试设置 `active_write_run_id`；随后 Worker 的 `_execute_change_run` 才调用 `route_message`。

这意味着聊天和修改共用的不是“消息入口”，而是“先创建修改任务，再允许 Lead 取消修改”。direct 虽然不创建新版本，但已经产生 Run、Job、写占用、事件和流水线 UI，违背默认对话语义。

**处理要求：** Project 消息先进入独立的持久化路由状态；`answer`、`clarify` 和未确认的 `propose_change` 均不得创建 `ai_edit` Run、BuildJob 或取得 Project 写占用。

### 2.2 Lead 没有收到正确的 Project Context

项目工作区文案声称 Lead 会检查当前 Project 代码，但 `_execute_change_run` 的路由调用只传入 `effective_prompt`。当前 Project 的产品说明、Blueprint 摘要、工作版本、功能模块、源码清单和最近可见对话都没有进入 Lead 的路由与直接回答 Context。

截图中的错误回答即为可见后果：用户讨论“额外开发的、基于大模型的软件”，Lead 却把当前 Project 误解为 DeepSeek 模型自身的内置翻译能力。仅把历史消息显示在浏览器中不等于把历史传给模型。

**处理要求：** Runtime 在每条 Project 消息路由前生成并持久化 `ProjectContextSnapshot`，至少包含：

- Project 名称、产品类型与当前状态；
- 当前 ProjectVersion、Git commit 和公开版本关系；
- 已批准 ProductSpec / Blueprint 的摘要、页面、核心模块与能力边界；
- 当前 AppSpec 的用户可见功能摘要与源码文件清单；
- 最近一段有界的用户可见 Project 对话；
- 当前选中页面、文件、错误或失败摘要；
- Context hash 与裁剪说明。

Lead 默认不需要完整仓库和无限历史；只有问题涉及具体实现时再按允许的相对路径补充相关文件。Context 不足时应澄清，不能用模型自身能力替代当前产品事实。

### 2.3 Engineer 没有收到完整的有效文档链和实际源码

当前修改链路虽然从基线 Git commit 创建 `BaseSourceSnapshot`，但调用 `revise_app_spec` 时主要传入 Blueprint、ArchitectureSpec、数据库中的 `base_app_spec`、ChangeBrief 和 RequirementDelta。`BaseSourceSnapshot` 主要在修改后用于计算 SourceDiff；完整 `docs/product-spec.md` 和实际 HTML/CSS/JavaScript 内容没有作为 Engineer 的修改输入。

因此“基于现有代码修改”目前更接近“基于旧 AppSpec 重新生成候选 AppSpec，再与 Git 源码比较”，不能证明模型理解了当前真实源码和已经批准的完整产品文档。代码一旦通过 Vim、文本编辑或未来 Adapter 产生 AppSpec 无法完整表达的变化，模型可能看不到这些事实。

**处理要求：** Runtime 在用户批准修改后，以基线 commit 组装不可变 `ModificationContextPackage`：

- 当前批准的完整 ProductSpec Markdown；
- 当前 Blueprint、ArchitectureSpec、AppSpec 与能力边界；
- 本轮 ChangeBrief、RequirementDelta、保留项和验收条件；
- 基线 Git commit 下全部受控源码及文件 hash；超出 Context 上限时明确停止；
- 用户选中页面、文件、错误与最近一次相关失败摘要；
- `product_spec_hash / contract_hashes / source_manifest_hash / context_hash`。

这里的“完整文档”指当前修改的有效产品与技术 Contract，不包括 `docs/review`、无关历史日志、过期 Artifact、`.git`、Secret 或模型私有推理。不同角色仍按最小必要原则读取 Package 的不同视图；Engineer 必须得到完整有效 Contract 和基线 commit 下的全量受控源码，而不是只得到文件清单或 AppSpec。

V1 目标修改方式固定为：Engineer 返回绑定基线 hash 的结构化 Unified Diff；Runtime 先在隔离候选工作区执行 `git apply --check`，再 apply；模型没有直接写 Repository 的 Tool。apply 后 Runtime 从工作区重新计算真实 SourceDiff，不能直接把模型声明的 Diff 当成最终证据。全量代码超出 Provider Context 上限时返回 `SOURCE_CONTEXT_TOO_LARGE`，不允许静默裁剪后继续执行。

## 3. P1 发现

### 3.1 缺少代码修改的显式 Human-in-the-loop Gate

当前 `LeadDecision` 只有 `direct | team`。一旦进入 `team`，系统直接执行 Product Manager、Architect、Engineer、Data Analyst、Validator 和 Reviewer，没有“这次准备修改什么、基于哪个版本、保持什么不变”的待确认对象。

本轮产品决定要求将“识别修改意图”和“授权修改代码”拆开：Lead 只能提出 `propose_change`；Runtime 创建绑定当前基线与 ChangeBrief hash 的 pending Approval；用户点击“修改代码”后才启动修改 Run。拒绝或继续聊天都不得产生代码副作用。

普通修改与高风险修改共用这一执行前 Gate。范围变化、预算增加和能力适配应在同一张修改任务卡上增加风险说明；执行后才发现的破坏性 Diff 仍可在提交版本前触发第二个精确对象 Approval，不能用第一次授权覆盖未知影响。

### 3.2 完成状态被错误投影为全部角色已执行

Timeline 使用 Run 的 `completed` 作为所有阶段完成条件。direct 路径把临时 `ai_edit` Run 标为 completed 后，UI 将 Lead、产品经理、架构师、工程师、数据分析师和审查员全部显示为完成，实际事件只包含 Lead 回答。

**处理要求：** answer/clarify 不进入 Run Timeline；修改 Run 的阶段完成状态必须根据实际阶段事件或持久化 StageRun 判断，不能由顶层终态反推所有角色都执行过。

### 3.3 修改触发条件不稳定且用户缺少可见控制

当前 Provider Prompt 要求只有明确的 build/create/revise 等动词才路由 team。自然语言讨论中即使用户意图逐步变成修改，也可能持续判为 direct；反过来，页面按钮固定写“发送修改”，又会让普通聊天看起来已经授权执行。

**处理要求：** 输入框默认动作统一为“发送”；Lead 输出结构化 `answer | clarify | propose_change`。`propose_change` 必须展示可检查任务和独立“修改代码”按钮，用户不再依赖特定动词碰运气触发执行。

## 4. 修改顺序

1. **消息层**：Project Chat POST 只持久化消息并创建可恢复的 ConversationJob，不创建修改 Run。
2. **Context 层**：服务端先组装用于对话路由的 `ProjectContextSnapshot`；批准修改后再从有效 Contract 和基线 Git commit 组装不可变 `ModificationContextPackage`。Project owner、基线和文件引用全部由服务端重建。
3. **路由层**：新增 `ProjectLeadDecision(intent=answer|clarify|propose_change)`；direct 回答写入 Chat 后结束。
4. **提案层**：`propose_change` 生成 ChangeBrief 卡片，绑定 `base_version_id / base_git_commit / context_hash / change_brief_hash`。
5. **HITL 层**：创建 `approval_type=project_change` 的 pending Human Task；按钮文案为“修改代码”。修改提案变化后旧 Approval stale。
6. **执行层**：批准事务重新校验基线，成功后才创建 `ai_edit` Run、BuildJob、写占用和固定团队任务；Engineer 接收全量受控源码并输出结构化 Diff。
7. **Apply 层**：Runtime 在隔离候选工作区检查并 apply Diff，重新计算真实 SourceDiff；任何 apply 冲突、越权路径或基线 hash 不一致都停止执行。
8. **展示层**：普通问答不展示流水线；待确认只展示修改任务卡；批准后才展开真实执行阶段。
9. **结果层**：成功展示 SourceDiff、新版本与验证证据；实际 Diff 超出已批准对象时在版本提交前再次暂停。

## 5. 验收与测试要求

- 普通项目问题只新增用户消息和 Lead 回答，不新增 Run、BuildJob、ProjectVersion 或写占用。
- Lead 回答能够引用当前产品名称、已批准功能和最近对话；Context hash 可追溯，跨 Project 信息不可见。
- Engineer 输入包含完整有效 ProductSpec、Blueprint、ArchitectureSpec、AppSpec 和基线全量受控源码；修改非 AppSpec 独有的源码后仍能被下一轮正确理解。
- 全量源码超过 Provider Context 上限时明确失败，不静默裁剪、遗漏文件或声称完成修改。
- 模型 Diff 只能作用于允许的 Project 相对路径；Runtime 的 `apply --check`、实际 apply 和重算 Diff 均有测试证据。
- 明确修改请求只产生 ChangeBrief 和 pending `project_change` Approval；批准前没有 `ai_edit` Run。
- 点击“修改代码”后只创建一个 Run 和一个 BuildJob；重复批准不重复执行。
- 拒绝、取消、刷新和服务重启后，当前版本、Git 与线上版本不变。
- Approval 等待期间基线变化后 subject stale，不能基于旧版本执行。
- direct、clarify、pending approval 和 executing 四种界面状态互斥，Timeline 只展示真实执行阶段。
- 实际 Diff 扩大到未批准范围时不创建 ProjectVersion，并触发提交前确认。
