## 背景

已有 Project 的修改链路被描述为“基于旧版本修订”。检查需要区分两个不同问题：Architect 是否同步修订完整设计文档，以及 Engineer 是否已经迁移为模型产出 Patch、Runtime 本地校验并 apply。两者都涉及“增量修改”，但 Contract、执行权限和完成状态不同，不能混为一项。

## 摘要

- **[已修复｜ArchitectureDesign 同步不完整]** 原实现只让 Architect 修订 `ArchitectureSpec`，随后复制旧 `ArchitectureDesign` 并替换 `visual_tokens`。功能变更可能使组件、数据流、接口、测试策略和验收映射继续停留在旧版本。现已改为基于旧完整文档生成完整修订稿，并由 Runtime 强制对齐视觉 Token。
- **[待办｜Engineer Patch 执行链未实现]** 当前 Engineer 仍返回完整候选 `AppSpec`，Runtime 再计算 `SourceDiff`。`EngineerAction`、`SourcePatchSet`、ContextReceipt、Patch 校验、隔离 apply 和 apply 后真实 Diff 均未进入实现。
- **[边界]** Architect 返回完整新版设计文档是维护完整 Artifact Contract，不代表从零重写；Engineer 的目标则是只返回受控源码 Patch。这两个输出策略可以并存，不冲突。

## 1. 检查证据

### 1.1 修复前的 ArchitectureDesign 路径

修改 Run 先调用 `revise_architecture_spec()`，但已有 `ArchitectureDesign` 时只执行：

1. 把旧 `ArchitectureDesign` 转回 `ArchitectureDesignDraft`；
2. 用新版 `ArchitectureSpec` 替换 `visual_tokens`；
3. 重新渲染 Markdown 并保存。

这只能保证视觉 Token 一致，不能保证组件、交互、状态与数据流、接口、目录、测试和验收映射与本轮需求一致。

### 1.2 当前 Engineer 路径

`LLMProvider.revise_app_spec(...)` 的返回类型仍是完整 `AppSpec`。Orchestrator 在收到完整候选后调用 `calculate_source_diff(source_snapshot, app_spec)`，再从该 `AppSpec` 生成 `SourceBundle` 并进入 Build/Test/Validation。

代码中尚无以下执行 Contract：

- `EngineerAction = NeedContext | ProducePatch | CannotProceed`；
- `SourceContextReceipt` 与 Patch 修改权限绑定；
- `SourcePatchSet` Schema、路径和基线 hash 校验；
- 隔离候选工作区中的 `apply --check` 与 apply；
- 从 apply 后真实文件重算 `SourceDiff`。

因此当前 `SourceDiff` 只能证明“完整候选相对基线发生了什么变化”，不能证明“模型提交的 Patch 已被本地校验并应用”。

## 2. 本次设计决定

### 2.1 ArchitectureDesign

已有设计文档时，Architect 接收完整基线 `ArchitectureDesign`、当前有效 `ProductSpec`、`Blueprint`、`ChangeBrief`、`RequirementDelta` 和新版 `ArchitectureSpec`，返回完整的 `ArchitectureDesignDraft`。Prompt 明确要求：

- 基于旧文档修订，不从零设计；
- 保留本轮未改变的设计约束；
- 同步修订所有受影响的功能章节；
- `visual_tokens` 与新版 `ArchitectureSpec` 完全一致；
- 只有超出已确认产品边界时才触发产品重新确认。

Runtime 仍在持久化前强制覆盖 `visual_tokens`，避免模型输出与已校验 `ArchitectureSpec` 分叉。旧 Project 没有 `ArchitectureDesign` 时保留一次迁移生成路径。

### 2.2 Engineer

目标保持为：模型最终输出 `SourcePatchSet`，Runtime 校验后在隔离候选工作区 apply，再从真实候选源码计算 `SourceDiff`。本次不通过改名、把 Diff 包装成 Patch 或在现有完整 `AppSpec` 上增加伪 Patch 字段来提前宣称完成。

实施拆为两阶段：先按 [V1 静态源码 Context 与 Patch 执行](../../design/V1/技术设计/10-[Agent][TODO]-静态源码Context与Patch执行.md)完成一次性 Context 下的 Patch/apply 闭环，再按 [V1 受控动态源码 Context 与 Patch 执行](../../design/V1/技术设计/09-[Agent][TODO]-受控动态源码Context与Patch执行.md)补 `NeedContext` 与多轮读取。

## 3. 本次实现

- Provider Contract 新增 `revise_architecture_design()`，Mock 与 Ollama/DeepSeek 路径同时实现。
- 修改 Orchestrator 在已有设计时调用完整设计修订；缺少旧设计时才走迁移生成。
- Runtime 在 Agent 输出后继续强制对齐视觉 Token，渲染并保存新的 ArchitectureDesign Artifact 与项目 Markdown。
- 增加单元测试验证未变结构被保留且本轮交互/验收映射进入完整设计；增加集成测试验证修改 Run 持久化的 ArchitectureDesign 包含本轮需求。

## 4. 待办与归档条件

本 Review 暂不归档，因为 Engineer Patch 执行链仍未实现。归档前必须满足：

1. `EngineerAction`、动态 Context、ContextReceipt 和 `SourcePatchSet` Contract 落地；
2. Runtime 在隔离候选工作区完成 Patch 检查和 apply；
3. `SourceDiff` 由 apply 后真实文件重新计算；
4. Build/Test/Validation、失败恢复、基线冲突和越权 Patch 自动化测试通过；
5. 设计文档中的 `[TODO]` 状态与部署验收证据同步更新。

## Update 2026-07-15（动态读取与有界修正设计补全）

- 动态终态不再止于“补充读取后生成一个最终 Patch”，而是补齐为“受控读取 → 小步 Patch → 隔离 apply → 每步 Build/Test/Validator → 基于确定性错误生成有界 Repair Patch”。
- 新设计以 CandidateRevision 区分基线和每次成功 apply 后的候选源码；Repair Patch 必须绑定当前 revision、Manifest、ContextReceipt 和逐文件 `before_hash`，不能始终套用原始基线。
- V1 默认最多三次 Patch：一次 initial、最多两次 repair；初始阶段最多两轮补读，每次修复最多一轮补读。多个 Patch 不逐次 commit，最终只把完整验证通过的 revision 写成一个 ProjectVersion 和 Git commit。
- Runtime 决定失败是否可修复。平台错误、基线冲突、需求歧义、能力缺口、越权和范围扩大不能通过继续调用 Engineer 掩盖；失败或达到上限时丢弃候选，当前版本和发布指针不变。
- 当前代码只完成静态 Context 下的单次 SourcePatchSet/apply，不包含 CandidateRevision、PatchAttempt 链、RepairContext 或验证反馈循环；本 Review 的待办范围相应扩大，不能因第一阶段 Patch 已实现而归档。
