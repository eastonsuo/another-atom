# 首次生成修复与流式事件批处理

> 类型：Agent 检查｜状态：待办｜日期：2026-07-15｜版本范围：V1｜代码基线：`8b0871c`｜运行证据：Run `864c55b3-b262-4d77-8af0-6989ce34f80c`

- **规范设计：** [V1 多 Agent 设计](../../design/V1/技术设计/01-[Agent]-多Agent设计.md#83-自动修复输入输出)

## 背景

本次运行在首次生成后的 Runtime 单元测试中失败，Engineer Repair 随后执行，但第二轮仍出现同一测试错误。同时，调试日志累计 945 条事件，模型结构化输出被拆成大量 `agent.output.delta` 逐条保存。本文只检查这两个问题。

## 摘要

- **[P0｜Repair 无法修改测试]** 旧实现的 Repair Provider 只返回 `AppSpec`。Orchestrator 随后明确从旧 `SourceBundle` 提取 `prior_tests` 并原样复用，因此任何由生成测试自身导致的失败都无法修复。
- **[P1｜流式事件写放大]** Provider 以新增 256 字符或 0.15 秒为阈值发出原始输出事件；Orchestrator 对每个事件分别更新消息、写 RunEvent 并提交事务。长 Engineer 输出会形成数百条低价值事件。
- **[修复边界]** Repair 改为接收并返回完整 `EngineerOutput`，包含 AppSpec 与全部单元测试；原始输出事件改为 2048 字符或 1 秒一批，流结束强制补齐。前端既有逐字符动画保持不变。

## 证据

### 1. 单元测试失败被原样带入第二轮

运行中的 `tests/services/storageService.test.js` 在 VM Context 内执行：

```text
localStorage = new MockLocalStorage()
```

但 `MockLocalStorage` 只存在于测试文件的外层 CommonJS Scope，没有注入 VM Sandbox。第一次和 Repair 后第二次执行都报 `ReferenceError: MockLocalStorage is not defined`。

旧 Orchestrator 在 Repair 后从旧 `SourceBundle` 读取测试并构造新的 `EngineerOutput(app_spec=repaired_app_spec, unit_tests=prior_tests)`。所以即使 ValidationReport 包含准确堆栈，Provider 也既看不到测试正文，也没有返回修订测试的 Contract。

### 2. 原始模型输出逐批提交过密

旧 Provider 的 `agent.output.delta` 阈值是 256 字符或 0.15 秒。事件处理器收到每一批后都会：

1. 把 delta 追加到 `ProjectMessage.payload.model_output`；
2. 新增一条 RunEvent；
3. 立即提交数据库事务。

该机制能显示输出，但事件粒度与 Provider chunk 过度接近。日志中的 945 条事件与该写放大一致；仅凭日志不能把 945 条全部归因于 Engineer，但可以确认该阈值会显著放大长结构化响应。

## 修复设计

### Engineer Repair

1. 当前 `AppSpec` 与 `SourceBundle` 中的全部 `tests/*.test.js` 组装为当前 `EngineerOutput`。
2. `ValidationReport` 连同该完整输出交给 `repair_engineer_output()`。
3. Provider 返回完整修订版 `EngineerOutput`，允许修改 AppSpec、测试或两者。
4. Runtime 对颜色等架构 Token 做确定性对齐，再生成新 SourceBundle。
5. 新 SourceBundle 必须重新执行完整 Build、Unit Test 和 Validation；失败后不继续自动循环。

### 流式事件批处理

1. `agent.message.delta` 继续承载短的用户可见角色说明。
2. `agent.output.delta` 只保存结构化响应，累计新增达到 2048 字符或经过 1 秒才发出一批，结束时强制补齐。
3. Studio 继续读取累计 `model_output`，在浏览器内每 10ms 追加一个字符；后端批次变大不改变逐字显示。
4. Provider 的 `reasoning_content` / `thinking` 只用于进度统计，不写入用户可见原始输出。

## 验收标准

1. 构造只有生成测试失败、AppSpec 本身可通过的首次构建，Repair 能返回修订测试并完成第二轮验证。
2. `engineer_output_repair` Artifact 包含修订后的 AppSpec 和测试文件，`repair.completed` 事件列出修订测试路径。
3. 任意长度的流式结构化响应在合并所有 `agent.output.delta` 后与 Provider 完整响应一致。
4. 在没有真实时间推进的固定 chunk 测试中，原始输出事件数不超过 `总字符数 / 2048 + 2`。
5. Studio 仍按字符播放累计模型输出，工作期间不允许发送新消息。
