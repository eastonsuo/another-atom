## 背景

当前后端保留了 Project 版本列表和 Restore 接口，Studio 也仍有 `Preview / Edit / Versions` 结果标签。但该结果区只在最新 Run 为 `completed` 或 `completed_degraded` 时渲染。用户打开 Project 时加载的是最新 Run，因此一次后续修改失败后，即使 Project 仍有已验证版本，版本入口也会随结果区一起消失。

失败页目前只提供“使用已保存需求重新构建”。该动作解决的是重新执行失败任务，不能让用户直接返回已有的稳定结果。版本能力存在、恢复入口却在最需要时不可达，形成了产品闭环断点。

## 摘要

- **[P1｜Project 能力错误绑定 Run 状态]** 版本历史属于 Project，不应依赖最新 Run 是否完成。当前绑定使执行中、失败、等待输入和取消状态都无法从主界面访问历史版本。
- **[P1｜失败恢复只有重试]** Retry 创建新 Run 并再次消耗执行成本；Restore 应复用已有稳定结果。两者目标、成本和反馈信号不同，不能互相替代。
- **[P1｜Restore 写边界不完整]** 现有 Restore 只读取 `active_write_run_id`，没有像 Edit 一样原子取得 Project 写占用。并发写入可能在检查之后开始。
- **[版本语义]** “自动发布一个版本号”在本功能中表示创建递增的项目版本（ProjectVersion），来源为 `restore`；不表示自动公开发布（Publish），也不改变 Public Route 的版本指针。
- **[可用版本定义]** V1 的可用版本必须已有通过的校验记录，并在恢复时用当前 Runtime/Validator 重新执行仍通过。只有通过后才能生成新的恢复版本号。该定义不扩大为 Railway 部署健康或未实现运行时的可用性声明。

## 1. 现状证据

1. `GET /api/projects/{project_id}/versions` 能返回完整版本历史；`POST /api/projects/{project_id}/restore/{version_id}` 会重新校验并创建新的 `restore` 版本。
2. Studio 的 Versions 标签位于 `ResultWorkspace` 内，而 `ResultWorkspace` 只在最新 Run 完成且存在版本时渲染。
3. Studio 打开 Project 时请求 `/api/projects/{project_id}/runs/latest`。后续修改失败后，最新 Run 是失败 Run，即使 `Project.latest_version_id` 仍指向旧的已验证版本，Versions 标签也不可见。
4. 失败页只有 Retry，用户无法从失败上下文直接选择恢复或回到最近可用版本。

## 2. 修改结论

### 2.1 常驻版本入口

在右侧 Project 工具栏中新增“版本”入口，与“文件、Vim、日志”同层。只要 Project 已有版本，该入口在构建、失败、等待输入和完成状态下都可打开；没有版本时保留禁用入口并说明原因。

版本面板显示版本号、来源、创建时间、当前工作版本和 Restore 操作。选择历史版本后仍执行现有重新校验逻辑，通过后创建新版本号，不覆盖历史。

### 2.2 回到上个可用版本

失败页在 Retry 之外增加“回到上个可用版本”：

1. 后端从该 Project 的版本历史按版本号倒序查找候选；
2. 候选必须有通过的历史校验记录；
3. 使用当前 Runtime/Validator 重新执行候选；
4. 首个重新通过的候选生成新的 `restore` ProjectVersion 和 Git commit；
5. Project 当前工作版本移动到新版本，失败 Run 和全部历史继续保留；
6. 公开发布指针保持不变。

如果没有历史版本，或没有候选能通过当前校验，按钮不得制造成功假象：前者禁用并说明“尚无可用版本”，后者返回明确错误并保留当前状态。

### 2.3 写入与处理状态

指定版本 Restore 和“上个可用版本”共用同一原子写占用：绑定操作开始时的 `latest_version_id`，执行期间阻止 Edit、AI 修改、Vim Save 和第二次 Restore。界面在处理期间显示加载状态并禁用重复提交。

## 3. 验收标准

1. Project 已有版本时，右侧工具栏在最新 Run 失败和执行中仍显示可用的“版本”入口。
2. 失败页同时提供 Retry 和“回到上个可用版本”，并清楚说明两者分别重新执行和复用稳定版本。
3. 没有历史版本时恢复按钮禁用并显示原因。
4. “回到上个可用版本”跳过历史校验未通过的记录，并只在当前重新校验通过后创建版本。
5. 恢复创建递增的新版本号和独立 Git commit，来源为 `restore`，不删除历史。
6. 恢复后 Studio 打开该恢复版本的 Preview，而不是继续停留在失败页。
7. Restore 不改变已有 Deployment 的 `version_id`。
8. 并发写占用、跨用户访问、无可用版本和重新校验失败都有自动化覆盖。
