## 1. 基于现有代码的对话修改闭环

当前缺口不是再生成一份建议，而是让用户围绕已有 Project 和明确基线版本提出修改，并把结果真实写回同一个项目。

关闭条件：

- Project 对话按 `project_id / run_id / thread_id` 持久化并可恢复；
- 每次修改引用明确的 ProjectVersion 和 Git commit；
- Runtime 基于现有代码产生候选变更，并确定性计算文件 Diff；
- 修改经过校验后创建新的 ProjectVersion 和 Git commit；
- 版本冲突、校验失败和用户取消都有明确状态，不覆盖已有版本。

## 2. 失败后的原位恢复与 Resolve

现有失败页 Retry 已经能够使用保存的请求新建 Project 和 Run，并保留原失败记录；后台 Agent 阶段也有自动重试。当前缺口仅是从原 Project 的失败阶段继续，并复用仍然有效的 Artifact，以及实现 Resolve。

关闭条件：

- 失败页面说明失败阶段、原因和已经保留的 Artifact；
- 原位恢复从失败阶段继续，不重复执行已经成功且仍有效的阶段；
- 需要用户补充信息时进入 `needs_input`，补充后恢复原 Run 或创建有来源关系的新 attempt；
- Resolve 只处理可定位的应用问题，不把平台、Renderer 或未知错误伪装成已自动修复；
- 恢复次数、额外配额、停止和最终失败都有确定出口。
