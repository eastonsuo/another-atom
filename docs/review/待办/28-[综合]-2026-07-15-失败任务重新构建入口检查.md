## 背景

首次构建失败页提供“使用已保存需求重新构建”按钮，但点击后提示 `The retry request did not produce a code change proposal.`。页面文案表示该操作会保留失败记录并创建新任务，实际实现却先把原始 Prompt 发送到 Project Chat，再强制要求 Lead 返回代码变更提案。

## 摘要

- **[P0｜恢复命令误接对话路由]** Retry 是用户已经明确授权的运行恢复操作，不需要 Lead 再判断 `answer/clarify/propose_change`。旧实现把两种语义混在一起，Lead 未返回 `proposal_id` 时按钮必然失败。
- **[正确边界]** 已失败 Run 保留为历史证据；系统应在同一 Project 内用已保存输入创建新 Run。用户不需要手动创建新 Project。
- **[并发边界]** 新 Run 必须原子取得 Project 写占用，并校验当前版本仍与失败 Run 的基线一致；否则返回冲突，不允许覆盖新版本。

## 修复

1. 新增 `POST /api/runs/{run_id}/retry`，只接受当前用户拥有的 `failed` Run。
2. 首次构建失败时，新 Run 复用 Prompt、模型和 Session，从 Product Manager 阶段重新开始。
3. 现有版本修改失败时，新 Run 复用原 `base_version_id` 和选中文件；只有当前版本仍等于该基线时才重新进入修改队列。
4. 新 Run 写入 `project.recovery_started` 事件和带 `retry_of_run_id` 的 request 消息，不创建或批准 `change_proposal`。
5. Studio 失败页直接调用该接口，删除对 `sendProjectMessage()` 和 `proposal_id` 的依赖。

## 验收标准

1. 点击失败页 Retry 后返回 202，并在同一 Project 创建不同 `run_id`。
2. 原失败 Run 的状态、错误和 Artifact 不被覆盖。
3. 新 Run 保存 `retry_of_run_id`，Project 数量不增加，代码变更提案数量不增加。
4. 非失败 Run 调用 Retry 返回 `RUN_RETRY_NOT_ALLOWED`。
5. 当前版本已变化或 Project 正被写入时不创建第二个恢复 Run。
