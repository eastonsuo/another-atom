# Another Atom Review 文档规范与索引

[toc]

## 1. 作用

`docs/review/` 只负责检查、反思和 Bug，记录相应验证证据与阶段结论，回答“在什么基线下检查了什么、发现了什么、问题是否已经修复”。Review 不长期维护解决方案。

检查发现需要专门设计的问题时，应在 `docs/design/` 中创建或更新对应方案，并在两边建立链接。

## 2. 目录规则

Review 先按 `V1`、`V2`、`整体` 绑定检查范围，再按主要问题进入：

- `产品评审`：用户流程、交互、范围和价值表达；
- `Agent评审`：角色输出、Context、Memory、调度、Provider 和校验；
- `工程评审`：架构、数据、隔离、安全、部署、并发和恢复；
- `综合评审`：跨上述领域的实现盘点和里程碑总结。

Bug 按归属进入前三类，不建立“缺陷排查”同层目录。

## 3. 文件规则

1. 文件名使用 `YYYY-MM-DD-中文短主题.md`，日期表示首次创建日；
2. 标题和正文默认使用中文，Agent、Runtime、Context、Sandbox、API、Git 等固定技术名词可保留；
3. 标题使用 `【检查】`、`【反思】` 或 `【Bug】` 标签；
4. 首屏写清类型、状态、日期、范围和评审基线；
5. 结论引用代码、测试、部署或用户行为证据；信息不足时写“不能判断”；
6. 后续状态以带日期 Update 补充，不改写历史发现；
7. 需要系统性方案时链接 design，不在 Review 里维护第二套设计。
