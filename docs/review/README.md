# Another Atom Review 文档规范与索引

[toc]

## 1. 作用

`docs/review/` 是问题队列和历史证据库，回答“检查发现了什么、现在是否还需要处理”。Review 不长期维护解决方案；需要持续生效的产品或技术结论必须进入 [`docs/design/`](../design/README.md)。

## 2. 目录

Review 只分为`待办`和`归档`两类，不再按 V1、V2、整体或评审领域建立目录：

```text
docs/review/
├── README.md
├── 待办/
│   └── NN-[Tag]-YYYY-MM-DD-中文短主题.md
└── 归档/
    └── NN-[Tag]-YYYY-MM-DD-中文短主题.md
```

版本范围写在文档首屏元信息中，不进入文件路径。文件使用全局稳定编号，并在名称中标记评审类型，例如：

`08-[综合]-2026-07-12-关键设计与实现检查.md`

- `NN`：Review 的稳定编号，新文件使用下一个未占用编号，不因优先级或归档状态变化而重排；
- `Tag`：只使用`产品`、`Agent`、`工程`、`综合`；
- 日期：首次创建日期；
- 主题：足以独立识别问题的中文短标题。

文件从`待办`移入`归档`时保持编号、Tag、日期和主题不变。

## 3. 生命周期

```text
发现问题
  -> 写入 待办
  -> 修复、验证或做出明确范围决定
  -> 需要长期生效的结论回写 Design
  -> 在 Review 顶部追加 dated Update 和证据
  -> 移入 归档
```

### 3.1 待办

新 Review 默认进入`待办`。出现以下任一情况时不能归档：

- 仍有未明确处理方式的 P0/P1 问题；
- 需要修改 Design，但正式设计尚未更新；
- 声称已经修复，但缺少代码、测试或部署证据；
- 同一文档同时包含已关闭和未关闭问题，且未把遗留项转入新的待办 Review。

### 3.2 归档

归档表示该 Review 已经退出当前问题队列，不等于原文每条建议都被实现。每项发现必须已经落入以下一种结论：

- 已修复并有验证证据；
- 已转化为正式 Design；
- 明确决定不属于当前版本，并已写入版本边界；
- 被更新的待办 Review 接管，原文只保留为历史基线。

归档前在原文顶部追加 dated Update，说明处理结论、Design 链接、验证证据或接管它的待办 Review。原始发现不重写，文件名日期不改变。

## 4. 文件规则

1. 文件名使用 `NN-[Tag]-YYYY-MM-DD-中文短主题.md`；
2. 首屏写清类型、状态、版本范围、日期和评审基线；
3. 类型使用产品、Agent、工程或综合，但只作为元信息，不建立分类目录；
4. 结论引用代码、测试、部署或用户行为证据；信息不足时写“不能判断”；
5. 后续状态以 dated Update 追加，不覆盖历史发现；
6. 需要系统性方案时链接 Design，不在 Review 中维护第二套设计；
7. 文件移动到归档后，所有反向链接必须同步更新。
