# 【检查】多租户设计

> 类型：检查｜状态：历史评审，部分结论已更新｜日期：2026-07-11｜范围：用户隔离、授权、配额与审计

> 评审对象：`another-atom` 后端：`storage/models.py`、`api/routes.py`、`api/dependencies.py`、`domain/quota.py`、`agent/orchestrator.py`、`config.py`。
> 本文可独立阅读。背景一句话：平台生成的是用户资产（项目 / 版本 / 代码产物），一旦面向多个用户或组织，租户之间的数据、配额、权限如何隔离，就直接决定了安全性、计费正确性与可扩展性。

> **评审基线与当前状态：** 原始评审证据对应 `2c386c1`。截至 `8413ed7`，Preview 已增加 Project owner 校验，`BackgroundTasks` 已替换为带数据库 lease 的持久化 Worker，过期任务恢复会释放遗留预占。本文保留这些原始发现以记录问题来源；可信身份、Organization/Membership、统一租户过滤、租户级授权和审计等核心问题仍未解决。

## 1. 背景：为什么多租户是一个必须单独审视的议题

当前系统已经不是纯单机脚本：它有 `users` 表，每个 `Project/Run/Session` 都挂 `user_id`，配额也按 `User` 聚合。也就是说，**它已经在往多租户方向走，但只走了一半**：数据模型上区分了用户，鉴权、隔离强制性、后台链路的租户上下文却没有跟上。

多租户设计一旦在早期含糊，后期代价极高：一次漏加的 `WHERE user_id = ?` 就是一次跨租户数据泄漏；一次配额串号就是一次计费错误。这些问题在单租户 Demo 下永远不会暴露，却会在第一个真实的第二用户接入时集中爆发。所以现在就把隔离模型、上下文传递、配额边界讲清楚，是正确的时机。

## 2. 现状（基于评审基线代码）

### 2.1 租户的身份边界 = 单个 `user`，且未鉴权

- `dependencies.get_current_user` 通过请求头 `X-User-ID`（默认值 `demo-user`）识别用户；若该 user 不存在则自动创建一个 Demo 用户。
- **没有任何 token / 会话校验**：调用方声称自己是谁，就是谁。`X-User-ID: alice` 与 `X-User-ID: bob` 之间没有任何凭证壁垒。
- **没有 org / team / workspace 概念**：租户粒度目前等于一个 `user_id` 字符串，不存在“组织下多个成员共享项目”的模型。

### 2.2 数据隔离模型：共享库 + 共享表 + 应用层 `user_id` 过滤

- 采用单库单 schema、共享表 + 租户列（`user_id`）的模型：`Project`、`ProjectSession`、`Run`、`Approval`、`UsageLedger` 均带 `user_id` 外键。
- 隔离靠应用层手动过滤实现：`routes._owned_run` / `_owned_project` 在查询里显式增加 `Run.user_id == user_id` / `Project.user_id == user_id`。这是防越权的主要防线。
- 原评审基线存在多处按 id 直取、绕过租户过滤的路径：
  - `preview(version_id)` 曾直接 `db.get(ProjectVersion, version_id)`，无归属校验；当前版本已修复为 Version→Project owner join。
  - `public_app(public_id)` 设计上公开访问；`public_id` 一旦泄漏即等同公开，仍需发布审计和撤销语义。
  - `revise_project` / `restore_version` 入口先校验 Project，但版本对象归属依赖 Project→Version 关系推导。
  - `stream_events` / `event_history` 入口有 Run owner 校验；SSE 内部读取依赖入口已建立的授权事实。
- 子表（`Artifact`、`RunEvent`、`BuildJob`、`ProjectVersion`、`Deployment`、`Attachment`）不带 `user_id`，租户归属通过 join 父表推导。任何直接按子表 id 查询的代码都必须记得先校验父表，隔离仍主要依赖约定。

### 2.3 配额 / 计费：按 `user` 维度，无 org 维度

- 配额记在 `User.quota_limit / quota_used / quota_reserved`，明细写入 `UsageLedger(user_id, run_id, ...)`，没有 org/team 级配额池。
- 预留、结算和释放使用 `SELECT ... FOR UPDATE` 锁 user 行，已经考虑账户级并发。
- 原评审指出的剩余预占整笔结算、阶段重试漏计和 Worker 崩溃后预占悬挂问题已经修复：当前按实际 Provider 请求结算，记录 token，并在过期 lease 恢复时释放遗留预占。
- 仍缺少租户级周期对账与异常修复任务；计数字段与 Ledger 的长期一致性尚未由独立 reconciliation 机制保证。

### 2.4 后台任务的租户上下文

- 原评审基线通过 `BackgroundTasks` 仅传 `run_id`。当前已经迁移为持久化 `BuildJob` + DB lease，但 Job 仍主要通过 `run_id` 反查 `user_id`。
- 后台链路没有独立、强制的 tenant context；它信任 `job → run → user` 关系。若未来加入 Organization、跨项目合并或多租户 Worker，需要显式任务上下文和再次授权校验。

### 2.5 其他

- `config.py` 使用单一 `database_url` 与 `demo_quota_units`，无按租户配置维度。
- `RunEvent` 不含 `tenant_id`；审计需要 join Run/Project。

> **待确认**：是否有网关或反向代理在 `another-atom` 之外注入并校验真实身份，使 `X-User-ID` 成为可信头？当前仓库内未见此类中间件，本文按“无外部可信身份注入”判断。

## 3. 问题（架构 / 设计层面）

1. **身份不可信，隔离形同虚设。** 隔离过滤基于 `user_id`，而 `user_id` 来自无校验、有默认值、可伪造的请求头。没有可信身份，再严密的 `WHERE user_id = ?` 也只是约定。这是当前多租户设计最根本的问题。

2. **隔离靠“记得加过滤”而非机制强制。** 子表不带租户列，多处业务查询需要沿父表推导归属。随着路由增多，漏加过滤不是可以长期依赖人工避免的问题。

3. **租户上下文未作为独立 Contract 贯穿后台链路。** 当前持久化 Worker 解决了执行恢复，但 Job 仍只依赖 run/project 关系反查租户，缺少任务载荷与执行前的独立 tenant 校验。

4. **配额只有 user 维度。** 无法表达组织共享额度、成员子配额或组织级并发限制；同时仍缺少按租户的周期对账机制。

5. **鉴权与授权边界未成形。** 当前只有资源 owner 判断，没有 owner/editor/viewer 等租户内角色，也没有 Organization Membership。

6. **缺少租户级可观测性。** 事件和日志不直接带租户维度，难以按租户审计、限流和对账。

## 4. 更好的设计方向（可落地）

### 4.1 先确立可信身份与租户模型

- 建议引入 `Organization` 与 `Membership(org_id, user_id, role)`；`Project` 归属 `org_id`，user 通过 membership 访问组织资源。
- 若近期不做组织，也应抽象显式 `tenant_id`，不要长期把 `user_id` 同时当身份和租户边界。
- `X-User-ID` 必须替换为经校验的 JWT、session 或网关签名头。若由外部网关负责，应在应用中明确可信代理边界并拒绝绕过网关的直连。

### 4.2 把隔离从约定变成机制

- 引入租户感知 Repository/Query 层，默认注入 `tenant_id`；租户资源禁止直接裸用 `db.get(id)`。
- 子表可以冗余 `tenant_id`，或统一强制经父表 join 校验。两者至少选择其一作为工程约束。
- 在可信身份和应用层过滤之外，中长期评估 PostgreSQL Row-Level Security 作为纵深防御。

### 4.3 让租户上下文贯穿全链路

- 建立显式请求/任务上下文：`tenant_id`、`user_id`、`run_id`、`job_id`。
- BuildJob 领取和执行时重新校验 Job、Run、Project 与 tenant 的一致性，而不是只信任一个 id。
- 重放、恢复和失败处理都必须保持相同的租户上下文和审计记录。

### 4.4 配额按租户聚合且可对账

- 建立 org 级配额池与可选成员子配额，`UsageLedger` 增加 `tenant_id`。
- 以 Ledger 作为对账事实来源，增加定时 reconciliation；检测并修复长时间未结算的预占和聚合计数偏差。

### 4.5 租户内授权与资源限制

- 在租户边界上增加 owner/editor/viewer 等角色权限；发布、编辑、删除、仓库绑定按权限校验。
- 增加每租户并发 Run、QPS、Worker 时间和 Provider 用量限制，避免单租户影响其他租户。

### 4.6 租户级可观测性

- 给 `RunEvent`、Job、结构化日志和审计记录补 `tenant_id`（或 `org_id + user_id`）。
- 能按租户回答“谁在何时对什么资源做了什么、消耗多少、结果如何”。

## 5. 小结

当前系统已经具备多租户雏形：有 User、用户外键、owner 查询和账户级配额行锁；但隔离的强制性没有到位。可信身份、统一租户查询、租户上下文、Organization 授权和租户级审计仍是正式多租户上线前的必要条件。

**优先级建议（从高到低）：**

1. 可信身份，替换可伪造的 `X-User-ID`；
2. 统一租户感知的数据访问层，禁止租户资源裸 `db.get`；
3. 明确 `tenant_id` / Organization / Membership 模型；
4. 后台任务携带并校验租户上下文；
5. 配额按租户聚合并增加 reconciliation；
6. 补齐租户内授权、租户级限流和可观测性。
