## 1. Review 结论

当前代码已经形成一条真实可运行的 V1 纵向闭环：

```text
Lead 路由
  -> 创建 Project / Run
  -> Product Manager 生成 Blueprint
  -> supported 自动继续 / adapted 等待确认 / unsupported 等待改写
  -> Architect 生成 ArchitectureSpec
  -> Engineer 生成 AppSpec（HTML / CSS / JavaScript）
  -> Runtime 做确定性校验
  -> Data Analyst 解释校验结果
  -> 创建 ProjectVersion + Git commit
  -> Preview / Edit / Restore
  -> 用户显式 Publish / Unpublish
```

实现的核心不是“多个 Agent 自主协作”，而是 **LLM 生成受约束 Artifact，平台状态机控制交接与副作用**。这与 V1 的可检查、可恢复、可发布目标一致，也保留了用户对需求映射、版本和上线的控制权。

当前实现最需要在 Review 中把握的四个判断：

1. **Agent 链路已经落地，但自主性被有意压缩。** Lead 只做 `direct/team` 路由，四个角色按固定顺序执行；没有动态任务图、角色委派、并行协作或开放式 Tool Loop。
2. **Artifact 链是实现主干。** `Blueprint -> ArchitectureSpec -> AppSpec -> ValidationReport -> DataReview` 都有 Pydantic Contract，并持久化到数据库；后续阶段读取上游结构化结果，不依赖不可检查的聊天文本。
3. **“Build”目前主要是确定性校验和源码物化。** 生成应用是自包含 HTML/CSS/JavaScript；代码没有为每个项目执行 `npm run build`、浏览器测试或任意依赖安装。版本创建时才把 AppSpec 物化进项目 Git 仓库。
4. **工程闭环比 Agent 智能更完整。** 身份隔离、配额、异步 Job、lease 恢复、Git 版本、发布指针、失败状态和测试已形成实现；自动修复、完整项目对话和生产 Sandbox 验证仍有缺口。

## 2. 代码中的实际执行链路

### 2.1 Lead 与 Project 创建是两个独立步骤

Studio 先调用 `POST /api/lead/messages`。Lead 返回 `LeadDecision`：

- `direct`：保存一条 `LeadMessage`，展示回答，不创建 Project 或 Run；
- `team`：前端随后调用 `POST /api/runs`，创建 Project、ProjectSession、Run 和项目 Git 仓库，再异步启动 Product Manager；
- 用户点击强制调用团队时，`force_team` 直接返回 team，不消耗 Lead 模型调用，但后续团队仍正常计费。

这个拆分避免了能力咨询也产生空项目，但带来一个明确边界：Lead 消息和后续 Run 不是同一条持久化会话链，当前无法完整回放“咨询 -> 路由 -> 构建 -> 修改”的项目对话。

实现入口：`studio/src/App.tsx`、`another_atom/api/routes.py`、`another_atom/agent/provider.py`。

### 2.2 Product Manager 是能力边界的第一道门

Product Manager 输出 `Blueprint`，其中包含产品类型、页面、模块、视觉方向和能力映射。Runtime 按 `support_level` 进入三条确定性分支：

- `supported`：视为用户原始构建请求已授权，自动创建唯一 BuildJob；
- `adapted`：停在 `awaiting_approval`，用户可编辑并确认 Blueprint；
- `unsupported`：停在 `needs_input`，不创建 BuildJob，展示 PM 的 Web 改写草案。

用户接受 unsupported 草案时，系统会创建新 Run、复制已确认 Blueprint，并直接进入 Architect，不重复调用 Product Manager。用户要求重新生成草案时，才创建新的 Product Manager Run。

这一实现保护了产品身份：游戏、工具、看板可以生成自己的浏览器交互，不再被 Renderer 固定成商品目录。服务端认证、支付、数据库写入和外部服务则只能被适配或舍弃。

### 2.3 固定团队由状态机推进

Build Worker 领取 Job 后，`Orchestrator.execute_approved_run()` 固定执行：

```text
ArchitectureSpec
  -> AppSpec
  -> ValidationReport
  -> DataReview
  -> ProjectVersion
```

后一个阶段不能自由选择前一个阶段的替代角色，也不能调用 Shell、Git、发布或网络 Tool。文件、配额、数据库、构建状态和 Git commit 都由平台代码执行。

每个 Agent 阶段遵循同一套执行规则：

1. 如果该类型 Artifact 已存在，直接复用，支持中断后续跑；
2. 调用前预占配额并提交事务；
3. Provider 返回结构化对象后做 Pydantic 校验；
4. 保存 Artifact、结算实际请求量，再记录完成事件；
5. 输出失败时最多做 3 次阶段尝试，保留 `agent.retry` 事件；
6. 最终失败则写入 Run 错误并释放未结算配额。

需要注意：Ollama Provider 内部还会对不符合 Schema 的响应做一次 JSON 修复，因此“阶段最多 3 次”不等于“最多 3 个 HTTP 请求”。真实上限还受 Provider 内部修复和 DeepSeek fallback 影响。

实现入口：`another_atom/agent/orchestrator.py`、`another_atom/agent/provider.py`、`another_atom/domain/quota.py`。

### 2.4 Build 与 Validation 的真实含义

Engineer 输出的 `AppSpec` 同时保存页面元数据和 `html/css/javascript`。Runtime 当前不执行生成代码，而是做静态、确定性检查：

- HTML 和 CSS 是否存在；
- 是否包含外部 URL、`fetch`、XHR、WebSocket、动态 import、`eval`、iframe 等禁用能力；
- Blueprint 页面是否出现在 AppSpec 元数据中；
- Blueprint 模块是否完成 Engineer 交接；
- AppSpec 与 ArchitectureSpec 的颜色 Token 是否一致；
- 主色/背景和强调色/背景是否满足对比度阈值；
- 测试专用失败钩子是否触发。

Validation 通过后，Data Analyst 只能解释不可变的 `ValidationReport`，不能把失败检查改成通过。Validation 不通过时 Run 直接失败；设计文档中的 `Repairing -> BuildQueued` 自动修复状态目前没有落地。

因此，当前“构建成功”的准确含义是：**AppSpec Contract 合法、静态能力边界检查通过、源码可被 Preview Runtime 组合运行**。它不等价于浏览器端行为已经被自动化测试覆盖。

实现入口：`another_atom/build/renderer.py`、`studio/src/components/PreviewApp.tsx`。

## 3. Agent 实现 Review

### 3.1 角色边界

| 角色 | 实际输入 | 实际输出 | 当前权限边界 |
| --- | --- | --- | --- |
| Lead | 单条用户消息、模型选择、是否强制团队 | `LeadDecision` | 只做 direct/team；不创建文件、不发布 |
| Product Manager | 原始 Prompt、Mode | `Blueprint` | 定义 Web 范围和能力映射；不创建 BuildJob |
| Architect | 当前 Blueprint | `ArchitectureSpec` | 定义页面、数据实体和视觉 Token；不执行代码 |
| Engineer | Prompt、Blueprint、ArchitectureSpec | `AppSpec` | 生成自包含 Web 源码；无 Shell、依赖和网络权限 |
| Runtime Validator | 上述三个 Contract | `ValidationReport` | 确定性判定；不调用模型 |
| Data Analyst | Prompt、AppSpec、ValidationReport | `DataReview` | 解释数据与工程证据；不能改校验事实 |

这组边界的优势是结果可追踪、失败位置明确、模型不能越权执行副作用。代价是角色之间没有基于反馈的协商：Architect 不能要求 PM 补需求，Data Analyst 不能把问题退回 Engineer，Engineer 也没有可用 Tool 验证自己生成的交互。

### 3.2 Provider 与结构化输出

当前有两套 Provider：

- `MockLLMProvider`：本地开发和自动化测试使用，输出确定性 Contract；
- `OllamaCloudProvider`：真实模型入口，模型只允许配置列表中的 DeepSeek V4 Pro / Flash；Ollama 超时且配置了官方 Key 时可 fallback 到 DeepSeek 官方 API。

真实 Provider 把 Pydantic JSON Schema 放入请求，要求只返回 JSON；响应仍需经过括号平衡提取和 Pydantic 校验。第一次校验失败会把错误反馈给模型修复，第二次仍失败才抛出 Provider 错误。

这个实现的合理性在于不把 Provider 的“结构化输出承诺”当成可信事实。代价是调用次数和延迟可能高于 UI 表面展示的角色数，且 Prompt、Schema 和 Provider 行为目前没有独立的 prompt version 记录，复现实验只能依赖代码版本和持久化 Artifact。

### 3.3 Context 的实际实现

代码没有独立 Context Manager。各角色上下文由 Provider 方法参数直接组装：

```text
PM        = prompt + mode
Architect = blueprint
Engineer  = prompt + blueprint + architecture_spec
Analyst   = prompt + app_spec + validation_report
```

这已经避免传递完整聊天历史和其他角色隐藏推理，但设计文档中的 Artifact 引用、能力 Contract 版本、bounded failure evidence、Follow-up Context 等没有形成统一可审计的 Context Contract。

### 3.4 V1 Agent 能力边界

当前可以确认未实现：

- 动态任务拆分、并行角色、投票、仲裁和自主委派；
- ReAct 式 Action/Observation 循环；
- Agent 直接调用 Shell、文件、Git、Sandbox 或发布 Tool；
- Validation 失败后的自动 Repair / Resolve；
- 角色间多轮返工和用户后续自然语言修改；
- 完整项目对话及其压缩、检索和上下文恢复。

`Mode` Contract 仍保留 `engineer/team`，但 Studio 创建 Run 时固定传 `team`；即使服务端传 `engineer`，当前 Orchestrator 仍会生成 ArchitectureSpec、执行 Engineer、Validator 和 Data Analyst。它目前不是一条真正独立的单 Engineer 流程。

## 4. 工程实现 Review

### 4.1 数据与 Contract

SQLAlchemy 模型保存 User、AuthSession、LeadMessage、Project、ProjectSession、Run、Artifact、Approval、RunEvent、BuildJob、ProjectVersion、Deployment、Attachment、UsageLedger 和 SandboxSession。

核心业务事实分为三层：

- Run：一次 Agent 执行的状态、当前阶段、错误和配额；
- Artifact：该 Run 每种 Contract 的唯一最新结果；
- ProjectVersion：可预览、恢复和发布的不可变 AppSpec 快照，并关联 Git commit。

数据库默认 SQLite，也接受 PostgreSQL URL。启动时使用 `create_all` 和手写 `ALTER TABLE` 补列，没有 Alembic 或版本化迁移记录。对当前单实例 V1 足够直接，但生产升级的回滚、并发迁移和 Schema 演进证据较弱。

### 4.2 Auth 与租户隔离

身份由服务端 Session Cookie 解析，密码使用 `PBKDF2-HMAC-SHA256`（当前迭代次数 600,000）哈希；项目、Run、版本、Preview、文件、日志和 Sandbox API 都在查询时绑定当前 User。跨用户项目和 Preview 隔离已有集成测试。

当前 Cookie 的 `Secure` 由环境配置控制，本地默认关闭。架构文档中写的是 Argon2id，但实际 `another_atom/domain/auth.py` 使用 `hashlib.pbkdf2_hmac`，应以代码为准；如果 Argon2id 是验收要求，需要单独实现，不能在文档中视为已完成。

### 4.3 异步任务、持久化与恢复

Product Manager 通过 FastAPI BackgroundTask 启动；后续 Build 由进程内轮询 Worker 执行。BuildJob 有唯一 Run 约束、状态、attempt、lease owner 和 lease expiry：

- Worker 用行锁和 `skip_locked` 领取最早 Job；
- lease 过期的 building/validating Job 可以被重新领取；
- 已完成 Artifact 会被复用，避免重复调用已成功角色；
- 进程启动时会重放中断在 `product_running` 的 Blueprint；
- 已完成 Run 不因 Job 清理中断而再次执行。

优势是不用引入消息队列也能完成单实例恢复。边界是 Worker 与 API 同进程，lease 执行期间没有续租；长于 `worker_lease_seconds` 的真实模型链路可能被另一 Worker 误判为过期。V1 单副本下风险较低，多副本前需要重做并发模型。

### 4.4 Git、版本和发布

每个 Project 创建独立本地 Git 仓库。每次 Build、结构化 Edit、Vim Save 或 Restore 都：

1. 创建新的 ProjectVersion；
2. 写入 `app-spec.json`、`index.html`、`styles.css`、`app.js` 和版本标记；
3. 创建 Git commit；
4. 移动 Project 的 `latest_version_id`。

Restore 不覆盖旧版本，而是复制目标 AppSpec 生成新版本。Publish 使用独立 Deployment 记录；生成和编辑不会自动移动线上指针。Studio 当前固定使用 `specify_version`，API 虽接受 `always_latest`，但创建新版本时没有自动更新 Deployment 的 `version_id`，所以 `always_latest` 的语义尚未真正实现。

公开 URL `/apps/{public_id}` 由 Studio 加载匿名 `/api/public/{public_id}` 的 AppSpec。Unpublish 将 Deployment 置为 inactive，匿名 API 随即返回 404。

### 4.5 Preview 与生成代码边界

Preview 对 HTML/CSS/JavaScript AppSpec 使用 sandboxed iframe：父 Studio 负责取回 AppSpec，iframe 内执行组合后的自包含网页代码。静态 Validator 禁止明显网络和动态执行入口，Engineer Prompt 也要求离线内容。

这个边界能支持游戏、工具和看板的真实前端交互，同时不开放生成后端或宿主执行。代价是当前禁用检查主要是字符串扫描，不是完整 JavaScript 安全分析；代码是否能通过混淆、编码或其他浏览器 API 绕过，现有材料不足以证明。V1 不应把它描述为可执行任意不可信代码的强安全沙箱。

### 4.6 Vim Sandbox

Control Plane 只保存 SandboxSession 和代理 WebSocket。独立 Sandbox Host 为项目 HEAD 创建 detached Git worktree，移除 worktree 中的 `.git` 指针，再启动受限 Docker Vim：禁网、只读容器、drop capabilities、`no-new-privileges`，并设置 PID、内存和 CPU 限制。容器的 `--user` 使用 Sandbox Host 进程的 UID/GID；要满足非 root 边界，Host 进程本身必须以非 root 用户运行，代码没有额外拒绝 UID 0。

保存时 Control Plane 从 Host 读取 `app-spec.json`，重新做 Pydantic 和确定性校验，再创建 ProjectVersion 和 Git commit。并发 Save 通过 SandboxSession 的 `open -> saving` 条件更新避免重复版本。

实现代码与单元/集成测试可以确认命令参数、归属检查、并发保存和 Git 版本行为；当前材料不能证明目标 Linux Host 的 namespace、cgroup、Docker daemon、禁网和资源回收已经完成真实部署验收。

### 4.7 Studio 的可检查性

Studio 已展示固定角色时间线、Blueprint、持久化事件、Validation/Data Review、Preview、结构化编辑、版本历史、发布入口、项目文件、Run Artifact、日志下载和可选 Vim。SSE 用于即时事件，700 ms 轮询用于刷新完整 Run 状态，断线后可以通过 history API 恢复。

附件控件当前只把文件名、大小和 MIME 元数据提交到后端；`Attachment.storage_key` 没有写入，文件内容没有上传，也没有进入 Product Manager Context。因此它是可见入口，不是已完成的附件理解能力。

## 5. 设计与实现的主要差异

以下差异直接影响 V1 验收，不是措辞问题：

1. **自动修复未落地。** 架构/Agent 文档描述了 Repairing、Resolve 和失败后继续，代码当前在 Validation 失败后直接结束 Run。
2. **Approval Contract 被简化。** 数据库只保存每个 Run 唯一的 Blueprint Approval、布尔结果和 payload，没有 approval type、target hash、risk level、过期失效和拒绝状态。
3. **Build 没有执行项目构建。** 当前不做每项目依赖安装、编译或浏览器行为测试；BuildJob 实际编排 Agent、Validator 和版本创建。
4. **完整项目对话未实现。** LeadMessage、Run Event 和修改操作是分散事实，没有统一对话时间线。
5. **附件只有元数据。** 文件内容没有保存或传给 Agent。
6. **Always Latest 只有 API 枚举。** 新版本不会自动移动已发布 Deployment 指针。
7. **Auth 哈希实现与架构文档不一致。** 代码是 PBKDF2-HMAC-SHA256，不是文档所写 Argon2id。
8. **数据库迁移策略未达到生产化描述。** 当前是启动时手写补列，不是版本化迁移。
9. **Sandbox 有实现但部署证据不足。** 不能仅凭 Docker 参数把目标环境隔离视为已验收。
10. **Engineer Mode 不是独立流程。** UI 固定 team，服务端的 engineer 分支仍经过完整角色链的大部分阶段。

## 7. 建议的 Review 顺序

如果以 V1 验收而不是继续扩功能为目标，建议按以下顺序 Review：

1. **先确认 Build 的验收定义。** 如果 V1 接受“静态校验 + iframe 运行”，当前主线成立；如果要求真实浏览器行为验证，需要补执行层和测试证据。
2. **再确认失败恢复是否属于 V1 必须项。** 当前有进程恢复，但没有用户可操作的 retry/repair/resolve 闭环，这是完整性上的最大缺口。
3. **检查线上事实。** 在 Railway/PostgreSQL/真实 Provider 环境跑五轮主线、重启恢复、匿名发布和 Unpublish，决定交付声明能否成立。

5. **最后收敛文档声明。** 修正 Auth 哈希、Build 语义、Always Latest、附件和 Approval Contract，确保 README、PRD、架构与实际行为使用同一口径。

## 8. 对五项评价标准的映射

- **Completeness**：主线已到 Publish/Unpublish，并有版本、失败状态和恢复；用户级 retry/resolve、完整对话和真实浏览器验证仍不足。
- **Engineering Judgment**：Contract、平台权限、Job lease、配额和 Git 版本边界清楚；迁移、强 Sandbox 证明和多副本策略尚未完成。
- **User Experience**：关键阶段、Artifact、错误、Preview 和版本可见；附件、Always Latest 和部分失败恢复控件的能力口径需要收敛。
- **Innovation**：可检查 Artifact 链、固定交接和版本化结果已经进入代码，不依赖装饰性 Agent 行为。
- **Deliverability**：自动化主线较完整；当前 Ruff 未通过，本轮也没有真实部署、Provider 和浏览器验收证据，不能据此宣布最终交付完成。
