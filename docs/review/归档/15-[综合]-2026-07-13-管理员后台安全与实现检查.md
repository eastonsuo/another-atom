## 1. 检查结论

管理员后台的只读边界、权限依赖（`get_current_admin` 无测试环境兜底）、响应字段裁剪、`Cache-Control: no-store` 和 401/403 负路径测试都符合设计。本次发现的问题集中在**部署安全配置没有代码强制**：设计写了"公开部署必须覆盖默认凭据"，但任何一行代码都不阻止 `admin / admin12345` 直接上生产。

## 2. P1 发现

### 2.1 默认管理员凭据：版本边界决定

原始发现：`admin / admin12345` 是公开源码中的已知凭据，部署到公网不配环境变量时，任何人都可以用它进入后台查看全部用户、Project 和 Run 日志；设计 §2.1 写了"公开部署必须覆盖默认值"，但代码没有强制。

**2026-07-13 决定（产品负责人确认）：** V1 允许默认管理员直接用于公开部署，不在启动时拒绝。理由是 V1 属于演示交付，管理后台本身只读，可接受该暴露面。

预期的账号生命周期是：先用默认管理员登录并投入使用，之后更换为正式管理员账号，并取消默认账号的管理员权限。V1 后台没有账号管理写操作，这个流程通过环境配置完成：

1. 在部署环境把 `ADMIN_USERNAME` / `ADMIN_PASSWORD` 改为正式账号并重启；
2. 启动 bootstrap 创建/激活新管理员；
3. 同一次启动中，用户名不等于当前 `ADMIN_USERNAME` 的 admin 角色账号（含旧的默认 `admin`）被自动降级为普通用户（见发现 2 的修复），不需要任何手动取消步骤。

配套缓解措施：非 `development/test` 环境下凭据仍为默认值时，启动记录 `default_admin_credentials_active` 告警日志；设计文档 §2.1 已同步为"允许默认账号、建议覆盖"的表述。此项不再作为待修复问题，转为记录在案的版本边界。

### 2.2 更换凭据后旧默认管理员不降级

Bootstrap 只按**当前配置的用户名**匹配和升级。先用默认值部署过一次（创建 `admin` 账号），后来配置 `ADMIN_USERNAME=ops`，bootstrap 会创建 `ops`，但旧 `admin` 账号保持 `role=admin` 和旧密码，成为一个配置中不存在、页面上也看不到的遗留管理员（见发现 5）。

**处理方向：** bootstrap 在确保配置管理员之后，把所有 `role=admin` 但用户名不等于当前 `ADMIN_USERNAME` 的账号降级为 `user` 并记录日志。V1 的管理员身份唯一来源是环境配置，数据库中不应存在配置外的 admin 角色。

## 3. P2 发现

### 3.1 管理员登录无限流，失败不留日志

`admin_login` 只在成功时记录 `admin_login_completed`；错误密码或非管理员账号探测既无日志也无任何限流。默认用户名 `admin` 是公开已知的，爆破窗口完全敞开。设计 §6.2 要求记录管理员访问日志，失败尝试同样属于需要留痕的访问事实。

**处理方向：** 失败路径记录 `admin_login_failed`（用户名 + 原因）；按用户名做窗口限流（如 15 分钟 5 次失败后返回 429）。V1 单实例，进程内计数即可，不引入外部存储。

### 3.2 登录页无条件展示默认账号提示

`AdminApp.tsx` 登录表单固定渲染"默认开发账号：admin / admin12345"，生产构建同样显示，等于在公网登录页挂出默认凭据提示。

**处理方向：** 仅开发构建（`import.meta.env.DEV`）显示该提示。

### 3.3 管理员账号在后台不可见

用户列表和 `_admin_user` 都过滤 `User.role == "user"`，任何 admin 角色账号（包括发现 2 的遗留管理员）都不出现在后台。设计 §8.1 写"看到所有注册用户"，当前实现与其有偏差，且让遗留管理员无法通过产品自身被发现。

**处理方向：** 发现 2 的降级机制落地后，配置外不再存在 admin 账号，"管理员互相不可见"的风险面消失。V1 保持用户列表只含 `role=user` 的运营对象，此偏差作为版本边界决定记录，不单独改列表接口。

## 4. P3 发现

1. **LIKE 通配符未转义：** 搜索词中的 `%`、`_` 直接进入 pattern，输入 `%` 等于列出全部用户。只读接口影响有限，但与"受控查询"意图不符，应转义后带 `ESCAPE` 查询。
2. **最新 Run 查询无界：** `_latest_runs` 把候选项目的**全部** Run 读入内存再取第一条；用户的 Project 列表本身也不分页。数据量大时展开用户会显著变慢，应改为窗口函数每项目取一条。
3. **简介忽略历史 Blueprint：** `_blueprints` 只查最新 Run 的 Blueprint。项目曾成功构建、但最新 Run 在产品阶段前失败时，简介退回 prompt 截断，而设计 §2.3 的取值顺序是"已有 Blueprint 时"优先。应按项目取最近一次成功产出的 Blueprint。
4. **`session_cookie_secure` 默认 False：** 管理员与用户共用这套 Session Cookie，Railway 部署若不设置该值，Cookie 不带 Secure 标记。应加入生产环境启动告警和 Railway 验收清单。
5. **失败阶段文案拼接：** 前端 `runStage` 用 `${stage}失败` 拼接，`failed + complete` 组合会显示"预览就绪失败"。纯文案边角，概率低，可暂不处理但记录在案。

## 5. 已确认无问题的项

- `get_current_admin` 独立鉴权，无测试环境 `X-User-ID` 兜底，无法绕过；
- 全部响应经 `AdminUserSummary` / `AdminProjectSummary` 等 Pydantic 视图裁剪，不含 `password_hash`、Session token、Sandbox token、仓库路径和 Provider 密钥；
- 全部管理接口设置 `Cache-Control: no-store`；
- 未识别状态按原始值显示（设计 §7）；
- `tests/integration/test_admin_console.py` 覆盖 401/403、搜索、缺失资源、详情和日志下载。
