# V1 管理员后台需求与交互实现检查

[toc]

- 检查日期：2026-07-13
- 设计基线：[V1 管理员后台需求与交互](../../design/V1/产品设计/02-管理员后台需求与交互.md)
- 检查范围：管理员身份、只读 API、管理页面、负向权限路径和部署待验收项

> **2026-07-13 归档 Update：** 管理员后台的仓库内功能和自动化测试已经完成。外部环境验证不作为当前 Review 待办，因此本文归档，不再创建对应待办项。

## 1. 检查结论

管理员后台已经完成本地实现和自动化验证，当前仍缺 Railway 环境中的非默认管理员凭据与真实持久化数据验收。因此可以判定为“本地闭环完成”，不能判定为“部署验收完成”。

## 2. 已检查的实现

- User 已增加 `role`，数据库启动迁移会为旧用户补默认角色；已实现默认管理员初始化和 `get_current_admin`。
- 已实现用户分页、搜索、用户下 Project 列表、Project 详情和管理员 Run 日志等只读 API。
- 已实现 `/admin/login` 和 `/admin`，包含用户列表、Project 展开、详情抽屉、手动刷新和分页。
- 管理员查询复用现有 User、Project、Run、Artifact、RunEvent 和 UsageLedger 等持久化事实，没有建立脱离业务状态的第二套管理数据。
- 普通用户的 Project、Run、Preview、Git 和 Sandbox 归属校验没有因管理员入口而放宽。

## 3. 自动化证据

`tests/integration/test_admin_console.py`覆盖未登录 `401`、普通用户 `403`、搜索、缺失资源、Project 详情和管理员日志下载。2026-07-13 使用项目虚拟环境执行 `.venv/bin/pytest -q`，85 项后端测试全部通过。管理员页面此前已经过 Studio 类型检查与生产构建验证，本次文档整理未修改前端代码。

## 4. 待完成验收

在 Railway 配置非默认的 `ADMIN_USERNAME` 和 `ADMIN_PASSWORD`，重新部署后使用真实注册数据验证：

1. 管理员 Session 和重启后的角色持久化；
2. 用户与 Project 分页、搜索和详情显示；
3. 最新 Run 阶段、错误和日志下载；
4. 普通用户及未登录请求无法访问任何 `/api/admin/*` 资源；
5. 管理响应不泄漏密码哈希、Session、Sandbox token、仓库绝对路径或 Provider 密钥。
