## 背景

V1 的完整闭环不仅需要生成页面，还需要真实身份、持久化状态、Project Git、异步任务、受控编辑、失败恢复和发布边界。本文把这些工程能力放在同一架构中，说明各组件如何支撑产品和 Agent Contract。

## 摘要

- **Studio 与 API**
  - React Studio 承载工作台、Preview 和终端界面，FastAPI/Pydantic 提供命令、查询和显式 Contract。
- **身份与持久化**
  - 服务端 Session 确认用户归属，SQLite 持久化 Project、Run、Artifact、配额、事件、版本和发布状态。
- **源码事实**
  - 每个 Project 绑定服务端本地 Git 仓库，ProjectVersion 通过 commit SHA 对应可恢复源码。
- **异步执行**
  - 主服务的单并发持久化工作器（Worker）、任务租约（Job lease）和阶段产物（Artifact）检查点负责调度；共享独立执行服务完成源码物化、构建、测试和校验。
- **隔离边界**
  - 构建、测试和校验与主服务进程、持久化卷和业务密钥隔离；源码 WebIDE 仍依赖受限 Linux 沙箱宿主机（Sandbox Host）。模型不直接获得终端（Shell）、文件系统或发布权限。
- **部署边界**
  - Railway 同一项目和环境承载单副本主服务与共享执行服务；该服务级隔离不能替代任意代码或交互式编辑所需的任务级强沙箱。

## 1. 技术选型

V1 采用云端（Cloud）产品形态，但把控制面与执行面分开：React 提供可视化工作台，FastAPI 承担 API、智能体（Agent）编排和领域服务，SQLite 保存业务事实；Railway 同一项目、同一环境中的共享执行服务承担源码物化、固定构建、单元测试和确定性校验。主服务保持单副本，并把 SQLite 与项目 Git 放在持久化卷（Volume）；受限 Vim 的交互式编辑仍由独立 Linux 沙箱宿主机（Sandbox Host）承担。

| 层次 | V1 选型 | 选择原因 | V1 边界 |
| --- | --- | --- | --- |
| 前端工作台 | React + TypeScript + Vite + xterm.js | 适合实现状态密集的 Studio、类型化 API、iframe Preview 和受限 Vim 终端 | 不直接访问 LLM、数据库或 Volume；xterm.js 只连接经过授权的 Editor Session |
| API 与领域服务 | Python + FastAPI + Pydantic + Uvicorn | 与 Agent/结构化输出保持同一语言栈；Pydantic 同时约束 API 和模型产物 | HTTP 请求只创建任务，不同步执行构建 |
| 身份认证 | 用户名/密码 + Argon2id 哈希 + 服务端 Session Cookie | V1 需要两个真实账号切换并验证 Project 隔离；服务端 Session 比可伪造用户头更适合作为归属事实 | V1 租户粒度为 User，不实现 Organization/Membership |
| Agent 编排 | 自有 Orchestrator + Provider Adapter（Ollama Cloud / Mock）+ Pydantic | 支持按 Run 选择模型、角色 instruction、有界重试和结构化校验 | Ollama Cloud 不提供服务端 Structured Outputs，Schema 由本地严格校验 |
| 前后端通信 | REST + SSE + WebSocket + OpenAPI | REST 承担命令，SSE 推送事件，WebSocket 只承载终端字节流，OpenAPI 同步业务类型 | WebSocket 不承载 Agent 状态或任意 RPC |
| 业务数据库 | SQLite + SQLAlchemy | 用一个持久化文件完成 V1 单副本的用户、Session、配额、Job、版本和发布状态 | Railway 必须挂载持久化 Volume；不支持多副本并发写入 |
| 项目源码与构建 | 主服务项目 Git + 共享执行服务 + 运行时适配器（Runtime Adapter） | 主服务保存源码事实，执行服务接收源码包副本并返回构建、测试和校验证据；当前目标适配器为 `web-static-v1` | 非网页源码/构建适配器尚未实现；V1 不配置 Git 远端，也不接受模型生成的 Shell 命令 |
| 代码终端 | xterm.js + WebSocket + 隔离 PTY + 固定 Vim | 给用户真实键盘编辑体验，同时把修改限制在当前 Project 仓库 | 不启动登录 Shell；禁用 shell escape、插件、网络和仓库外路径 |
| 异步执行 | 主服务单并发构建工作器（Build Worker）+ 私有 HTTP 共享执行服务 | 主服务数据库仍是唯一队列事实源，执行服务无状态地流式返回阶段事件和结果 | 初始全局并发为一；提高并发前必须完成公平调度、资源和隔离压测 |
| 部署 | Railway 单副本主服务 + SQLite/Project Git Volume + 共享执行服务；WebIDE 另用 Linux Sandbox Host | 主服务处理 Web/API/Agent 和持久化，执行服务不挂卷、不持有业务密钥，编辑宿主机只承担受限 Vim | Railway 执行服务仅提供受限适配器的服务级隔离，不能声称为任意代码强沙箱 |

### 1.1 选型原则

1. **契约（Contract）优先**：产品规格（ProductSpec）、架构设计（ArchitectureDesign）、应用规格（AppSpec）、源码包（SourceBundle）、事件（Event）、错误（Error）和导出（Export）以文档及 Pydantic 模型约束为事实来源。
2. **模型与权限分离**：LLM 只产生结构化决策，平台 Runtime 执行配额、文件、Git、构建和发布操作。
3. **状态先持久化再推送**：Run、Job 和事件先写 SQLite，再通过 SSE 发送，断线后可重放。
4. **身份与归属先于资源读取**：Project、Repository、Version、Terminal 和 Job 都从服务端 Session 取得当前 User，不能信任客户端传入用户 ID。
5. **先完成纵向闭环**：V1 的 Lead 只做直接回答或固定团队二选一路由，不为 V2 动态 TaskGraph、并行或仲裁提前引入分布式组件。

## 2. 架构结论

Another Atom V1 只交付云端（Cloud）形态：主服务（Control Plane）和共享执行服务（Runtime Executor）部署在 Railway 同一项目、同一环境，通过私有网络通信；项目 Git 和业务状态由主服务持久化，执行服务无状态完成源码物化、构建、测试和校验。代码终端的交互式编辑仍部署在受控 Linux 沙箱宿主机（Sandbox Host）。详细执行边界以[共享独立执行服务设计](./08-[工程][TODO]-共享独立执行服务.md)为事实来源。

用户使用用户名密码登录 React 可视化工作台（Visual Studio）。团队负责人智能体（Lead Agent）对每条消息只做二选一：直接回答/澄清，或启动产品经理（Product Manager）→架构师（Architect）→工程师（Engineer）固定团队。工程师交付源码和单元测试后，由运行系统（Runtime）执行确定性构建（Build）、测试（Test）和校验（Validation）。通过验证的源码提交到项目（Project）绑定的服务端本地 Git 仓库，构建产物则作为独立版本产物由主服务持久化；用户既可通过结构化编辑，也可通过 xterm.js 中的受限 Vim 修改源码。保存版本时平台统一校验、构建、提交 Git commit，并将项目版本（ProjectVersion）映射到 commit SHA。

```text
┌──────────────────────────── 用户交互层 ────────────────────────────┐
│                                                                  │
│                     React 可视化工作台                     │
│  登录 / Lead 对话 / 风险确认 / 构建事件 / Preview / xterm+Vim    │
│                              │                                   │
└──────────────────────────────│───────────────────────────────────┘
                               │ REST + SSE + WSS
                               │ OpenAPI 类型同步
                               ▼
┌────────────────────主服务（Control Plane）────────────────────┐
│                                                                 │
│  FastAPI                                                        │
│     │                                                           │
│     ├── 认证 / 项目 / 仓库 / 版本 / 发布                    │
│     │                                                           │
│     ├── 团队负责人路由 + 固定顺序团队编排                  │
│     │      消息 → 回答 | 澄清 | 固定团队                     │
│     │                         │                                  │
│     │                         ▼                                  │
│     └── 构建任务 / 编辑会话队列                              │
└──────────────┬──────────────────┬───────────────────┬─────────────┘
               │                  │                   │
               ▼                  ▼                   ▼
      SQLite + Git 卷      共享执行服务        Linux 编辑沙箱宿主机
      用户/配额/任务/版本     构建/测试/校验        受限 Vim

外部：模型服务适配器（Provider Adapter）-- HTTPS --> Ollama Cloud API
```

V1 的 xterm.js 终端只提供当前 Project 仓库中的受限 Vim，不是 Terminal CLI 或 CC 式本地 Agent Runtime。V1 不实现用户电脑上的本地工作区、SQLite 同步、登录 Shell 或 `localhost` Dev Server。

## 3. 已确认的架构决策

### 3.1 V1 必须实现

#### 3.1.1 Agent 与产物 Contract

- 真实 LLM 调用，不使用预设文本替代 Blueprint 或 AppSpec。
- 结构化 Blueprint 与 AppSpec，所有模型输出都必须经过 Pydantic 校验。

#### 3.1.2 受控构建与异步执行

- 构建链按项目类型选择源码/构建/运行时适配器（Source/Build/Runtime Adapter）；第一阶段只实现受限 `web-static-v1`、固定网页运行边界和固定校验规则。
- 预览（Preview）是网页适配器（Web Adapter）的输出能力，不是所有项目（Project）的必备字段。没有匹配适配器的项目不得被改写成网页项目；运行系统（Runtime）应在构建前返回明确能力缺口。
- 适配器不安装动态依赖，也不执行模型生成的终端（Shell）命令；固定构建和测试命令固化在共享执行服务镜像中。
- 构建脱离 HTTP 请求生命周期，通过主服务持久化异步任务调度，并在共享执行服务中完成源码物化、构建、单元测试和校验。

#### 3.1.3 身份、租户、状态与配额

- 平台使用用户名和密码登录；密码只保存强哈希，浏览器使用服务端 Session Cookie，不信任客户端提交的 `X-User-ID`。
- V1 的租户边界是单个 User，不实现 Organization/Workspace；所有 Project、Session、Run、Version、Repository 和配额查询必须绑定当前登录用户。
- 用户通过退出当前账号并登录另一个账号完成切换；切换后只能看到新账号所属项目，公开发布页除外。
- SQLite 持久化用户、项目、Session、Run、Job、版本、配额和附件元数据。
- 主服务持久化卷保存附件、项目 Git 仓库和已经通过的构建产物；共享执行服务不挂载持久化卷。
- 同一用户拥有多个 Session，所有 Session 共享账户配额。

#### 3.1.4 Project 与本地 Git 仓库

- 每个 Project 默认绑定一个平台服务端本地 Git 仓库，关系为一对一。
- 仓库保存在主服务持久化卷的受控目录，不代表用户电脑上的本地仓库，也不连接 GitHub/GitLab 远端（remote）。
- Build、Edit、Resolve 和 Restore 生成 ProjectVersion 时必须创建对应 Git commit，并把 commit SHA 写回版本记录。
- Git 初始化、文件写入、提交和恢复由平台 Repository Service 执行固定命令；Agent 不获得任意 Git/Shell 权限。

#### 3.1.5 风险确认与公开交付

- 用户明确请求构建且 Blueprint 为 supported 时可直接进入固定团队；adapted、额外预算、范围变化、破坏性仓库操作和线上变更必须通过 Risk Policy 确认。
- React Visual Studio 通过 SSE 查看 LLM、构建和验证事件。
- 公开 Preview、显式 Publish、Update 与 Unpublish。

### 3.2 V1 明确不实现

#### 3.2.1 本地执行与数据同步

- Terminal CLI、用户电脑上的本地仓库执行或 CC 式本地 Agent Runtime。
- SQLite/PostgreSQL 双写或在线同步；V1 只维护一个 SQLite 事实源。

主服务持久化卷内的项目 Git 仓库属于 V1 平台存储，不属于本节排除的用户本地执行。

#### 3.2.2 尚未接入的源码与执行适配器

- `npm install`、动态依赖或模型生成的 Shell 命令。
- 非 Web SourceBundle、原生移动端、桌面端、CLI、服务端项目的 Build/Runtime Adapter。它们是当前实现缺口，不是允许系统改变用户项目类型的理由。
- 公网多租户任意代码执行沙箱。

#### 3.2.3 Agent 与模型扩展

- 任意 Provider 或不受限的模型标识；V1 只允许从平台配置的 DeepSeek allowlist 中按 Run 选择模型。
- 真实多 Agent 自主委派与并行执行（由 V2 实现）。

#### 3.2.4 生成应用的业务后端

- 生成应用内部的数据库、认证、支付和订单系统。

#### 3.2.5 平台商业化能力

- Stripe 真实支付、Wallet、充值和发票。

## 4. 产品执行链路

```text
[1. 登录与请求路由]

用户名 + 密码
    |
    v
校验密码哈希 -> 创建 AuthSession Cookie
    |
    v
用户消息 -> Lead Agent -> LeadDecision
                    |
                    +-- answer / clarify -> 对话回复，流程结束
                    |
                    `-- team -> 创建 Project + ProjectRepository + Session
    |
    v
[2. 固定团队与必要确认]

产品经理（Product Manager）-> 产品规格（ProductSpec）+ 产品蓝图（Blueprint）索引 -> Pydantic 校验
    |                    |
    |                    `-- 校验失败 -> 有限重试 -> run.failed
    v
supported / adapted -> 用户确认 ProductSpec 内容指纹
范围变化 / 超预算 -> approval.required -> 用户决定
unsupported -> needs_input + PM 可构建草案 -> 用户确认或编辑 -> 新 Run
    |
    v
[3. 设计与应用规格]

架构师（Architect）-> 架构设计（ArchitectureDesign）文档 -> Pydantic 校验引用
    |
    v
工程师（Engineer）-> 应用规格（AppSpec）+ 源码包（SourceBundle）+ 单元测试 -> Pydantic 校验
    |
    v
[4. 异步构建]

创建 Build Job -> API 返回 202
    |
    v
Worker 获取 Job lease
    |
    v
主服务读取已批准产物与源码包 -> 私有调用共享执行服务
    |
    v
运行时适配器（Runtime Adapter）在临时目录物化源码包（SourceBundle）
    |
    v
[5. 测试、校验与预览]

固定构建与单元测试 -> 执行报告（ExecutionReport）
    |
    +-- 失败 -> build.failed / test.failed -> 不创建版本
    |
    v
确定性校验 -> 校验报告（ValidationReport）
    |
    v
主服务校验指纹 -> 持久化源码/产物 -> Git commit -> Preview 就绪
    |
    v
[6. 修改、版本与发布]

用户结构化编辑或 xterm+Vim -> dirty worktree -> Save Version
    |
    v
构建（Build）+ 测试（Test）+ 校验（Validation）-> Git commit -> ProjectVersion(commit_sha)
    |
    v
用户显式发布选定 Version
    |
    v
公开 HTTPS URL
```

`publish` 不属于 Agent Loop。Lead 不能自动发布。普通受支持构建、预览、追加 Git commit 不需要额外审批；`adapted` 范围、额外预算、删除、Restore、Publish/Update/Unpublish 等高影响动作按风险策略插入确认。

## 5. 组件设计

### 5.1 React Visual Studio

技术：React、TypeScript、Vite。

职责：

- Prompt Composer 与附件选择。
- 用户名密码登录、退出和账号切换。
- Lead 单一对话入口，以及“直接回答 / 调用团队”的可见路由结果和用户覆盖入口。
- Blueprint 展示、编辑；只有命中风险策略时显示内联 Approval。
- 构建阶段、工具事件、错误和重试入口。
- iframe 应用预览及 Desktop/Mobile 尺寸切换。
- Follow-up 修改、版本选择、Restore 和 Resolve。
- xterm.js 受限 Vim 编辑器、dirty 状态和 Save Version。
- Publish、Update 与 Unpublish。

Visual Studio 不直接访问 LLM、数据库、Volume 或工作区文件。

### 5.2 FastAPI Application Service

技术：FastAPI、Pydantic、SQLAlchemy、SQLite、Uvicorn。

职责：

- 平台用户身份与资源归属校验。
- AuthSession、Project、ProjectRepository、Session、Run、Attachment、Version 和 Deployment API。
- Plan、Quota Account 和 Usage Ledger。
- 启动 LLM Run 与创建 Build Job。
- 将持久化事件通过 SSE 推送给浏览器。
- 托管 Visual Studio 和生成应用的静态构建结果。

### 5.2.1 Auth 与用户级租户边界

- `POST /auth/register` 创建唯一 username，并使用 Argon2id 保存 `password_hash`。
- `POST /auth/login` 校验凭证，创建只在服务端可解析的随机 Session；Cookie 设置 `HttpOnly`、`Secure`、`SameSite=Lax` 和过期时间。
- `POST /auth/logout` 撤销当前 Session。切换用户等价于 logout 后 login，不允许在请求里直接指定 `user_id`。
- V1 的 tenant 等于 User；所有 owner 查询从 AuthSession 取得 `user_id`。测试环境可以保留显式依赖覆盖，但生产路由不接受 `X-User-ID` 作为身份。
- 登录、失败登录、退出和越权拒绝写入安全审计事件；日志不记录密码、Session token 或完整 Cookie。

API 只负责创建异步任务和返回状态，不在请求协程中执行 `npm run build`。

### 5.3 Lead Router 与 Sequential Team Orchestrator

技术：httpx、Ollama Cloud API、Pydantic；Mock Provider 用于自动化测试。

本节只描述 Agent 与工程 Runtime 的集成摘要。执行范式、角色 Contract、Human-in-the-loop、Context、Tool、Sandbox、验收和有限修复以 [V1 多 Agent 设计](./01-[Agent]-多Agent设计.md)为事实来源。

V1 使用一个独立团队负责人智能体（Lead Agent）和三个专业模型角色。团队负责人（Lead）只生成团队负责人决策（`LeadDecision(route=direct|team)`）：`direct` 负责回答或澄清，`team` 启动固定顺序流水线（Pipeline）。运行系统（Runtime）校验路由、风险与预算；团队负责人不能自由挑角色、并行、返工仲裁或发布。

```text
团队负责人路由：`team`

产品经理（Product Manager）-> 产品规格（ProductSpec）+ 产品蓝图（Blueprint）索引
    -> 用户批准 ProductSpec 内容指纹
    -> 架构师（Architect）-> 架构设计（ArchitectureDesign）文档
    -> 工程师（Engineer）-> 应用规格（AppSpec）+ 源码包（SourceBundle）+ 单元测试
    -> 运行系统构建/测试（Runtime Build/Test）-> 执行报告（ExecutionReport）
    -> 确定性校验器（Validator）-> 校验报告（ValidationReport）
         |-- 通过 -> 预览就绪
         |-- 可修复 -> 工程师修复一次 -> 完整重跑
         `-- 其他 -> run.failed

团队负责人路由：`direct`

团队负责人智能体 -> 直接回答或澄清 -> 不创建团队运行（Team Run）/不修改仓库
```

V1 路由与团队约束：

- 团队负责人（Lead）只能 `direct/team` 二选一；用户可以显式覆盖为“调用团队”。
- `team` 内只按产品经理（Product Manager）、架构师（Architect）、工程师（Engineer）固定顺序执行，不由模型动态决定下一角色。
- 不并行执行，不进行智能体（Agent）间自由讨论。
- 阶段之间只传递经过模式（Schema）校验的显式产物，不共享隐藏长期记忆。
- 每个阶段独立记录模型、用量、尝试次数、输入产物版本和输出产物。
- 数据分析师（Data Analyst）和质量评审员（Reviewer）不进入 V1 新运行；历史数据分析（DataProfile）和质量评审报告（ReviewReport）只保留只读兼容。
- 运行系统构建（Runtime Build）、测试（Test）和校验器（Validator）生成不可由工程师改写的执行报告（ExecutionReport）和校验报告（ValidationReport）。只有确定性强制检查全部通过才创建版本。
- 动态角色子集、TaskGraph、独立工具权限、并行执行、反馈循环和仲裁属于 V2 自主多 Agent。

#### 5.3.1 工程集成边界

- Agent Service 接收 Runtime 组装的阶段输入，完成 Provider 调用、Pydantic 校验、Artifact 持久化和 Usage/Trace 记录。
- Orchestrator 只根据持久化 Artifact、Approval、ValidationReport 和状态机推进，不读取模型私有推理。
- Renderer、Build Worker、Validator 和 Publish Service 是独立工程组件，不注册为 V1 Agent 可自行调用的 Tool。
- Agent 验收权、自动修复条件和失败收敛规则见 [V1 多 Agent 设计：Agent 错误、验收与有限修复](./01-[Agent]-多Agent设计.md#8-agent-错误验收与有限修复)。

大语言模型（LLM）只允许输出领域协议：

- `LeadDecision`
- `Blueprint`
- `ProductSpec`
- `ArchitectureDesign`
- `AppSpec`
- `SourceBundle`
- `RevisionSpec`

LLM 不直接拥有 Shell、文件系统或 Publish Tool。Provider Adapter 只负责模型调用和响应提取；Pydantic 校验、事件、重试、业务 Session、配额和状态由应用 Runtime 与数据库管理。

Product Manager 的 Blueprint 必须包含：

```text
product_type: string
support_level: supported | adapted | unsupported
pages: string[]
modules: string[]
mapped_requirements: string[]
omitted_requirements: string[]
capability_policy_version: web-v1
```

`supported` 表示可以在自包含 Web Runtime 中完成；`adapted` 必须等待用户确认被替换的服务端、外部或原生能力；`unsupported` 不得进入 Architect 或 Build。

`unsupported` 的 `rewrite_suggestion` 必须保持原始产品目标、项目类型和目标平台，只能补充当前 Adapter 缺口或给出继续设计所需的信息，不能生成 Web 替代品、商品目录或其他产品。原 Run 保持终止且不创建 Build Job；只有用户提供的新方案确实落入现有 Adapter 能力并重新通过 Product Manager/Risk Policy 后，才能创建新 Run 进入后续阶段。

### 5.4 阶段产物契约

前端默认只展示 Lead 对话与当前结果；用户展开团队过程时，每个阶段必须绑定可检查产物：

| 阶段名称 | 实际执行 | 必须产生的产物 |
| --- | --- | --- |
| 团队负责人（Lead） | 团队负责人智能体判断直接回答还是调用完整团队 | 团队负责人决策（`LeadDecision`）、直接回复或团队摘要 |
| 产品经理（Product Manager） | 规范化需求、判断支持范围并生成完整产品文档 | 产品规格（`ProductSpec`）和产品蓝图（`Blueprint`）索引 |
| 架构师（Architect） | 把已批准产品文档转化为可实现、可验证的技术方案 | 架构设计（`ArchitectureDesign`）文档 |
| 工程师（Engineer） | 生成项目源码、单元测试和受控测试配置 | 应用规格（`AppSpec`）和源码包（`SourceBundle`） |
| 运行系统构建/测试（Runtime Build/Test） | 物化源码并执行固定构建和单元测试 | 构建任务（`BuildJob`）和执行报告（`ExecutionReport`） |
| 运行系统校验器（Runtime Validator） | 执行确定性产品、架构、安全和能力边界检查 | 校验报告（`ValidationReport`） |

团队展开视图统一使用“固定团队 · 分阶段接力”或“Sequential team pipeline”。不得使用“并行协作”“团队自主讨论”等文案。

### 5.5 运行时适配器（Runtime Adapter）与预览沙箱（Preview Sandbox）

本节只保留系统级关系；通用 Source Contract、版本化 Runtime Contract、校验分层、结果状态与 Web 兼容迁移以[通用源码与 Runtime 校验 Contract](./12-[工程][TODO]-通用源码与Runtime校验Contract.md)为事实来源。

V1 不按商品类型选择预制渲染器（Renderer）。工程师（Engineer）生成真实源码包（SourceBundle）与交付规格，运行系统（Runtime）校验主服务绑定的 Runtime Contract，并由满足该 Interface 的已登记 Adapter 负责确定性物化、构建、测试，以及 Contract 支持时的隔离预览：

- 只接受通过模式（Schema）校验且指纹一致的应用规格和源码包。
- 适配器声明允许的目录、入口、固定构建/测试命令和预览能力；源码包不能自带任意执行命令。
- 当前已实现网页适配器（Web Adapter）仍只向当前项目 Git 写入 `index.html`、`styles.css`、`app.js`、`app-spec.json` 和版本标记；这是迁移现状，不是通用源码契约的文件上限。
- 当前 `web-static-v1` 兼容路径把 HTML 解释为语义化 body Fragment；交互代码位于 JavaScript 字段，不能在 HTML 中夹带 script/iframe/object/embed。该规则不是通用 Source Contract，新的完整 Document Web Contract 使用独立标识，不复用含义模糊的 `html` 字段。
- 允许公网 HTTP(S)/WS(S) API 请求；禁止 localhost、127/8、`::1`、`0.0.0.0` 和用户设备本地服务。远程可执行资源、dynamic import、eval、运行时依赖安装和模型提供的命令字符串仍禁止。
- Preview 使用 `sandbox="allow-scripts"` 且不授予 `allow-same-origin`；CSP 只为 `connect-src` 开放 HTTP(S)/WS(S)，外部脚本、字体、媒体和表单提交仍关闭，并在 Preview 与物化 `app.js` 中安装 loopback 请求 Guard。
- 同一个源码包（SourceBundle）必须物化为语义一致的源码文件；绑定 Runtime Contract 的候选在构建、单元测试或强制校验失败时不得创建项目版本（ProjectVersion）。没有 Runtime Binding 的有效源码可以形成 `source_ready` 源码版本，但不能展示伪造的 Build/Test/Preview 证据。

V1 的结构化修改同步更新 AppSpec 和源码字段，再创建新版本。受限 Vim 的任意源码编辑仍要求独立 Linux Sandbox Host。

### 5.6 Build Job Runner

构建必须脱离用户 HTTP 请求异步调度，但实际工程执行位于共享独立执行服务：

```text
提交构建请求（POST）
      |
      v
写入 `build_jobs(status=queued)`
      |
      v
HTTP 202 + 构建任务标识（`build_job_id`）
      |
      v
主服务工作器领取任务租约
      |
      v
私有 HTTP -> 共享执行服务
               物化 -> 构建 -> 测试 -> 校验
       |
       v
主服务校验并持久化结果
```

V1 使用主服务中的持久化调度工作器（Worker）和无状态共享执行服务：

- `MAX_CONCURRENT_EXECUTIONS=1` 作为初始全局执行上限；
- 使用数据库状态比较并交换（CAS）与租约（lease）防止同一任务重复执行；
- 主服务重启后重新领取排队中（queued）或租约过期的任务；
- 主服务通过 Railway 私有网络发送源码包，执行服务使用固定适配器运行子进程；
- 每个任务设置可配置总时限、输出上限和临时目录上限；
- 禁止多个 Uvicorn 工作器无协调地消费同一队列；
- 执行服务不维护第二套持久化队列，不创建项目版本或发布。

当前代码仍采用一个 FastAPI 主服务进程、一个进程内构建调度工作器和一个 SQLite 事实源；运行期源码物化、固定构建、单元测试和确定性校验已经通过私有 HTTP 迁入[共享独立执行服务](./08-[工程][TODO]-共享独立执行服务.md)。本地测试环境允许使用同一执行器实现进行进程内验证，但生产配置必须将 `RUNTIME_EXECUTOR_URL` 指向独立服务。Railway 私网部署和真实多用户运行证据尚未完成，因此不能把本地实现描述为已部署能力。

V1 不引入 LISTEN/NOTIFY、外部消息队列、执行工作器集群或跨主服务实例的租约所有者（lease owner）。服务端事件流（SSE）继续从主服务数据库读取持久化事件。只有准备增加主服务或执行服务并发副本时，才重新设计通知、队列、所有者隔离（owner fencing）和共享阶段产物存储。

部署前必须在执行并发为一且真实构建和测试发生时观测：排队时间、分阶段耗时、执行服务 RSS/CPU、主服务 API p95、SSE 保活间隔、日志体积和临时目录清理结果。

目标协议：

```text
主服务 BuildJob Worker
        |
        | private HTTP + ExecutionRequest
        v
共享 Runtime Executor
        |
        `-> BuildArtifact + ExecutionReport + ValidationReport
```

源码事实保存在主服务的项目 Git。执行服务只接收源码包副本并返回有界构建产物与报告，不能写主服务卷或把自身绝对路径作为持久化引用。

### 5.7 Repository Service

Repository Service 是 Project 与源码历史的可信所有者：

1. 创建 Project 时同步创建唯一 `ProjectRepository(status=provisioning)`。
2. 在受控 repo root 下执行固定 `git init --bare`，写入平台生成的初始模板并创建初始化 commit，随后标记 ready。
3. Build/Edit/Resolve/Restore 都从指定 base commit 导出临时 worktree；通过校验后由 Repository Service 收集 allowlist 文件并提交。
4. Workspace 文件树通过只读 HTTP API 列出 Repository 文件和当前 Run Artifact；API 隐藏 `.git`、内部 worktree 与宿主机绝对路径，限制 UTF-8 文本大小，并复用 User/Project/Run 归属校验。Artifact 文件明确标记为未提交，不能与 Git working tree 状态混淆。
5. `ProjectVersion.git_commit_sha` 必须指向该 Project 仓库中的 commit；数据库 Version 和 Git commit 在同一业务操作中成功或进入可恢复失败状态。
6. V1 不配置 remote、credential helper 或 hook，不支持 clone 用户仓库、push GitHub、force push 或改写历史。

推荐受控目录：

```text
/srv/another-atom/
├── repos/{user_id}/{project_id}.git          trusted bare repo
├── sessions/{editor_session_id}/worktree     ephemeral, no .git
└── artifacts/{project_id}/{version_id}/      build/publish snapshot
```

路径全部由服务端根据 AuthSession 与 Project ID 生成，API 不接收绝对路径。每个 Project 同时最多一个写 lease；base commit 不匹配时保存返回冲突，不覆盖另一会话修改。

### 5.8 Terminal Service 与 Sandbox Manager

```text
xterm.js
   |
   | WSS + one-time editor token
   v
终端网关 -> 所有者/会话检查 -> 伪终端（PTY）代理
                                         |
                                         v
                                编辑沙箱（Editor Sandbox）
                                固定命令：Vim
```

- `POST /projects/{id}/editor-sessions` 校验登录用户、Project owner、Repository ready 和写 lease，返回短时一次性 WebSocket token。
- WebSocket 只转发 PTY input/output、resize 和 heartbeat，不传递任意 RPC；断线后 token 失效。
- Sandbox Manager 在独立 Linux 执行宿主机上启动 rootless 容器：非 root、只读根文件系统、禁网、drop capabilities、seccomp、`no-new-privileges`、资源/时限上限。
- Sandbox 只挂载没有 `.git` 的临时 worktree，固定启动受限 Vim；不提供登录 Shell、Docker socket、Secret、平台数据库或其他用户目录。
- `:write` 只改变临时 worktree。保存版本必须通过仓库服务（Repository Service）同步到主服务受信仓库，再由主服务调度共享执行服务完成固定构建和验证。
- 仅靠目录前缀、`chroot` 或 Vim restricted mode 不构成隔离；目标宿主机无法提供容器/namespace 隔离时，真实终端功能必须关闭。

### 6.1 事件协议

```json
{
  "event_id": "evt_123",
  "project_id": "project_123",
  "session_id": "session_123",
  "run_id": "run_123",
  "type": "build.started",
  "timestamp": "2026-07-11T00:00:00Z",
  "payload": {
    "build_id": "build_123"
  }
}
```

核心事件：

```text
run.started
llm.streaming
blueprint.generated
approval.required
role_stage.started
role_stage.completed
app_spec.generated
build.queued
build.queue_updated
build.started
build.cancelled
build.failed
validation.issue_detected
validation.completed
qa.degraded
preview.ready
run.failed
run.completed
run.completed_degraded
deployment.started
deployment.completed
deployment.failed
```

每条 SSE 消息使用数据库 `event_id` 作为原生 SSE `id`：

```text
id: evt_123
event: build.started
data: { ... }
```

事件先写入 `run_events`，再推送 SSE。浏览器断线重连时由 EventSource 发送 `Last-Event-ID`，服务端校验用户/Session/Run 归属后，从该游标之后补发；不能只依赖内存广播，也不能因重连重新触发 Agent 或 Build。

- 没有业务事件时，服务端默认每 15 秒发送 SSE comment keepalive；keepalive 不写入 `run_events`。
- V1 默认保留详细 `run_events` 30 天，保留天数通过配置管理；Run 终态、Artifact、Version 和错误摘要按 Project 生命周期保留。
- 清理任务只能删除超过窗口且所属 Run 已终止的详细事件，不删除运行中的游标范围。
- `Last-Event-ID` 已超出保留窗口时，服务端发送 `stream.reset`，前端先通过 Run/Build snapshot API 恢复当前状态，再从新游标继续。

## 7. 状态模型

### 7.1 Agent Run

```text
Created -> LeadRouting -> ProductRunning -> AwaitingProductApproval
             |                    |
             v                    v
           Failed          ArchitectRunning
                                    |
                                    v
                             EngineerRunning
                                    |
                                    v
                               BuildQueued
                                    |
                                    v
                                Building
                                    |
                                    v
                                Testing
                             |           |
                             v           v
                           Failed     Validating
                                      |       |
                                      v       v
                         EngineerRepairing  Completed / CompletedDegraded
                                  |
                                  `-> BuildQueued（最多一次）
```

### 7.2 Build Job

```text
Queued -> Leased -> Materializing -> Building -> Testing -> Validating -> Succeeded
   ^         |            |           |          |           |
   |         +------------+-----------+----------+-----------+--> Failed
   |
manual retry creates a new attempt
```

### 7.3 Publish

```text
Unpublished -> Publishing -> Live -> Updating -> Live
                    |                    |
                    v                    v
                  Failed               Failed

Live -> Unpublishing -> Unpublished
```

Build 成功不会自动发布。Publish 必须携带用户选择的 `version_id`。

公开访问与缓存约束：

- `public_id` 使用不可预测的随机标识，不暴露顺序 Project/Deployment ID；`/apps/{public_id}/*` 查不到或已 Unpublish 时统一返回 404。
- 发布快照不可变，Update 通过数据库事务切换 stable public route 指针，不原地覆盖旧快照。
- 公开入口 HTML 和发布元数据使用 `Cache-Control: no-store`；带 content hash 的静态资产可以使用 immutable cache。
- Unpublish 先原子更新 Deployment 状态并撤销路由，再异步清理快照；即使文件尚未删除，公开入口也不能继续命中。
- V1 不接 CDN。未来增加 CDN 时，Publish/Update/Unpublish 必须包含 purge/invalidation 结果，不能只修改数据库指针。
- Link Only 页面增加 `noindex`，但不可预测 ID 不能被描述为完整访问控制；带 token 和过期时间的访问控制属于未归属版本的后续候选。

## 8. 数据设计

### 8.1 存储分工

SQLite：

- 用户、AuthSession、安全审计、Plan、Subscription、Quota 和 Usage Ledger。
- Project、Session、AgentRun、AgentStageRun、Artifact、Approval、RunEvent、ProductEvent 和 BuildJob。
- ProjectRepository、EditorSession、Attachment 元数据、ProjectVersion 和 Deployment。

主服务持久化卷（Volume）：

- SQLite 数据库和上传附件。
- 每个项目（Project）的受信本地 Git 裸仓库（bare repository）。
- 主服务已验证并接收的构建产物、有界日志和发布版本不可变快照。

共享执行服务不挂载持久化卷。源码、测试、`dist/` 和运行日志只存在于单次执行的临时目录，终态结果返回主服务后清理。编辑沙箱宿主机（Editor Sandbox Host）只保留编辑会话所需的临时工作树（worktree），不作为项目 Git 或构建产物的事实源。

### 8.2 核心关系

```text
User
  |---- n AuthSession
  |---- n Project
  |         |---- 1 ProjectRepository ---- n EditorSession
  |         |---- n Attachment
  |         |---- n Artifact
  |         |---- n Approval
  |         |---- n Session
  |         |          `---- n AgentRun ---- n AgentStageRun
  |         |                    |---- n RunEvent
  |         |                    `---- n BuildJob
  |         |---- n ProductEvent
  |         |---- n ProjectVersion
  |         `---- n Deployment
  |
  |---- 1 Subscription ---- 1 Plan
  |---- 1 QuotaAccount
  `---- n UsageLedger
```

### 8.3 核心表

- `users`：唯一 username、password hash、平台用户状态；不保存明文密码。
- `auth_sessions`：哈希后的随机 Session token、user、过期时间、撤销时间和最近活动；浏览器只持有 Cookie。
- `security_events`：登录成功/失败、退出、越权拒绝和敏感操作审计。
- `plans`：周期配额与功能边界。
- `subscriptions`：当前方案、有效期和状态；V1 可由种子数据或管理操作设置。
- `quota_accounts`：可用、预占和已结算额度。
- `usage_ledger`：每次预占、结算和释放记录。
- `projects`：项目名称、owner user、状态和当前版本。
- `project_repositories`：Project 一对一本地 Git 仓库状态、默认分支、当前 commit 和受控 storage key。
- `editor_sessions`：xterm/Vim 会话、owner、base commit、worktree key、写 lease、状态和过期时间。
- `sessions`：项目下可恢复的模型上下文。
- `agent_runs`：一次需求或修改对应的 LLM 运行。
- `agent_stage_runs`：团队负责人（Lead）、产品经理（Product Manager）、架构师（Architect）、工程师（Engineer）各阶段的输入产物、输出产物、模型、用量、尝试次数和状态；历史数据分析师与质量评审员阶段仅供兼容读取。
- `artifacts`：不可变的产品规格（ProductSpec）、产品蓝图（Blueprint）、架构设计（ArchitectureDesign）、应用规格（AppSpec）、源码包（SourceBundle）、修订规格（RevisionSpec）、执行报告（ExecutionReport）和校验报告（ValidationReport）；历史数据分析（DataProfile）与质量评审报告（ReviewReport）仅供兼容读取。
- `approvals`：绑定精确 Artifact 版本与 hash 的 Human-in-the-loop 决策记录。
- `run_events`：可重放的 SSE 事件。
- `product_events`：用于价值漏斗的用户行为事件，与运行时 `run_events` 分离。
- `build_jobs`：异步构建状态、lease、attempt 和错误。
- `attachments`：附件元数据和 Volume 路径。
- `project_versions`：应用规格（AppSpec）、源码包（SourceBundle）、执行报告（ExecutionReport）、校验报告（ValidationReport）、构建产物引用和对应 `git_commit_sha`。
- `deployments`：公开 URL、所选版本和发布状态。

#### 8.3.1 身份与用户级租户字段

```text
users: id, username(unique), password_hash, status, created_at
auth_sessions: id, user_id, token_hash(unique), expires_at, revoked_at, last_seen_at
security_events: id, user_id?, event_type, outcome, request_id, created_at
```

所有业务查询从 AuthSession 得到当前 `user_id`。生产 API 不接受客户端指定 owner；Project、Repository、EditorSession、Version、Attachment、Run 和 Job 必须通过 owner join 或统一 Repository 层查询。V1 是 user-level tenancy，不在字段中伪装尚未实现的 org/team。

#### 8.3.2 ProjectRepository 与 ProjectVersion

```text
project_repositories:
  id, project_id(unique), user_id, provider=local_git
  default_branch, status, storage_key
  head_commit_sha, write_lease_owner, write_lease_expires_at
  created_at, updated_at

project_versions:
  ...
  git_commit_sha
  base_commit_sha
  repository_id
```

数据库不保存宿主机绝对路径，只保存由 Repository Service 解析的 `storage_key`。Git commit 成功但数据库事务失败时，通过 operation id 和 commit trailer 幂等补写；数据库成功但 commit 缺失时 Version 进入 failed，不允许发布。

#### 8.3.3 EditorSession

```text

base_commit_sha, worktree_key, mode=read_write|read_only
status=starting|ready|connected|dirty|saving|closed|expired|failed

created_at, last_seen_at, closed_at
```

WebSocket token 只使用一次并绑定 Session、User、Project 和过期时间。Terminal Gateway 每次连接重新校验 AuthSession 与 EditorSession；不能只凭可猜测 session id 建立 PTY。

## 9. 用量记录与订阅边界

V1 只实现 Provider 用量记录，不实现基于 Plan 的额度限制或支付。

```text
Plan -> Subscription -> Quota Account -> Usage Ledger
```

每个角色阶段及其结构化输出重试都作为独立模型请求计量：

```text
verify user and session
    -> reserve estimated quota in a database transaction
    -> call LLM
    -> read actual Provider request count and token usage
    -> settle actual usage
    -> release unused reservation
```

调用失败也必须记录实际发生的 Provider 请求并释放未使用预占。计量单位为 Provider 请求次数，输入/输出 token 作为审计字段保存。预占只解决并发调用的记账一致性，不读取 `quota_limit` 拒绝请求；Lead 和后续 Agent 即使超过该兼容字段也继续执行。

当前实现把“成功阶段 Artifact 写入”和该次 Provider 用量结算放在同一数据库事务中。非 LLM 异常同样读取已观测到的 Provider usage：实际请求已发生则结算实际次数，其余预占释放；请求尚未发生则全部释放。Worker 恢复时先释放中断阶段遗留预占，再从最后一个已提交 Artifact 继续。

V1 不接 Stripe。未来 Stripe Webhook 只更新 Subscription，不直接修改 Session 或历史 Ledger。

## 10. 工程错误与恢复

Agent Provider、结构化输出、Stage retry、Validation repair 和 Rework 的语义归 [V1 多 Agent 设计](./01-[Agent]-多Agent设计.md)管理。本章只定义 Build Worker、持久化、事件和发布基础设施如何失败与恢复。

### 10.1 Build 执行错误

- 编译、资源或超时错误先写入 `build.failed`，随后将聚合 Run 标记为 Failed，错误阶段为 BUILD。
- 固定输入导致的编译失败不自动重复执行，因为相同输入不会产生不同结果。
- Worker 被终止或 lease 过期时允许自动重新领取一次；这属于执行恢复，不是构建逻辑重试。
- 用户点击 Retry 后创建新的 attempt，并保留原日志。
- 错误码至少区分 `RENDER_FAILED`、`BUILD_FAILED`、`BUILD_TIMEOUT`、`RESOURCE_LIMIT`。

### 10.2 持久化与事件恢复

- 创建 Run、StageRun、Artifact、Approval、BuildJob 和 Event 的状态迁移必须使用数据库事务；事务失败不能发送成功事件。
- SSE 断线不改变 Run 状态；客户端按 `event_id` 重连并补取持久化事件，不重新触发 Agent 或 Build。
- Worker 只能领取 queued 或 lease 已过期的 Job；审批使用 `awaiting_approval -> build_queued` 条件更新 CAS，并以 `approvals.run_id`、`build_jobs.run_id` 唯一约束兜底，因此并发审批只有一个请求能创建 Approval 和 Build Job。
- 每个成功智能体（Agent）阶段以唯一阶段产物（Artifact）作为恢复检查点；阶段产物与该阶段用量结算同事务提交。恢复时复用已有产品规格（ProductSpec）、架构设计（ArchitectureDesign）、应用规格（AppSpec）、源码包（SourceBundle）、执行报告（ExecutionReport）和校验报告（ValidationReport），不重复结算已提交阶段。
- 若 Run 已完成但 Worker 在 Job 收尾前退出，重新领取逻辑只把 Job 对齐为 succeeded/failed，不再次执行 Pipeline；Build 版本按 `run_id + source=build` 复用，避免重复 ProjectVersion。
- 外部 Provider 不提供幂等键时，进程在“Provider 已处理请求、Artifact 事务尚未提交”的极窄窗口崩溃，重启后仍可能重新发起一次 Provider 请求；V1 能保证数据库 Artifact、结算和版本不重复，不能无依据声称外部调用 exactly-once。
- SQLite 元数据与 Volume 产物不一致时，状态进入 Failed/Needs input，不允许 Agent 猜测或重新生成路径。
- Publish/Update 失败保留上一个 Live Version，不能让稳定公开 URL 指向不完整产物。

### 10.3 错误归属边界

| 错误类型 | 事实来源 | 处理所有者 |
| --- | --- | --- |
| Provider、Prompt、Pydantic 输出 | Agent StageRun / Provider response | Agent Runtime，见 V1 Agent 设计 |
| AppSpec 可修复校验问题 | ValidationReport | Agent Orchestrator，见 V1 Agent 设计 |
| 渲染器（Renderer）、编译、测试、超时、资源 | 执行报告（ExecutionReport）/ 执行进程退出 | 共享执行服务 |
| 主服务工作器中断、租约（lease）过期 | SQLite Job/lease 状态 | 主服务构建工作器（Build Worker）恢复逻辑 |
| 数据库、Volume、SSE、Publish | 服务与存储状态 | API/Storage/Publish Service |

Agent 只能消费工程 Runtime 产生的错误证据，不能修改错误码、伪造成功事件或决定基础设施重试。

## 11. 安全边界

V1 的模型没有任意执行权限。必须满足：

- Gateway 只从已验证 AuthSession 建立 `user_id`，生产环境拒绝用 `X-User-ID` 冒充身份。
- 登录接口使用限速、统一失败文案和安全事件审计；密码、Session token 和 Cookie 不进入日志。
- AppSpec、RevisionSpec 和附件引用全部经过 schema 与资源归属校验。
- Project、Repository、EditorSession、Version、Job 和 Preview 必须经统一 owner 查询，不允许租户资源裸 `db.get(id)`。
- 可信 Git bare repo、临时 worktree 和发布 Artifact 使用不同根目录；Sandbox 永远看不到 bare repo 与 `.git`。
- Renderer 只写当前 Project 的受控源码目录，并拒绝网络访问、动态依赖和宿主 Shell 能力。
- 运行时禁止安装依赖、修改依赖清单和执行任意 Shell。
- 构建子进程使用固定命令、超时、输出和磁盘上限。
- 云端用户不能读取其他用户的 Project、Repository、附件、worktree、日志、EditorSession 或 Preview。
- Terminal WebSocket 使用短时一次性 token；PTY 固定启动 Vim，不启动宿主 Shell。
- Sandbox 非 root、禁网、无 Secret、无 Docker socket，并使用 capability/seccomp/cgroup/生命周期限制。
- LLM API Key 只存在于 Railway 服务端环境变量。

当前纵切尚未达到第一条目标边界：测试环境允许按 `X-User-ID` 自动创建隔离账户；非测试环境只接受已预置用户，未知 ID 返回 401，不再自动获得满配额。已有 ID 仍可被请求头冒用，因此 Railway 公网验收前必须完成 Session Gateway；本轮加固不能被描述为完整认证。
- Preview 响应设置隔离策略，不允许生成应用访问平台管理 API。
- 发布前检查文件类型、总体积和入口文件。
- 配额预占使用数据库事务。

如果未来开放任意 Shell、依赖安装或网络访问，必须升级为更强的每次运行容器/VM 策略与审批，不能在当前受限 Vim Sandbox 中逐项放宽。

### 12.1 镜像与进程边界

镜像构建阶段：

1. 主服务镜像安装 Studio 构建依赖，构建 React Studio，并安装 Python API、智能体（Agent）和数据库依赖。
2. 执行服务使用独立最小镜像，预装已登记运行时适配器（Runtime Adapter）所需的固定工具链、构建器、测试运行器和校验器。
3. 执行服务镜像不包含平台数据库凭证、模型密钥、Git 凭据、发布密钥或容器管理 socket。
4. 不在运行时安装依赖，也不执行模型或用户提供的 Shell 命令。
5. 启用 WebIDE 时，编辑沙箱（Editor Sandbox）使用另一独立最小镜像，不与执行服务共享运行时。

依赖锁定与升级：

- Studio 提交 lockfile，镜像构建使用 lockfile 的冻结安装模式；lockfile 与声明不一致时构建失败。
- Python 运行依赖同样锁定精确版本，镜像不能在构建时解析浮动最新版本。
- Studio 依赖升级使用独立变更：更新 lockfile -> 安全检查 -> Golden Path 构建/交互回归 -> 镜像压测 -> 合并。
- 常规依赖更新在计划窗口执行；高危漏洞可以进入紧急升级，但同样不能跳过构建与 Golden Path 回归。
- ProjectVersion 记录生成源码与 Artifact 版本，旧版本恢复仍能定位其对应输入与 commit，不能静默改写历史。

启动阶段：

1. 先启动共享执行服务，检查工具链和健康接口。
2. 在主服务持久化卷（Volume）上初始化或检查 SQLite 模式（schema）与 Project Git 目录。
3. 启动单实例 FastAPI 和持久化构建工作器（Build Worker）。只有执行服务健康时，工作器才可从 SQLite 领取构建任务（BuildJob）并通过私有接口派发。

### 12.2 SQLite 与持久化 Volume

Railway 将 `/app/data` 挂载为持久化 Volume，SQLite 数据库与当前单实例 Project Git 都写入该目录。没有 Volume 时，容器重建后文件可能丢失，因此这不是可选配置。

服务重启或重新部署后，SQLite 与 Repository/Version 状态必须保持一致。V1 通过单副本约束避免多个进程同时写一个 SQLite 文件；不得为了扩容直接增加 Railway 副本。

V1 明确采用**单主服务实例 + 单共享执行服务 + 主服务单持久化卷**，不承诺水平扩展。启用 WebIDE 时另增单个 Linux 编辑沙箱宿主机。受信 Git 裸仓库是用户源码事实，已验收构建产物和发布快照由主服务持久化；执行服务的临时目录和编辑沙箱的工作树只是可回收副本。

扩展路径：

1. 优先把发布快照和带 hash 的静态资产迁移到 S3-compatible Object Storage。
2. 共享执行服务提高并发前，把构建产物（BuildArtifact）交换迁移到对象存储或受控产物传输。
3. 增加水平副本前，将 SQLite 迁移到 PostgreSQL，由 PostgreSQL 保存 Artifact 元数据、hash、对象 key 和发布指针。
4. 完成迁移前不得增加 Web/Worker 水平副本并假设 Volume 可共享。

### 12.3 平台源码 GitHub 与用户 Project Git

`github.com/eastonsuo/another-atom` 只负责平台自身源码、文档和部署触发。用户项目（Project）的源码保存在主服务持久化卷中的本地 Git 仓库，不配置 GitHub 远端（remote），也不使用平台仓库承载用户代码。执行服务和编辑沙箱只接收单次任务副本。GitHub Pages 不能运行 FastAPI、智能体（Agent）、SQLite 持久化服务、终端网关（Terminal Gateway）或沙箱管理器（Sandbox Manager）。

## 14. V1 实施顺序

1. 定义团队负责人决策（LeadDecision）、产品规格（ProductSpec）、产品蓝图（Blueprint）、架构设计（ArchitectureDesign）、应用规格（AppSpec）、源码包（SourceBundle）、执行报告（ExecutionReport）、校验报告（ValidationReport）、事件（Event）、导出（Export）和错误（Error）契约。
2. 建立 SQLite/SQLAlchemy 持久化，并实现 username/password、AuthSession Gateway 和双用户 Project 隔离测试。
3. 建立 ProjectRepository、EditorSession、ProjectVersion↔commit 模型和本地 Git Repository Service。
4. 实现 Plan、配额预占和 Usage Ledger。
5. 实现团队负责人（Lead）的 `direct/team` 路由、固定专业团队、风险策略、阶段持久化和配额结算。
6. 实现架构设计（ArchitectureDesign）文档持久化、通用源码包（SourceBundle），并在 Railway 同一项目与环境建立共享执行服务，实现已登记运行时适配器（Runtime Adapter）、固定构建/单元测试，以及执行报告（ExecutionReport）和校验报告（ValidationReport）。
7. 建立 Linux 编辑沙箱宿主机、无根编辑沙箱（rootless Editor Sandbox）、终端网关（Terminal Gateway）与 xterm.js/Vim。
8. 实现 Visual Studio 的 Lead 对话、Artifact 检查、SSE、WebIDE 和 iframe Preview。
9. 实现附件、ProductEvent、Export、Follow-up、Resolve、Restore 和显式 Publish。
10. 完成主服务 + 共享执行服务 + Linux 编辑沙箱宿主机的部署、备份和故障恢复。
11. 完成 API/SSE、终端并发、Sandbox 逃逸边界、CPU/RSS、构建、Git 一致性和重启恢复压测。

## 15. V1 验收

- 两个用户名密码账号可分别登录；切换账号后 Project、Repository、Version、EditorSession、事件和配额互不可见。
- 团队负责人（Lead）对每条消息只输出 `direct/team`；`direct` 只回答或澄清，`team` 按产品经理（Product Manager）、架构师（Architect）、工程师（Engineer）固定顺序执行。新运行不调用数据分析师（Data Analyst）或质量评审员（Reviewer）。
- 架构师必须读取已批准产品规格（ProductSpec）并生成代码库中的架构设计（ArchitectureDesign）文档；工程师必须读取两份文档并交付应用规格（AppSpec）、源码包（SourceBundle）和单元测试。
- 运行系统（Runtime）使用固定适配器执行构建（Build）和测试（Test），再由校验器（Validator）产生校验报告（ValidationReport）；只有全部强制检查通过才创建项目版本（ProjectVersion）。
- UI 默认只呈现 Lead 对话，团队过程可展开检查，不声称动态角色选择、并行或自主仲裁。
- 任意软件产品需求都必须保留原始项目类型和目标平台；只有存在匹配 Source/Build/Runtime Adapter 的方案才能进入 supported。缺少适配器或关键能力时进入 adapted/unsupported，且不得改写成 Web；unsupported 不创建 Build Job。
- `supported` 和 `adapted` 产品规格都必须绑定当前内容指纹由用户确认；额外预算、范围变化、破坏性仓库操作和线上变更另行触发风险确认。
- 当前网页应用规格（Web AppSpec）可以包含 HTML/CSS/JavaScript，但不能增加依赖、网络访问、构建命令或扩大工作区边界；通用源码包（SourceBundle）不能被该三文件实现限制，其他项目类型在对应契约（Contract）和适配器（Adapter）落地后必须拥有各自校验规则。
- HTTP 创建构建后立即返回，构建由异步 Worker 完成。
- 同一实例的构建并发不超过配置上限，初始为 1。
- 每个 Project 唯一绑定一个 local Git repository；Build/Edit/Resolve/Restore 版本均映射到可验证 commit SHA。
- xterm.js 只能连接当前用户当前 Project 的 Editor Sandbox；Sandbox 看不到 `.git`、Secret、宿主 Shell或其他用户路径。
- Vim 修改先形成脏工作区（dirty worktree）；保存版本（Save Version）完成构建（Build）、测试（Test）、校验（Validation）和提交（commit）后才生成项目版本（ProjectVersion），失败不移动当前版本。
- 构建运行期间，异步创建/查询 API p95 仍满足 1 秒返回基线，首条用户可见事件在 2 秒内到达，SSE keepalive 间隔不超过配置值。
- 压测报告记录队列等待、CPU、RSS 和构建耗时；若单进程方案不满足基线，部署必须改为独立 Worker Service。
- 用户可以恢复两个不同 Session，且上下文不会串线。
- 并发模型调用的实际请求次数必须完整记入同一账户，不能重复或遗漏结算。
- 附件元数据与 Volume 文件状态一致。
- Build Error 与预设 Resolve 问题使用不同事件和状态。
- 用户明确选择版本后才能 Publish。
- Railway 重启后 queued Job、项目、版本和发布结果可以恢复。
- SSE 使用 `Last-Event-ID` 补取；游标过期时通过 snapshot resync 恢复，不重复执行 Run。
- Unpublish 后原公开入口返回 404，Specify/Latest 指针与缓存策略不会继续暴露旧入口。
- 部署后获得可公开测试的 HTTPS 地址。
- Golden Path 连续 5 次完成，产品漏斗事件完整率为 100%，跨 Session 串事件为 0。
- Export JSON 满足最小 Contract，且不包含密钥、凭证、绝对路径、原始对话和内部用量流水。

## 16. 未归属版本：CC 式本地执行

该独立方向是类似 Claude Code 的本地 Agent Runtime，当前不占用 V1 或 V2 的版本定义：

```text
Terminal CLI
     |
     v
Local Python Agent Runtime
     |
     ├── 读取和修改本地项目文件
     ├── 执行受控 Shell / Git / npm
     ├── SQLite Session 与版本索引
     └── 启动 localhost Dev Server
                    |
                    v
             Local Visual Studio

Local Runtime ---- HTTPS ----> Cloud Auth / Quota / LLM Gateway
```

后续实现必须重新设计：

- Python/Node 安装与跨平台打包。
- 本地权限审批和工作区信任模型。
- 本地进程、端口和 Dev Server 生命周期。
- SQLite 项目如何上传、同步或发布到云端。
- 本地项目与 Cloud Project 的冲突处理。

在这些协议确定前，V1 README 和产品界面不得声称已经支持用户电脑上的本地仓库执行。V1 的服务端 local Git 与云端受限 Vim WebIDE 不等同于该方向。

## 17. V1 工程边界与演进取舍

本节承接原 README 中的 V1 工程取舍，并与前文实现、部署和验收章节形成索引关系。

- **[单副本 Control Plane]** V1 只面向本地单实例或 Railway 单副本，因此不实现 API/Worker 水平副本、跨实例分布式锁或 lease-owner fencing。
- **[主服务调度]** 产品蓝图（Blueprint）继续使用主服务进程内任务；构建任务由主服务单工作器与持久化任务/租约（Job/Lease）调度，再派发到共享执行服务。V1 不引入第二套消息队列或 PostgreSQL `LISTEN/NOTIFY`。
- **[数据库轮询事件]** SSE 复用读取 Session 轮询持久化事件，避免为单副本引入消息总线；Event Contract 保留未来多实例广播所需的序列与恢复语义。
- **[服务端 Session]** V1 使用用户名密码、随机服务端 Session、HttpOnly/SameSite Cookie 和用户级 Project 隔离，不扩展 JWT/OAuth、Organization、Membership 或共享 Project 权限。
- **[服务端 local Git]** 一个 Project 对应一个服务端仓库，并映射 commit/ProjectVersion；V1 不实现 GitHub/GitLab OAuth、remote、push/pull、SSH key 或用户电脑仓库操作。
- **[共享执行服务]** 所有用户共享一个无状态 Railway 执行服务，V1 全局并发为一；该边界只允许固定受限适配器，不实现每任务容器、微型虚拟机（MicroVM）或任意代码强沙箱。
- **[独立编辑沙箱宿主机]** 编辑会话使用临时工作树（worktree）和受限 Vim；V1 不实现多租户沙箱池或容量调度，无根（rootless）、磁盘和逃逸边界必须在目标 Linux 宿主机实机验收。
- **[应用内配额]** V1 记录 Provider 请求和 token，使用事务防止并发透支；Stripe、Wallet、充值、发票和 Edit/Restore 的商业计费策略不属于当前范围。
