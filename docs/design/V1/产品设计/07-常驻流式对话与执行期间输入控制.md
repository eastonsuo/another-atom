## 背景

当前 Workspace 已能展示 Project 对话、角色阶段和 Provider 生命周期事件，但执行过程仍主要表现为进度页和日志：进入 PM、Architect 或 Engineer 后，对话输入区可能被阶段界面替代；模型的真实可见回复没有持续进入同一条 Chat，用户只能通过日志判断系统是否仍在工作。

本设计把 Project Chat 固定为用户与 AI 团队协作的主界面。对话框始终保留；同一轮工作尚未结束时禁止发送新的独立消息；PM 补充、产品方案修改和确认继续使用同一条对话时间线，但根据当前状态形成不同的提交语义。

## 摘要

- **常驻对话框**
  - Workspace 始终保留同一个 Project Chat 输入框，不再为 PM 补充、失败重试或代码修改创建并列输入区。
- **单活动轮次**
  - 同一 Project 同一时刻只允许一个活动对话轮次或执行 Run；活动期间输入框可保留草稿，但发送按钮和 Enter 发送均禁用。
- **状态决定提交语义**
  - 空闲时发送新消息，`needs_input` 时回答当前 Human Task，ProductSpec 待确认时提交 PM 修改要求；这些动作不能被后端当成同一种“新建 Run”。
- **对话与执行分层**
  - 模型的用户可见回复以消息增量流式进入 Chat；文件、Diff、Artifact、构建和校验显示为操作卡片；Provider 和调试事件保留在日志层。
- **PM 不建立第二套交互**
  - PM 工作时禁发，PM 提问时用同一个输入框回答，ProductSpec 待确认时通过确认卡批准或通过 Chat 提出修改，两者使用并发控制避免冲突。
- **桌面工作区分区滚动**
  - Project Workspace 固定在顶部栏以下的可视区域内；阶段流水线、主阶段内容、对话历史和右侧文件工具分别管理滚动，不由浏览器页面整体滚动。

## 1. 目标与边界

### 1.1 目标

1. 用户在首次需求、产品方案、架构、工程、构建、失败和继续修改阶段始终知道下一条消息从哪里发送。
2. 模型工作期间持续呈现用户可读的对话内容，而不是只显示循环动画或技术日志。
3. 同一 Project 不因重复点击、多标签页或模型仍在执行而创建并发对话轮次或相互竞争的 Run。
4. PM 的补充信息、方案修改和用户批准保留各自业务含义，但统一显示在 Project Chat 中。
5. 页面刷新、SSE 重连和服务重启后能够恢复同一条消息、活动状态、草稿和等待用户对象。

### 1.2 不在本设计中引入

- 不支持把新消息排队到当前 Run 之后自动执行；V1 直接禁止发送。
- 不展示模型私有推理过程、Chain of Thought、未完成 JSON 或未校验 Tool 参数。
- 不因为界面变成对话形态而取消 ProductSpec Approval、Runtime Validation、Artifact 或版本边界。
- 不允许用户消息在 Build、Test 或版本写入期间隐式改变当前 Run 的目标。
- 不改变 V1 的固定角色顺序，也不在本功能中引入 V2 动态 TaskGraph。

## 2. 页面结构

Workspace 的主信息结构固定为：

```text
Project Chat 时间线
  ├─ 用户消息
  ├─ AI 团队可见回复（流式）
  ├─ PM 问题与 ProductSpec 确认卡
  ├─ 执行操作卡：读取 Context / 生成文件 / Build / Test / Validate
  └─ 结果卡：Artifact / Diff / Preview / Version / Failure

常驻对话输入框
  ├─ 当前提交语义提示
  ├─ 草稿
  ├─ 发送或回答按钮
  └─ 活动轮次的停止入口

辅助区域
  ├─ Project 文件与 Artifact
  ├─ Preview / Diff / Version
  └─ Run 日志与 Provider 调试事件
```

对话时间线是默认主界面。技术日志不再承担“AI 正在说什么”的职责，只用于展示 Provider、重试、超时、配额和排障证据。

### 2.1 滚动边界

桌面端 Project Workspace 使用固定的可视区边界：顶部栏占据固定高度，下面的三栏工作区只占剩余高度，不能由对话区或阶段状态继续撑高 `body`。

- 左侧阶段流水线独立纵向滚动，切换主内容时不丢失当前阶段位置；
- 中间列由常驻 Project Chat 和阶段内容组成，对话历史在自身区域滚动，阶段文档、状态或 Preview 在剩余区域滚动；
- 右侧 Project 文件、Vim 和日志沿用自己的文件树、预览或日志滚动容器；
- 桌面端页面外层不承担 Project Workspace 的内容滚动，避免三栏顶部和当前位置相互错位；
- 窄屏布局需要把右侧工具区堆叠到主内容下方，因此继续使用自然页面滚动，不强行保留桌面三栏的固定高度。

## 3. 常驻输入框

### 3.1 始终可见

输入框在以下状态中均不得从页面移除：

- Lead 正在判断；
- PM 正在整理或重新生成 ProductSpec；
- 等待 PM 补充；
- 等待 ProductSpec 确认；
- Architect、Engineer 正在工作；
- Runtime 正在 Build、Test 或 Validate；
- Run 成功、失败、取消或等待继续。

阶段页面、角色动画、产品方案卡和 Preview 只能出现在对话时间线或辅助区域，不能替换输入框。

### 3.2 活动期间禁止发送

V1 将“正在工作”定义为当前 Project 存在尚未结束的对话轮次或执行 Run，而不只限于正在接收模型 Token。以下阶段均禁止发送新的独立消息：

- Lead routing；
- PM、Architect、Engineer 模型请求与 Contract 修正；
- Runtime Build、Test、Validate；
- Artifact 保存、候选版本创建和恢复处理。

输入框可以继续编辑草稿，但：

- 发送按钮禁用；
- Enter 不触发提交；
- 页面显示当前角色或 Runtime 状态，以及“完成后可以继续发送”；
- 用户草稿不得因阶段变化、刷新或 SSE 重连丢失；
- 如果当前阶段允许停止，输入框旁提供“停止”，停止成功后再恢复发送。

禁发必须由后端状态保证。前端禁用只用于即时反馈；来自旧页面、多标签页或重复请求的提交仍必须被拒绝，并返回当前活动轮次或 Run 供页面刷新。

## 4. 输入框状态与提交语义

| Project/Run 状态 | 输入框 | 提交语义 | 是否创建新 Run |
| --- | --- | --- | --- |
| `idle / completed / failed / cancelled` | 可发送 | 新 Project 消息，交给 Lead 路由 | 只有 Lead 后续形成并获批执行任务时才创建 |
| `routing / generating / executing / building / validating` | 保留草稿，禁止发送 | 无 | 否 |
| `needs_input` | 可发送 | 回答当前 `input_request`，恢复原 Run | 否 |
| ProductSpec `awaiting_approval` | 可发送修改要求 | 明确要求 PM 基于当前 generation 修改并重新生成 | 否，恢复当前产品阶段并创建新 generation |
| 其他 `awaiting_approval` | 输入框保留 | Approval 通过卡片处理；普通消息不得代替批准 | 否 |

输入框必须展示当前语义，不能让用户猜测：

- 空闲：`询问项目，或描述你希望修改的内容`；
- PM 补充：`回答产品经理的问题`；
- ProductSpec 待确认：`告诉产品经理需要调整什么，或使用上方按钮确认`；
- 执行中：`工程师正在工作，完成后可以继续发送`。

## 5. PM 阶段如何避免冲突

### 5.1 PM 正在生成

PM 生成 ProductSpec 或根据修改要求重新生成时，当前轮次处于活动状态：

- PM 的用户可见内容流式进入同一条消息；
- 输入框保留草稿但禁止发送；
- ProductSpec 卡显示生成状态，不提供可重复触发的确认操作。

### 5.2 PM 等待补充

PM 缺少会实质改变交付结果的信息时创建 `input_request`：

1. PM 问题作为对话消息和 Human Task 卡出现在时间线中；
2. 同一个输入框恢复发送，并明确标记为“回答产品经理”；
3. 用户提交绑定当前 `human_task_id`，原任务原子变为 `answered`；
4. 原 Run 恢复 PM 阶段，不创建第二个 Project 或 Run；
5. PM 恢复后输入框再次进入禁发状态。

如果是结构化选择，选项作为消息内卡片呈现；输入框仍保留。选择未提交前不进入下一步，提交后与普通回答遵循相同的单活动轮次规则。

### 5.3 ProductSpec 待确认

ProductSpec 生成完成后，时间线展示简介、完整 Markdown 入口和确认卡：

- “确认并进入技术设计”只批准当前 generation 的内容指纹；
- 用户在 Chat 中发送修改要求，视为一次明确的 PM 重新生成请求；
- 修改请求原子地将当前 Approval 置为 `stale`，并以当前简介、Markdown、上一 generation、原始需求和本条修改要求作为重新生成输入；
- PM 开始重新生成后禁发；成功后创建新的 generation 和新的 Approval subject。

用户直接编辑简介或 `docs/product-spec.md` 时，仍遵循现有 ProductSpec 设计：内容进入 `needs_regeneration`，由用户明确点击重新生成。Chat 中发送修改要求本身已经是一次明确的重新生成授权，不再要求用户重复点击第二个按钮。

### 5.4 确认与修改并发

“确认并进入技术设计”和“发送 PM 修改要求”可能来自不同标签页。后端必须对 Approval 状态和 ProductSpec generation 使用 CAS：

- 只有一个操作可以从当前 `pending` 状态成功迁移；
- 确认先成功时，后续修改请求不能继续改写已批准对象，应提示阶段已变化并刷新；
- 修改先成功时，旧 Approval 已 stale，后续确认必须失败并打开新的 generation；
- 失败的并发请求不能创建额外 Run、重复 Provider 调用或重复配额结算。

## 6. 流式对话与执行反馈

### 6.1 对话中展示什么

模型流式进入 Chat 的内容只包含用户可见信息：

- 对当前任务的简短理解；
- 当前准备处理的对象或范围；
- 需要用户决定的问题；
- 已完成工作的摘要；
- 失败后可继续的具体信息。

不得流式展示：

- Chain of Thought 或隐藏推理；
- 未完成 JSON；
- 未校验代码或 Tool 参数；
- Provider 原始响应体；
- 密钥、绝对路径或其他用户的 Context。

### 6.2 操作卡与 Artifact

读取 Context、生成文件、应用 Patch、Build、Test 和 Validate 以操作卡呈现。操作卡展示真实状态和证据引用，不伪装成模型自然语言：

```text
工程师：我会先检查目标语言选择器和初始化逻辑。

[正在读取当前版本代码]
[正在生成候选修改：2 个文件]
[正在运行测试]

工程师：目标语言选项初始化已经修复，现有翻译流程保持不变。
[查看 Diff] [查看测试结果]
```

代码和文档内容进入右侧文件、Diff 或 Artifact 视图。生成中的候选内容可以标记为“未完成/未应用”，但只有完整 Contract 通过并由 Runtime 处理后才能进入 Preview 或版本。

### 6.3 日志位置

Provider 请求开始、首 Token、timeout、fallback、Contract 修正和租约续期继续保存为 Run Event，并显示在可展开日志中。主对话只在这些事件影响用户选择或结果时生成简洁错误卡，不把全部事件逐条复制成聊天消息。

## 7. 消息恢复、停止与失败

### 7.1 流式消息恢复

每条 AI 可见消息具有稳定 Message ID 和以下状态：

- `streaming`：正在接收用户可见增量；
- `completed`：消息已完整结束；
- `failed`：模型或协议失败，保留已确认可见的部分内容；
- `interrupted`：用户停止或 Worker 中断，保留部分内容并明确标记未完成。

服务端按有界批次合并持久化可见消息，SSE 只负责低延迟投影。刷新或重连后从持久化消息和 Run Event 恢复，不能依赖浏览器内存拼接完整回答。

### 7.2 停止

用户停止当前轮次后：

1. Runtime 记录取消请求并停止后续 Provider、Build 或版本写入；
2. 当前可见消息标记为 `interrupted`；
3. 未通过 Contract 和 Runtime 校验的候选内容不能成为新版本；
4. 停止完成后释放 Project 活动轮次，恢复发送；
5. 用户下一条消息可以引用已保留的错误、Artifact 或部分结果，但仍以上一个成功版本为基线。

### 7.3 失败

模型、Contract、Build 或 Validation 失败后，主对话展示：

- 失败发生在哪个用户可理解的阶段；
- 已经保留哪些文档、代码候选或测试证据；
- 当前版本是否变化；
- 用户可以继续说明、重试还是调整范围。

技术错误详情继续进入日志。失败进入终态后恢复发送，不保持无期限禁用。

## 8. 并发与一致性约束

1. 同一 Project 同一时刻只有一个 active turn；active turn 可以关联零个或一个 Run。
2. 每次提交携带消息幂等键、当前 Project 状态版本，以及存在时的 `human_task_id` 或 ProductSpec generation/Approval 引用。
3. 后端根据持久化状态决定提交语义，不能相信前端自报“这是新消息”或“这是 PM 回复”。
4. 活动期间收到普通消息提交时，不排队、不创建新 Run，返回当前 active turn 和可恢复状态。
5. Human Task 回答、Approval 决策、PM 修改请求和 Run 恢复使用事务与 CAS；部分成功必须回滚或进入可恢复的明确状态。
6. Project 写占用、对话 active turn 和 BuildJob lease 是不同对象，但状态必须互相校验，不能用前端禁发替代后端单写保护。

## 9. 实现状态与剩余差距

### 9.1 2026-07-15 已实现

- Workspace 在 ProductSpec 确认、PM 补充、执行中、完成、失败和取消状态下复用同一个 Project Chat；阶段内容不再移除输入框。
- 输入框根据 Run 状态决定提交语义：`needs_input` 回答原 Human Task，ProductSpec `awaiting_approval` 请求 PM 基于当前文档重新生成，终态发送普通 Project 消息，其余状态禁发。
- 草稿按 Project 写入浏览器本地存储，阶段变化和刷新不清空；执行中发送按钮禁用并显示当前阶段说明。
- 普通 Project 消息在后端先检查非终态 Run，并使用 Project 级持久化 active turn 锁阻止多标签页并发 Lead 请求；遗留锁超过 Agent 阶段时限后允许恢复。
- ProductSpec Chat 修改使用 Run 与 Human Task CAS：修改先成功时旧 Approval Task 变为 `stale`，PM 以原始需求、当前 Markdown 和本条修改要求为输入生成新 ProductSpec，并创建新的 Approval Task；确认先成功时修改请求失败。
- PM、Architect 和 Engineer 的新建主链路采用 `{message, result}` 双通道结构化响应：`message` 以稳定 Message ID 增量写入 Project Message，`result` 继续完整缓冲并通过原 Contract 校验。
- 用户可见消息按有界字符批次持久化，并通过既有 Run SSE 事件触发 Studio 重放；日志仍保留 Provider、重试、Build 和 Validation 事件。
- Project Message 已区分 Product Manager、Architect、Engineer、Lead、User 和 System 角色，并保存 `streaming / completed / failed` 状态。

### 9.2 尚未完成

- Stop/Cancel 入口和 `interrupted` 消息状态尚未实现。
- 消息幂等键和显式 Project 状态版本尚未加入提交协议；当前依靠 active turn、Run 状态和 Human Task CAS 防止并发执行。
- Lead 的 Project 问答仍为请求完成后一次性写入消息；当前只有 PM、Architect、Engineer 主生成链路提供用户可见增量。
- Context、文件生成、Diff、Build、Test 和 Validate 尚未全部转成主对话内的操作卡，部分证据仍只在阶段视图、文件区和日志中展示。
- 浏览器内真实交互和部署环境 SSE 尚待验收；当前证据是后端集成测试、Provider 流式测试及 Studio TypeScript 生产构建。

## 10. 验收标准

1. Project Workspace 在 Lead、PM、Architect、Engineer、Build、Validation、成功和失败状态下始终显示同一个输入框。
2. 活动轮次期间发送按钮和 Enter 发送禁用；直接调用 API、多标签页和重复提交也不能创建第二条活动消息或第二个 Run。
3. 用户在禁发期间输入的草稿在阶段变化、刷新和 SSE 重连后仍然保留。
4. PM `needs_input` 通过同一个输入框回答，回复绑定原 Human Task 并恢复原 Run，不出现独立补充页面。
5. ProductSpec 待确认时，确认卡批准当前 generation；Chat 修改要求使旧 Approval stale 并触发基于现有内容的新 generation。
6. 确认和修改请求并发时只有一个成功，失败操作不产生重复 Provider 请求、配额或 Run。
7. AI 的用户可见回复以同一 Message ID 流式更新；刷新后可以从持久化内容恢复，失败或停止时明确标记未完成状态。
8. Provider、重试、Build 和 Validation 事件默认显示为操作卡或日志，不取代主对话，也不将日志逐条复制为模型回复。
9. Chat 不展示 Chain of Thought、未完成 JSON、未校验代码、密钥或跨 Project Context。
10. 当前轮次成功、失败、取消或进入需要用户回答的状态后，输入框按照状态恢复正确的提交能力和提示文案。
