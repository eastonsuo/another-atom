## 背景

首次生成后，现有结构化 Edit、Vim 和重新构建不能让固定团队理解当前 Project、基线版本和本轮修改目标。用户需要在同一个 Project 中继续对话，并让修改基于已有代码而不是原始 Prompt 重新生成。

## 摘要

- **修改基线**
  - 每轮修改由 Runtime 自动从当前 ProjectVersion 和 Git commit 建立基线，用户不需要手动选择 Snapshot 或源码文件。
- **对话归属**
  - 用户消息、澄清、修改任务、Run 和结果版本都归属于当前 Project，并在刷新或重启后可恢复。
- **执行方式**
  - Lead 梳理任务后交给固定下游团队，默认保留未要求变化的页面、交互和视觉。
- **变更证据**
  - 候选代码与基线代码由 Runtime 确定性比较，向用户展示真实 Diff，并通过 Validator 和质量复核。
- **版本与失败边界**
  - 成功修改创建新的 ProjectVersion 与 Git commit；冲突、失败或取消不覆盖原版本和已发布版本，风险变化继续由用户确认。

## 1. 产品结论

生成第一版应用后，Project 不应只剩下结构化 Edit、Vim 和重新构建。用户应当可以继续向 Lead 描述修改目标，由 Lead 结合当前 Project 重新梳理本轮工作，再把明确的修改任务交给固定下游团队。

这条链路必须满足四个条件：

1. **[仍在同一个 Project]** 修改不会创建新 Project，也不会把用户带回首次生成入口。
2. **[基于明确版本]** 每次修改都标明基于哪个当前版本和 Git commit，不能只拿原始 Prompt 重新生成。
3. **[默认保留已有结果]** 团队修改当前代码，只改变本轮要求涉及的部分；未要求变化的页面、交互和视觉应继续保留。
4. **[修改形成新版本]** 通过校验后创建新的 ProjectVersion 和 Git commit；原版本与线上版本均不被覆盖。

核心链路：

```text
当前 Project + 当前版本代码
          |
          v
    用户继续与 Lead 对话
          |
          v
 Lead 梳理本轮修改任务
          |
    +-----+----------------+
    |                      |
    v                      v
回答 / 澄清          交给固定下游团队
                           |
                           v
                 基于当前代码生成候选修改
                           |
                           v
                 校验 -> 新版本 -> 预览
                           |
                           v
                    用户决定是否更新线上
```

## 2. 要解决的问题

当前能力存在两个断点：

- **[对话断点]** 首页 Lead 消息不属于某个 Project。项目生成后，用户不能围绕当前代码继续一段可恢复的对话。
- **[修改断点]** 结构化 Edit 只支持少数固定字段，Vim 要求用户自己定位和修改代码；两者都不能承接“增加价格筛选，并保留现有商品卡片和移动端布局”这类自然语言修改。

新功能解决的是：用户不需要重新描述整个项目，也不需要先找文件；系统能够理解“在现有结果上改什么、什么必须保留、怎样算完成”，并把修改真实落到已有代码和版本历史中。

它不用于证明 V1 已经支持任意 Coding Agent、任意技术栈或任意 Shell 操作。

## 3. 核心产品对象

### 3.1 Project 对话

每个 Project 拥有一条可恢复的对话时间线，至少包含：

- 用户原始修改要求；
- Lead 的回答、澄清问题和修改任务摘要；
- 本轮基线版本；
- 下游执行状态和可检查产物；
- 失败原因或生成的新版本；
- 用户后续补充、停止和继续修改记录。

对话只保存用户可见内容和结构化结果，不保存模型私有推理过程。

### 3.2 修改任务摘要

Lead 不直接改代码。它把用户消息与当前 Project Context 整理成一张可检查的“修改任务”卡片：

- **用户原话：** 保留用户实际提出的修改要求；
- **本轮目标：** Lead 对要完成结果的简明复述；
- **保持不变：** 当前页面、功能或视觉中不应被本轮修改影响的部分；
- **完成标准：** 用户最终能够观察或操作验证的结果；
- **上下文对象：** 当前页面、用户选中的元素、文件、错误或版本；
- **基线版本：** 本轮实际读取的 ProjectVersion 和 commit；
- **影响判断：** 当前范围内修改、范围变化或需要额外确认的风险。

这张卡片是 Lead 向下游交接的事实摘要，不是新的完整 Blueprint，也不能覆盖初始需求和历史 Blueprint。

### 3.3 候选修改

下游团队基于修改任务和基线代码产生候选结果。用户最终看到的候选修改至少包括：

- 修改摘要；
- 基线版本与结果版本；
- 实际变更文件；
- 可展开的代码 Diff；
- 新版本 Preview；
- 确定性校验和质量复核结论；
- 未完成项和 warning。

实际变更文件和 Diff 必须由 Runtime 比较修改前后代码得出，不能只使用模型自己声明的“我修改了什么”。

## 4. 用户入口与交互

### 4.1 入口出现条件

Project 已经存在可预览版本时，Workspace 显示固定的 Project 对话入口，文案为“继续告诉 Lead 你想修改什么”。

用户可以直接描述修改，也可以附带当前上下文：

- 当前 Preview 页面；
- Preview 中选中的元素；
- 文件树中选中的文件；
- 当前错误或未通过检查；
- 当前工作版本。

未选择具体对象时，默认上下文是当前工作版本和当前打开页面。用户正在查看历史版本时，系统必须说明“修改仍基于当前工作版本”；如果用户确实要基于历史结果继续，应先 Restore，避免在 V1 中引入隐式分支。

### 4.2 对话结果

Lead 对每条 Project 消息只产生三种互斥结果：

1. **[回答]** 解释现有功能、代码、版本或错误，不创建修改 Run。
2. **[澄清]** 缺少会实质改变结果的信息时，提出一个聚焦问题；澄清完成前不修改代码。
3. **[修改]** 生成修改任务摘要，并交给固定下游团队执行。

用户明确说“先讨论，不要修改”时必须走回答；用户明确说“请修改、修复、增加或删除”且信息足够时不能降级成只给建议。

### 4.3 修改过程反馈

进入修改后，Workspace 保持 Preview、文件和版本可查看，同时展示：

- Lead 已理解的修改任务；
- 当前处理角色和阶段；
- 基线版本；
- 预计会影响的功能区域；
- 持久化日志；
- Stop 入口。

修改期间不显示“正在重新生成整个应用”。阶段文案应明确为“正在分析现有代码”“正在修改已有实现”“正在校验本次变更”。

### 4.4 修改完成

成功后自动切换到新版本 Preview，并在对话中插入结果卡片：

- `基于 v3 创建 v4`；
- 修改了哪些文件和功能；
- 哪些验收项已经通过；
- 是否存在 warning；
- 查看 Diff、继续修改、查看版本、更新线上版本。

“继续修改”会创建下一条 Project 消息，默认以上一个成功版本为新基线，不复用一段无限增长的隐藏上下文。

## 5. Lead 与下游团队如何分工

V1 沿用已经确认的固定顺序流水线，不在这项功能中新增角色、按任务跳过角色或并行执行：

```text
Lead 修改任务
   -> 产品阶段：确认需求变化与保留项
   -> 设计阶段：确认界面、交互和结构影响
   -> 工程阶段：读取基线代码并生成候选修改
   -> 质量阶段：校验修改任务、代码 Diff 和执行证据
```

- **[Lead]** 负责理解用户这次要改什么、引用哪个当前对象，以及是否需要澄清或风险确认；不直接生成代码。
- **[产品阶段]** 只描述本轮需求差异，不重新定义整个产品，也不能把用户目标改成另一类应用。
- **[设计阶段]** 只说明本轮修改对现有界面、交互和结构的影响；无变化时应明确保持现状，不能为了证明角色存在而制造重构。
- **[工程阶段]** 必须读取基线版本的 AppSpec 和实际项目代码，优先产生定向变更；不得回退到 starter template 或只根据最初 Prompt 重建应用。
- **[质量阶段]** 检查数据和本地状态是否被破坏，并提供不可被模型改写的确定性证据；未通过门禁的候选结果不能进入版本历史。

固定流水线的收益是沿用现有 Artifact、配额和恢复链路；代价是小修改仍有固定阶段延迟和调用成本。动态角色选择属于 V2，不在本功能中隐式实现。具体角色名称和 Contract 继续以 V1 基线文档为准，本功能只增加“基于已有代码修改”的任务语义。

## 6. 基于已有代码的约束

每次修改开始时冻结一个明确基线：

```text
base ProjectVersion
  + base Git commit
  + base AppSpec
  + 当前 Blueprint / ArchitectureSpec
  + 用户选中的页面、元素、文件或错误
  + 本轮修改任务摘要
```

产品必须保证：

- 修改 Run 属于原 Project，并引用基线版本；
- 下游读取的是基线版本代码，不是最新模板或另一个 Project 的代码；
- 未涉及文件默认保持不变；
- 结果 Diff 能从基线代码和候选代码确定性计算；
- 基线之后若出现新的工作版本，旧候选不能静默覆盖，必须提示“项目版本已经变化”，让用户重新基于最新版本执行；
- 同一 Project 同时只运行一个会写代码的任务，避免两个修改竞争当前版本。

V1 不提供自动合并、隐式 rebase 或并行写分支。用户在修改执行期间提出的下一项写操作保留为输入草稿，不自动排入隐藏队列。

### 6.1 代码接入对用户无感的边界

用户不需要理解 Git commit、BaseSourceSnapshot 或 Context 组装。只要代码已经归入当前 Project，Runtime 自动识别当前版本、读取受控源码并建立修改基线，Project 对话入口可以直接使用。

代码来源分为两类：

- **[平台已有 Project]** 平台生成、结构化 Edit、受限 Vim Save 或 Restore 形成的代码已经绑定 ProjectVersion 和 Git commit。该路径无需额外接入步骤，本功能直接读取当前版本。
- **[首次接入外部代码]** GitHub 仓库或本地文件夹仍需要一次明确授权、仓库选择或文件选择。浏览器不能静默读取用户本机目录，平台也不能在未授权时读取外部 Git。完成接入后，Runtime 应把源码统一转换为 Project Repository、初始 ProjectVersion 和源码清单，之后的对话修改与平台生成项目相同。

“用户无感”指后续修改不要求用户手动选择代码文件、commit 或 Context，不代表系统可以绕过首次资源授权、秘密扫描或不受支持技术栈检查。外部 GitHub/本地目录首次导入不属于本轮已实现范围。

## 7. Human-in-the-loop

用户发送明确修改要求，已经授权系统在当前 Project、当前范围和基础预算内完成一次可恢复修改。因此修改任务摘要始终可见，但不必统一停下来审批。

以下情况自动继续：

- 修改文案、样式或现有交互；
- 修复明确错误；
- 在现有页面和模块范围内调整功能；
- 结果只创建新的工作版本，不改变线上版本。

以下情况必须在执行前或副作用发生前确认：

- 新增、删除或替换页面与核心模块，改变已确认产品范围；
- 大面积删除或重写已有代码；
- 需要追加预算或额外重试；
- 需要放弃未保存修改或处理版本冲突；
- Restore、Publish、Update 或 Unpublish；
- 要求当前 Runtime 不支持的服务端、网络、依赖或外部能力。

用户确认的对象必须是当前修改任务、基线版本、风险说明和预算。上述任一对象变化后，旧确认失效。

## 8. 版本、发布与回退

- 成功修改创建新的 ProjectVersion 和 Git commit，版本来源显示为“AI 修改”。
- 原版本继续保留，可以预览和 Restore。
- 确定性校验或质量复核未通过时不创建 ProjectVersion，不移动 Project 当前版本。
- 新工作版本不会自动改变 Public Route；用户必须在确认 Preview 后明确 Update。
- 已发布项目的修改结果必须同时显示“当前工作版本”和“线上版本”，避免把生成成功误解成已经上线。
- 用户对结果不满意时，可以继续告诉 Lead 调整，也可以 Restore 到旧版本；两者都会保留历史，不执行覆盖式回滚。

## 9. 失败与恢复

失败不是 Project 对话的终点，也不能只显示“修改失败”。失败 Run 已经生成的 Artifact、事件和错误证据继续保留，但未通过校验的候选代码不写入 ProjectVersion。用户可以直接在失败页继续告诉 Lead 如何调整；新一轮修改仍以最后一个成功版本为代码基线，同时向 Lead 提供上一轮失败阶段、错误码和已有 Artifact 类型。

```text
最后成功版本 v3 + Git commit
          |
          v
      修改 Run A
          |
     生成中间 Artifact
          |
          v
       校验失败
          |
    +-----+-------------------------+
    |                               |
    v                               v
保留 Run A 的 Artifact、         v3 与线上版本不变
事件和错误证据
    |
    v
用户在失败页继续与 Lead 对话
    |
    v
修改 Run B = v3 代码基线
           + Run A 失败摘要 / Artifact 引用
    |
    v
通过校验后创建 v4
```

结果区至少说明：

- 失败阶段；
- 未通过的具体检查；
- 基线版本是否仍安全；
- 是否已经产生候选 Diff；
- 可以继续补充、从失败处重试还是必须重新梳理任务。

失败时原版本、Git head 和线上版本保持不变。当前实现会为新的修改 Run 重新执行受影响的 Agent 阶段，但 Lead 和下游能够看到上一轮失败摘要；后续可以再按 Artifact 依赖精确复用仍有效的上游产物，不能在尚未实现时宣称已经从任意失败阶段续跑。

Provider 用量继续按账户记录，但不因 `quota_limit` 阻断修改 Run。

## 10. V1 边界

本功能第一版明确不做：

- 多个修改任务并行或自动合并；
- Agent 直接执行任意 Shell、安装依赖或访问外部网络；
- 未经质量门禁直接写入 ProjectVersion；
- 基于历史版本自动创建分支并合并到当前版本；
- 跨 Project 自动复用代码或用户长期画像；
- 把整个聊天记录无上限地发送给每个下游角色；
- 自动更新已发布版本。

结构化 Edit 和受限 Vim 继续保留：结构化 Edit 适合少数字段的即时修改，Vim 适合用户直接精确改代码，Project 对话负责自然语言驱动的跨文件或行为修改。三个入口最终都进入同一版本与校验体系。

## 11. 验收路径

### 11.1 主路径

1. 用户打开已有 Mono Market 商品目录 Project，当前工作版本为 v1。
2. 用户告诉 Lead：“在目录页增加价格筛选，并保留现有商品卡片、编辑风格和移动端布局。”
3. Lead 展示修改任务：增加价格筛选；保留商品卡片、视觉方向和移动端布局；基于 v1。
4. 该请求属于现有产品范围，系统自动交给固定团队。
5. Engineer 基于 v1 的实际代码修改，Runtime 展示真实 Diff 并完成校验与审查。
6. 系统创建 v2；v1 仍可预览，Project 不新增副本。
7. 用户在 v2 Preview 验证筛选结果，再继续告诉 Lead 调整筛选控件的手机布局。
8. 第二次修改基于 v2 创建 v3。
9. 如果项目已有线上版本，公开地址在用户明确 Update 前保持原版本。

### 11.2 必须验证的反路径

- 用户问“当前商品排序在哪里定义”时只回答，不创建 Run 或版本。
- 修改目标存在关键歧义时先澄清，不自行选择会改变产品结果的方案。
- 用户要求删除页面或增加新的核心模块时出现范围确认。
- 修改执行期间 Project 被其他入口保存为新版本时，旧候选不能覆盖新版本。
- 质量门禁未通过时不创建版本，原 Preview、Git head 和公开地址不变。
- 模型失败或服务重启后，用户消息、基线版本和失败原因仍可恢复。
- 用户不能通过修改 Project ID、Version ID 或文件路径读取或修改其他账号的代码。

## 12. 当前实现状态

已实现：

- Project 工作区提供“继续和 Lead 对话”入口，并持久化用户请求、Lead 修改摘要、完成结果和失败消息；
- 修改 Run 自动绑定当前 ProjectVersion 和 Git commit，从 commit 读取受控源码形成 BaseSourceSnapshot；
- Lead、Product Manager、Architect、Engineer、Data Analyst、Runtime Validator 和 Reviewer 复用现有 Worker 顺序执行；
- Runtime 计算 SourceDiff，校验通过后创建来源为 `ai_edit` 的新 ProjectVersion 与 Git commit；
- 同一 Project 的 AI 修改使用数据库写占用，结构化 Edit 与 Restore 在修改期间拒绝写入；最终提交再次检查基线版本；
- 修改失败释放写占用，保留原工作版本和发布指针；自动化测试覆盖主路径、并发、失败、归属隔离和源码快照。

仍未实现：

- Project 消息的 answer/clarify 路由；当前工作区输入会被视为明确修改请求；
- 范围变化、破坏性 Diff 和额外预算对应的风险确认界面；
