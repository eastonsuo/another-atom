## 背景

用户现在可以在首页选择附件，但界面只记录文件名、大小和类型，AI 团队没有获得图片内容。对于“参考这张截图实现”“这里为什么显示异常”这类需求，用户仍需手工描述图片，且附件按钮会造成“系统已经看到了图片”的错误预期。

本设计把图片作为对话消息的一部分：用户可以复制截图并直接粘贴到对话框，也可以从文件选择器添加；界面明确展示上传、理解、失败和移除状态。图片只用于补充用户文字，不改变现有 ProductSpec 确认、固定 Agent 顺序和 Runtime 验证。

## 摘要

- **主入口**
  - 用户聚焦对话框后直接粘贴截图；文件选择器作为补充入口。普通文本粘贴保持不变。
- **发送条件**
  - 用户仍需填写文字说明，且所有图片必须上传完成；图片上传或视觉理解不能被静默跳过。
- **即时反馈**
  - 粘贴后立即出现缩略图、文件名和状态。发送后用户消息立即进入时间线，随后显示图片理解和 Agent 工作进度。
- **可检查结果**
  - 用户可以展开查看图片概要、识别文字和不确定项，确认 AI 实际理解了什么。
- **失败恢复**
  - 单张图片失败不清空文字或其他图片；用户可以重试、移除失败图片，或明确选择不带该图片继续。
- **范围边界**
  - V1 只处理作为需求参考的静态图片，不包含图片生成、编辑、视频、PDF 或自动把参考图加入生成项目。

## 1. 用户目标与范围

### 1.1 用户目标

1. 把当前屏幕截图复制后直接粘贴进首页或 Project 对话框。
2. 在同一条消息中同时提供文字要求和多张参考图。
3. 在发送前确认哪些图片会进入本次 Context。
4. 在 AI 工作时看到图片是否已上传、是否正在理解以及理解是否成功。
5. 图片失败时继续编辑原消息，不需要重新输入需求。

### 1.2 V1 支持范围

- PNG、JPEG 和静态 WebP。
- 每条消息最多 5 张，单张最大 10 MB。
- 图片用途固定为 `reference`：帮助 AI 理解需求、界面和问题。
- 首页首次需求和已有 Project 的常驻对话框使用同一套交互。
- 所选主模型仍为 Pro 或 Flash；图片由平台配置的视觉模型预处理，用户不需要手动切换两次模型。

### 1.3 V1 不支持

- 只有图片、没有任何文字说明的消息；发送前至少需要一段非空文字。
- GIF 动画、SVG、PDF、视频和压缩包。
- 图片生成、局部重绘、抠图和 OCR 文档工作流。
- 自动把参考图复制为生成项目的 Logo、商品图或其他业务素材。
- 声称根据截图实现像素级一致页面。

## 2. 页面入口

### 2.1 首页 Composer

首页对话框支持两种等价输入：

- 直接把截图粘贴到已经聚焦的输入框；
- 点击回形针选择本地图片。

粘贴图片后仍停留在当前输入框，光标和已输入文字不得丢失。模型选择器保持原位置；当选择 `deepseek-v4-pro` 时，图片卡提示“图片将先由视觉模型理解，再交给 Pro”。

### 2.2 Project 常驻对话框

Project 在空闲、失败、完成、等待 PM 补充或 ProductSpec 修改状态下，按照[常驻流式对话](./07-常驻流式对话与执行期间输入控制.md)既有提交语义接受图片。

Agent 或 Runtime 正在工作时：

- 对话框继续显示并允许编辑草稿；
- 用户可以把图片粘贴进草稿并看到本地缩略图；
- 发送仍然禁用，不能用附件绕过“单活动轮次”限制；
- 当前轮次结束后，草稿文字和图片保持原样。

## 3. 粘贴和选择图片

### 3.1 粘贴行为

对话框收到粘贴操作后按内容处理：

| 剪贴板内容 | 结果 |
| --- | --- |
| 只有文字 | 使用浏览器正常文本粘贴 |
| 只有支持的图片 | 添加图片卡，不向文本框写入 Base64 或文件路径 |
| 文字和图片 | 保留文字，同时添加图片卡 |
| 不支持的文件 | 不添加，显示具体类型错误 |

同一条消息内重复粘贴内容相同的图片时只保留一张，并提示“这张图片已经添加”。

### 3.2 图片卡

每张图片以紧凑卡片展示：

```text
[缩略图] screenshot.png  824 KB
         正在上传… / 已就绪 / 上传失败
         [重试] [移除]
```

图片卡必须提供：

- 可辨认的缩略图；
- 原始文件名或“剪贴板图片 + 时间”；
- 文件大小；
- 当前状态；
- 移除按钮；
- 失败时的原因和重试按钮。

点击缩略图打开有界大图预览，不离开当前草稿。移除正在上传的图片时取消该图片上传；移除不会改变已输入文字和其他图片。

### 3.3 数量和大小反馈

- 第 6 张图片不会加入，提示“每条消息最多添加 5 张图片”。
- 单张超过 10 MB 时不开始上传，提示实际大小和限制。
- 不支持类型时直接说明支持 PNG、JPEG 和 WebP，不显示笼统的“上传失败”。
- 图片已经添加但服务端识别为损坏或伪造类型时，图片卡进入失败状态并展示服务端结论。

## 4. 发送条件

发送按钮只有同时满足以下条件时可用：

1. 文字说明去除空白后非空；
2. 已选择可用主模型；
3. 所有保留图片都已上传完成；
4. 当前 Project 没有活动对话轮次或 Run；
5. 当前提交语义允许发送或回答。

禁用时按钮附近显示唯一的主要原因，例如：

- `请先描述你希望 AI 根据图片完成什么`
- `2 张图片正在上传`
- `请移除或重试上传失败的图片`
- `工程师正在工作，完成后可以继续发送`

`Shift + Enter` 与点击发送使用相同条件，不允许键盘提交绕过禁用状态。

## 5. 发送后的时间线

### 5.1 用户消息立即出现

提交成功后，用户消息立即进入对话时间线，包含文字和附件缩略图。界面不等待图片理解或 Lead 返回后才显示用户消息。

图片存在时，系统依次展示：

```text
你：参考这张截图，调整项目文件面板的布局。
[截图缩略图]

[正在理解 1 张参考图]
[图片理解完成，可展开查看]

团队负责人：……
```

“正在理解图片”属于运行状态，不伪装成 Agent 自然语言回复。

### 5.2 图片理解结果

理解成功后显示可展开卡片：

- 图片内容概要；
- 识别到的主要文字；
- 可见区域和元素关系；
- AI 无法确认的内容。

默认折叠，不占满对话时间线。用户打开卡片是为了核对 Context，不承担修改 ProductSpec 或批准方案的职责。

### 5.3 后续角色

- Lead 使用图片概要判断直接回答、澄清或调用团队。
- 产品经理把图片证据和用户文字整理进 ProductSpec。
- 架构师和工程师继续使用同一份图片理解结果，不重复显示多张相同卡片。
- ProductSpec 确认仍是产品事实门禁；图片本身不等于用户批准某个实现方案。

## 6. 失败与恢复

### 6.1 上传失败

上传失败发生在发送前。保留图片卡和草稿，用户可以：

- 重试该图片；
- 移除该图片；
- 修改文字；
- 保留其他已经成功的图片。

只要仍有失败或上传中的图片，发送保持禁用。

### 6.2 图片理解失败

图片理解失败发生在消息已经进入时间线之后。系统必须：

- 保留用户消息、文字和原图；
- 标明失败发生在“图片理解”，不是 Pro、PM 或工程师；
- 不继续执行一个缺少图片 Context 的团队流程；
- 提供“重试图片理解”；
- 提供“移除失败图片后继续”，并要求用户明确确认这次范围变化。

如果视觉模型没有配置，图片入口直接显示不可用原因；纯文本消息仍可正常发送。

### 6.3 刷新和恢复

- 未发送草稿沿用现有草稿恢复策略；已上传附件 ID 随草稿保存。
- 已发送消息、附件、理解状态和理解结果由服务端恢复，不依赖浏览器内存。
- 刷新后不能重新上传同一张已就绪图片，也不能重复触发已完成的视觉调用。

## 7. 隐私和能力说明

- 上传入口明确说明图片会发送给当前配置的视觉模型。
- 当使用 Ollama Cloud 时，界面不能写“图片只在本地处理”。
- 日志界面不展示图片 Base64、服务端绝对路径或完整 OCR 内容。
- 另一个用户不能通过图片 URL、消息 ID 或附件 ID 查看图片。
- 删除 Project 时，其绑定图片也进入删除范围；删除前应遵循现有 Project 删除确认。

## 8. 产品验收标准

1. 首页和 Project 对话框都能通过系统截图快捷键复制后直接粘贴图片。
2. 粘贴图片不会清空文字、移动页面或写入 Base64 文本；立即显示缩略图和真实状态。
3. 普通文字粘贴不受图片功能影响。
4. 上传中、失败、超限、无文字和 Project 正在工作时，发送按钮分别显示准确原因。
5. 发送成功后用户消息立即出现，随后显示图片理解进度，而不是等待模型结束后一次性出现。
6. 用户可以检查图片概要、OCR 和不确定项，并能看到 Pro 仍是主模型。
7. 图片理解失败时不进入 PM/架构师/工程师流程，文字和附件保持可恢复。
8. 刷新页面后，已发送图片、理解结果和活动状态保持一致。
9. 纯文本请求不出现图片状态，也不产生视觉模型用量。
10. 不同用户之间无法访问彼此图片。
