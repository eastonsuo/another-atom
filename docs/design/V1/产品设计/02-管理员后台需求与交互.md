## 背景

普通用户只能查看自己的 Project 和 Run，这满足用户隔离，但运营人员缺少跨用户定位注册、项目和执行状态的受控入口。管理员后台用于补足受控运营视角，并允许现有管理员把指定注册用户提升为管理员；它不替代用户工作台。

## 摘要

- **后台职责**
  - 管理员通过数据库中的显式 `admin` 角色进入管理页面，集中查看用户、Project 及最新 Run 状态，用于运营观察和问题定位。
- **有限写边界**
  - 后台唯一允许的用户写操作是把普通注册用户提升为管理员；不修改密码、套餐和 Project，也不代替用户构建或干预 Run。

## 1. 要解决的问题

当前用户只能看到自己的 Project 和 Run。运营人员无法在一个受控入口中回答以下问题：

1. 现在有哪些注册用户？
2. 每个用户创建了哪些 Project？
3. 这些 Project 大致在做什么？
4. 最新一次执行到了哪个阶段，是在进行、等待用户、已完成还是失败？
5. 发生问题时，能否快速定位对应用户、Project 和 Run？

V1 管理员后台先提供“看清当前系统状态”的闭环，并增加单一、显式的管理员提升操作；其他用户数据修改和构建干预仍不进入本版本。

## 2. 产品原则与关键取舍

### 2.1 管理员是独立权限，不是特殊用户名

管理员使用已有用户名密码和服务端 Session Cookie 登录，但账户必须在数据库中具有明确的 `admin` 角色。

- 不根据用户名、邮箱、请求头或前端参数判断管理员。
- 公开注册只能创建普通用户。
- 默认初始化 `admin / admin12345`。V1 作为演示交付允许该默认账号直接用于公开部署；建议通过 `ADMIN_USERNAME` 和 `ADMIN_PASSWORD` 覆盖，未覆盖时启动记录 `default_admin_credentials_active` 告警日志。
- 环境配置保证始终存在一个可恢复的管理员入口；后台提升的其他管理员写入数据库并在服务重启后保持有效。
- 后台用户列表只展示仍可被提升的普通注册用户；当前管理员和已提升管理员不在待配置列表中重复出现。
- 后端所有 `/api/admin/*` 接口都必须独立执行管理员授权，不依赖页面是否隐藏菜单。

### 2.2 执行进度来自持久化状态

管理员看到的“执行到哪一步”必须来自最新 Run 的 `status` 和 `current_stage`，不根据前端是否还在转圈推断。

Project 状态和 Run 阶段分开展示：

- **Project 状态：** 说明项目整体是草稿、构建中、已就绪、已发布或已暂停。
- **最新 Run 阶段：** 说明这次任务正在做什么，或为什么已停下。

这两类信息不能合并成一个“进行中 / 已完成”字段。例如，一个已有可用版本的 Project，最新 Run 仍可能失败。

### 2.3 Project 简介确定性生成

V1 不为管理页额外调用 LLM 生成摘要。Project 简介按以下顺序取值：

1. 已有 Blueprint 时，使用 `product_type` 和前几个 `modules` 组成短描述。
2. Blueprint 还未产生时，使用 `Project.prompt` 清理换行后的前 80 个字符。
3. 无可用内容时显示“暂无简介”。

这个简介只帮助管理员快速识别 Project，不作为新的 Agent Artifact，也不改写用户原始需求。

### 2.4 V1 只开放管理员提升

管理员可以查看用户、Project 和最新执行情况，但不能直接：

- 修改用户密码、额度或套餐；
- 禁用、删除或冒充登录某个用户；
- 修改、删除、发布或恢复用户 Project；
- 进入用户 Sandbox、修改 Git 仓库或重试 Run。

唯一例外是管理员可以把一个普通注册用户提升为 `admin`。该操作必须显示目标用户名和权限后果并要求确认；当前版本不提供降级入口，避免误操作撤销唯一可用管理员。

先把可见性做准确，避免在没有操作审计和二次确认的情况下引入跨用户写操作。

## 3. 页面与交互

### 3.1 管理员登录页

- **地址：** `/admin/login`
- **输入：** 用户名、密码
- **成功：** 创建服务端 Session Cookie，进入 `/admin`
- **普通用户登录：** 显示“该账户无管理权限”，不返回用户列表
- **登录失败：** 在表单中显示明确错误，不使用无限 loading 代替失败状态

管理员和普通用户复用同一套 Session 机制，但后台使用独立路由和布局，避免把运营功能混入用户 Studio。

### 3.2 用户与 Project 总览

`/admin` 默认按用户展示，用户行可展开对应的 Project 列表。

#### 用户层

- 显示名称 `display_name`
- 用户名 `username`
- 注册时间 `created_at`
- 套餐 `plan`
- Provider 请求用量：只显示 `used`
- Project 数量
- “设为管理员”操作；成功后该用户从普通用户列表移除，并可使用原用户名密码登录管理后台

#### Project 层

- Project 名称
- 简短介绍
- Project 状态
- 最新 Run 执行阶段
- 最新 Run 结果状态
- 最近更新时间
- 最新 Run ID 的短形式，可复制完整 ID
- 失败时显示 `error_code` 和简短错误信息

默认只请求用户摘要。展开某个用户时再加载其 Project，避免首屏产生用户数 × Project 数的无界查询和返回体。

### 3.3 Project 详情抽屉

点击 Project 后打开只读抽屉，展示：

- 完整 Project ID、完整最新 Run ID；
- 原始需求的短摘要和 Blueprint 的支持等级；
- 最新 Run 的模型、开始时间、更新时间、用量和错误；
- 最新 Run 的阶段事件时间线；
- 跳转至现有 Run 日志下载接口的入口。

详情抽屉不直接展示密码哈希、Session token、Sandbox token、服务端仓库绝对路径或 Provider 密钥。

### 3.4 查询与空状态

V1 支持：

- 按显示名称或用户名搜索用户；
- 用户列表和用户下的 Project 列表分别分页，默认每页 20 条，最大 100 条；
- 手动刷新当前页；
- 无用户、用户无 Project、Project 无 Run 和请求失败的独立空状态。

V1 不做复合报表、图表大屏、导出全库数据、自定义列或实时秒级推送。管理员手动刷新即可获得最新持久化状态。

## 4. 执行阶段的显示规则

| 持久化状态 / 阶段 | 管理员页显示 | 含义 |
| --- | --- | --- |
| 无 Run | 未开始 | Project 已存在，但没有构建记录 |
| `product_running` / `product_manager` | 产品经理正在整理需求 | Blueprint 尚未完成 |
| `awaiting_approval` / `blueprint_approval` | 等待用户确认方案 | 任务不是卡住，而是 Human-in-the-loop 停顿 |
| `needs_input` / `scope_review` | 等待用户补充或调整范围 | 当前需求不能直接继续 |
| `build_queued` / `build_queue` | 已进入构建队列 | 等待 Worker 获取任务 |
| `architect_running` / `architect` | 架构师正在设计 | 正在生成架构设计（ArchitectureDesign）及架构设计文档 |
| `engineer_running` / `engineer` | 工程师正在生成代码和单元测试 | 正在生成或修复应用规格（AppSpec）与源码包（SourceBundle） |
| `dispatching` / `build` | 正在提交验证任务 | 主服务正在向独立执行服务提交受控执行请求（ExecutionRequest） |
| `materializing` / `build` | 正在准备源码 | 独立执行服务正在临时目录物化源码包 |
| `building` / `build` | 正在构建 | 独立执行服务正在执行固定构建命令 |
| `testing` / `build` | 正在运行单元测试 | 独立执行服务正在执行工程师交付的单元测试 |
| `validating` / `build` | 正在确定性校验 | 独立执行服务正在检查源码和交接契约（Contract） |
| `completed` / `complete` | 已完成 | 已形成可用版本 |
| `completed_degraded` / `complete` | 已完成，有警告 | 强制构建、测试和校验（mandatory Build、Test、Validation）已通过并形成版本，但执行报告（ExecutionReport）或校验报告（ValidationReport）有非阻断警告 |
| `failed` | 失败 | 同时显示 `error_code` 和错误摘要 |
| `cancelled` | 已取消 | 任务不再执行 |

Run 结果状态优先于普通阶段文案。例如 `failed + engineer` 应显示“工程师阶段失败”，不能显示“工程师正在工作”。

## 5. 数据与 API Contract

### 5.1 数据模型调整

`User` 增加权限角色：

```text
role: user | admin
```

- 现有用户迁移时默认为 `user`。
- 公开注册必须强制写入 `user`。
- 管理员权限只从数据库加载，不接受客户端传入。

Project、Run、Artifact 和 RunEvent 不需要为此复制新表。后台通过现有外键联查真实数据。

### 5.2 后端依赖

增加 `get_current_admin`：

1. 读取现有 Session Cookie；
2. 校验 Session 未失效、未撤销；
3. 读取对应 User；
4. 校验 `User.role == admin`；
5. 未登录返回 `401`，已登录但无权限返回 `403`。

### 5.3 管理 API

```text
POST /api/admin/login
GET  /api/admin/me
GET  /api/admin/users?query=&page=1&page_size=20
PATCH /api/admin/users/{user_id}/role
GET  /api/admin/users/{user_id}/projects?page=1&page_size=20
GET  /api/admin/projects/{project_id}
GET  /api/admin/runs/{run_id}/logs/download
```

- `POST /api/admin/login` 复用密码验证和 Session 创建逻辑，但只允许 `admin` 角色完成登录。
- `PATCH /api/admin/users/{user_id}/role` 当前只接受 `{"role":"admin"}`；公开注册和普通用户接口不能调用该能力。
- 列表接口返回分页元数据，不一次返回全部用户和 Project。
- Project 列表为每个 Project 联查最新 Run；“最新”按 `created_at` 倒序，相同时间再按 ID 排序，保证结果稳定。
- 管理员日志下载复用现有 Run 日志渲染内容，但使用独立管理员授权，不绕过权限层直接读文件。

### 5.4 项目列表响应的最小字段

```text
user
  id
  username
  display_name
  plan
  quota_limit
  quota_used
  quota_reserved
  project_count
  created_at

project
  id
  name
  summary
  status
  updated_at
  latest_run
    id
    model
    status
    current_stage
    error_code
    error_summary
    quota_spent
    created_at
    updated_at
```

响应不包含 `password_hash`、`token_hash`、`terminal_token`、`repository_path` 和任何 Provider 密钥。

## 6. 权限与隐私边界

### 6.1 管理员可见范围

管理员为了运营和问题定位可以查看：

- 用户名、显示名称、注册时间、套餐和额度摘要；
- Project 名称、简介、状态和时间；
- 最新 Run 的阶段、结果、错误和调试日志。

管理员权限不代表应默认展开所有原始 Prompt、全部 Agent Artifact 或源码。首屏只返回运营必要的摘要，详情查看必须由管理员显式打开某个 Project。

### 6.2 日志与安全要求

- 后端日志记录管理员 User ID、访问的资源类型和资源 ID，不记录管理员密码、Session Cookie 或用户源码内容。
- 管理员列表查询使用服务端分页与参数上限，避免无界全表读取。
- 生产部署应设置 `SESSION_COOKIE_SECURE=true`；未设置时服务启动记录 `session_cookie_secure_disabled` 告警。
- 页面和 API 返回 `Cache-Control: no-store`，避免管理数据被浏览器或中间缓存。
- 管理员退出复用现有 Session 撤销逻辑。

## 7. 异常和边界状态

- **账户无权限：** 登录页明确返回无管理权限，不进入后台。
- **目标已是管理员或不存在：** 不重复写入角色，返回目标不存在，不把管理员重新列为普通用户。
- **Session 过期：** 返回登录页，并提示会话已过期。
- **用户无 Project：** 保留用户行，展开后显示“还没有 Project”。
- **Project 无 Run：** 显示“未开始”，不伪造阶段。
- **Run 失败：** 显示失败阶段、`error_code` 和安全截断后的错误摘要，可下载完整持久化调试日志。
- **数据查询失败：** 保留页面结构并提供重试，不将空列表伪装成查询成功。
- **状态未识别：** 显示原始状态值和“未识别阶段”，方便发现后端与前端映射漂移。

## 8. 验收标准

### 8.1 功能验收

- 管理员可以使用专用入口登录并看到所有普通注册用户；管理员账户不重复列入待配置对象。
- 管理员确认后可以把指定普通用户提升为管理员；新管理员可以使用原账号登录后台，服务重启后角色仍有效。
- 每个用户的显示名称、用户名、Project 数量和额度摘要正确。
- 展开用户后，可以看到其所有 Project 的名称、简介、Project 状态和最新 Run 阶段。
- 失败 Run 能显示失败阶段、错误代码和错误摘要，并能下载对应日志。
- 用户和 Project 较多时，列表仍通过服务端分页稳定返回。

### 8.2 权限验收

- 未登录请求管理 API 返回 `401`。
- 普通用户请求管理 API 返回 `403`。
- 普通用户不能提升自己或其他用户；角色变更只接受服务端管理员 Session。
- 修改 Cookie、URL 中的 User ID、Project ID 或 Run ID 不能绕过管理员权限。
- 后台 API 不返回密码哈希、Session token、Sandbox token、仓库绝对路径和 Provider 密钥。
- 普通用户的 Studio 查询范围保持不变，不因增加管理后台而放宽。

### 8.3 状态验收

- 运行（Run）从产品经理、产品规格确认、架构师、工程师到独立运行系统构建、测试和校验的各阶段都能显示正确文案。
- `awaiting_approval` 和 `needs_input` 明确显示为等待用户，不标记为系统故障。
- 终态 Run 不显示“正在工作”。
- 服务重启后，后台显示与数据库中已恢复的 Run 状态一致。

## 9. V1 明确不做

- 用户封禁、删除、密码重置或冒充登录。
- 管理员降级和细粒度后台角色。
- 套餐、额度、计费和充值操作。
- 强制停止、重试、恢复或删除用户 Run。
- 代替用户发布、更新或下线 Project。
- 批量导出用户 Prompt、Artifact、源码或调试日志。
- 运营数据仓库、趋势图、实时监控大屏和多组织权限系统。
