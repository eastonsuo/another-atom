# Another Atom 设计文档规范与索引

[toc]

## 背景

随着 V1、V2、整体原则和参考资料逐步增加，仅靠目录和文件名已经不足以区分正式 Contract、计划能力与历史检查。设计文档需要统一职责、版本归属、命名和首屏信息，避免同一结论分散在 Design、Review 和 README 中。

## 摘要

- **文档职责**
  - `docs/design/` 是产品与技术设计的持续事实源，Review 只保存检查事实、证据和处理状态。
- **内容分类**
  - 产品需求与交互合并维护，技术设计区分 Agent 与工程问题。
- **目录与版本**
  - 技术设计使用单层目录和 `[Agent]` / `[工程]` Tag，版本范围按 V1、V2、整体区分。
- **首屏要求**
  - 每篇正式设计文档在元信息后先写“背景”和“摘要”，摘要使用缩进列表提炼要点。

## 1. 文档职责

`docs/design/` 是持续维护的设计事实源，只保存已经确认或明确标为草案的产品、Agent 与工程 Contract。它回答“产品应该怎样工作、技术如何保证它成立”，不记录某次检查发现了什么或当前测试跑到了多少项；实现证据、缺陷和阶段结论统一放入 [`docs/review/`](../review/README.md)。

设计文档只分为两类：**产品设计**和**技术设计**。

### 1.1 产品设计

产品设计回答“做什么、为谁做、为什么这样做、用户最终看到什么”。内容可以包括：

- 产品目标、目标用户、版本范围和明确不做；
- 用户路径、信息结构、页面与交互；
- 用户可感知的状态、失败出口和能力边界；
- 功能验收标准、关键产品取舍及适用边界。

产品需求和产品层设计写在同一份文档中，不再维护相互重复的 PRD 与交互设计。产品设计可以定义需要哪些 Contract，但不展开 API、表结构、调度算法、部署命令或 Sandbox 实现。

### 1.2 技术设计

技术设计回答“如何实现并可靠地满足产品设计”。文档放在同一个`技术设计/`目录中，通过文件名 Tag 标记主要类型：

- `Agent`：角色职责、输入输出 Contract、Context、Memory、编排、Handoff、Tool、Human-in-the-loop，以及 Agent 与确定性 Runtime 的控制边界；
- `工程`：系统架构、API、数据与持久化、状态推进、并发、恢复、安全、Sandbox、测试和部署。

同一问题只保留一个主要事实源。涉及另一个领域时通过链接引用，不在多份文档中复制完整定义。Tag 按文档主要回答的问题选择，不因作者或临时工作阶段变化。

### 1.3 首屏结构

每篇 Design 文档在标题、`[toc]` 和元信息之后，正文之前固定写：

1. **背景**：用一至两段说明本文在做什么、为什么需要这篇文档，以及它解决的具体问题；不在背景中提前展开完整方案。
2. **摘要**：使用缩进列表展示文章要点，不写成连续长段落。一级列表按正文中相互独立的同层问题组织，二级列表写对应结论；只提炼正文已有内容，不新增正文无法支撑的判断。要点数量由文章实际内容决定，不设固定数量，不为保持格式对称而强行拆成三条或凑维度。

推荐格式：

```markdown
## 背景

本文用于……，解决……。

## 摘要

- **核心结论**
  - ……
- **范围边界**
  - ……
```

## 2. 版本范围

- `V1`：当前实现和验收基线；V1 完成前不把 V2 能力写成当前事实。
- `V2`：V1 验收后的计划版本；所有能力默认视为未实现。
- `整体`：跨版本长期有效的产品原则和外部参考资料，不作为版本归属不清时的临时目录。

每个版本不再设置独立 README。版本说明和全部入口统一维护在本文，避免索引与正式设计重复。

## 3. 目录与编号

```text
docs/design/
├── README.md
├── V1/
│   ├── 产品设计/
│   └── 技术设计/
│       └── NN-[Agent|工程]-中文主题.md
├── V2/
│   ├── 产品设计/
│   └── 技术设计/
│       └── NN-[Agent|工程]-中文主题.md
└── 整体/
    └── NN-[产品|参考]-中文主题.md
```

版本内产品设计文档使用 `NN-中文主题.md`，例如 `01-核心产品需求与交互.md`；技术设计文档使用 `NN-[Agent|工程]-中文主题.md`，例如 `03-[工程]-系统架构.md`；`整体/`使用 `NN-[产品|参考]-中文主题.md`。序号只表示同一目录内的建议阅读顺序，不表示优先级或实施状态；已有链接建立后，不因临时插入文档随意重排。

Review 文件使用全局稳定编号和类型 Tag：`NN-[产品|Agent|工程|综合]-YYYY-MM-DD-中文短主题.md`；从待办移入归档时文件名不变。

## 5. 与 Review 的关系

```text
design 定义预期
  -> 实现
  -> review 记录检查事实、证据和差异
  -> 需要长期生效的结论回写 design
```

Design 可以持续修订；dated Review 保留历史原貌，通过追加 Update 记录后续修复与验收。讨论稿、问答、测试数量、当前完成度和一次性检查结论不作为长期 Design 单独维护。
