# Another Atom 设计文档规范与索引

[toc]

## 1. 文档职责

`docs/design/` 是持续维护的设计事实源，只保存已经确认或明确标为草案的产品、Agent 与工程 Contract。它回答“产品应该怎样工作、技术如何保证它成立”，不记录某次检查发现了什么或当前测试跑到了多少项；实现证据、缺陷和阶段结论统一放入 [`docs/review/`](../review/README.md)。

设计文档只分为两类：**产品设计**和**技术设计**。

### 1.1 产品设计

产品设计回答“做什么、为谁做、为什么这样做、用户最终看到什么”。内容可以包括：

- 产品目标、目标用户、版本范围和明确不做；
- 用户路径、信息结构、页面与交互；
- 用户可感知的状态、失败出口和能力边界；
- 功能验收标准、关键产品取舍及适用边界。

产品需求和产品层设计写在同一份文档中，不再维护相互重复的 PRD 与交互设计。产品设计可以定义需要哪些 Contract，但不展开 API、表结构、调度算法、部署命令或 Sandbox 实现。

### 1.2 技术设计

技术设计回答“如何实现并可靠地满足产品设计”。按主要问题再分为两类：

- `Agent`：角色职责、输入输出 Contract、Context、Memory、编排、Handoff、Tool、Human-in-the-loop，以及 Agent 与确定性 Runtime 的控制边界；
- `工程`：系统架构、API、数据与持久化、状态推进、并发、恢复、安全、Sandbox、测试和部署。

同一问题只保留一个主要事实源。涉及另一个领域时通过链接引用，不在多份文档中复制完整定义。无法只靠目录判断的文档，按它主要回答的问题归类。

## 2. 版本范围

- `V1`：当前实现和验收基线；V1 完成前不把 V2 能力写成当前事实。
- `V2`：V1 验收后的计划版本；所有能力默认视为未实现。
- `整体`：跨版本长期有效的产品原则和外部参考资料，不作为版本归属不清时的临时目录。

每个版本不再设置独立 README。版本说明和全部入口统一维护在本文，避免索引与正式设计重复。

## 3. 目录与编号

```text
docs/design/
├── README.md
├── V1/
│   ├── 产品设计/
│   └── 技术设计/
│       ├── Agent/
│       └── 工程/
├── V2/
│   ├── 产品设计/
│   └── 技术设计/
│       ├── Agent/
│       └── 工程/
└── 整体/
    ├── 产品设计/
    └── 参考资料/
```

正式设计文档使用两位序号，例如 `01-产品范围与交互.md`。序号只表示同一目录内的建议阅读顺序，不表示优先级或实施状态；已有链接建立后，不因临时插入文档随意重排。

Review 文件继续使用 `YYYY-MM-DD-中文短主题.md`，不叠加设计文档序号。

## 5. 与 Review 的关系

```text
design 定义预期
  -> 实现
  -> review 记录检查事实、证据和差异
  -> 需要长期生效的结论回写 design
```

Design 可以持续修订；dated Review 保留历史原貌，通过追加 Update 记录后续修复与验收。讨论稿、问答、测试数量、当前完成度和一次性检查结论不作为长期 Design 单独维护。
