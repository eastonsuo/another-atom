# Another Atom 设计文档规范与索引

[toc]

## 1. 作用

`docs/design/` 是 Another Atom 的设计事实源，回答“为什么这样设计、系统应该怎样工作、实现必须满足什么边界”。设计文档持续维护，但必须区分当前 V1、计划 V2 和整体原则，不能把未来能力写成当前实现。

## 2. 目录规则

```text
docs/design/
├── V1/
│   ├── 产品设计/
│   ├── Agent设计/
│   ├── 工程设计/
│   └── 参考资料/
├── V2/
│   └── 同上
└── 整体/
    └── 同上
```

- `V1`：当前实现与验收基线；
- `V2`：V1 验收后的计划版本，不代表已经实现；
- `整体`：只保存系统级原则、版本演进和不属于单一版本的参考资料。

领域按需建立，回答不同问题：

- `产品设计`：做什么、为谁做、为什么做；
- `Agent设计`：模型看见什么、如何决策、协作和申请能力；
- `工程设计`：平台、数据和执行环境如何可靠实现；
- `参考资料`：外部事实和设计输入，不直接构成项目 Contract。

## 3. 版本入口

- [V1 设计](./V1/README.md)：当前实现、P0 设计与交付基线。
- [V2 设计](./V2/README.md)：动态 TaskGraph、多 Agent 和 Task Sandbox 计划。
- [整体设计](./整体/README.md)：半 AI Coding Agent Runtime 演进和参考资料。

## 4. 新设计文档要求

新文档至少写清：状态、版本范围、解决和不解决的问题、当前代码事实、目标 Contract、优势、代价、边界和验收方式。文件名和标题默认使用中文。

版本归属明确的文档不能放入 `整体`；涉及多个领域时按主要问题归类，其他领域通过链接引用。

## 5. 与 Review 的关系

```text
design 定义预期
  -> 实现
  -> review 检查事实与差异
  -> 发现系统性问题
  -> 新建或更新 design 专项方案
  -> 再实现、再 review
```

Review 可以给出证据、问题等级和设计输入，但专项解决方案必须回到 `docs/design/`。
