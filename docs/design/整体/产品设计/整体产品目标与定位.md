## 1. 产品结论

Another Atom 与 [Atoms](https://help.atoms.dev/en) 面向同一个核心目标：**让用户通过 Vibe Coding，把一个模糊想法持续推进成可以运行、可以修改、可以管理代码并可以发布的软件项目。**

它不是单次网页生成器，也不是只有对话框的 AI 助手。用户面对的是一个由多智能体参与、以 Project 为中心的 AI 软件工作台：自然语言负责表达意图，专业 Agent 负责规划和实现，项目工作区负责预览、文件、代码、版本和发布。

```text
想法 / 资料 / 现有项目
          |
          v
     与 Lead 对话
          |
          v
   多智能体规划与执行
          |
          v
  可运行应用 + 项目代码
          |
    +-----+-------------------+
    |                         |
    v                         v
预览 / 视觉修改          查看 / 编辑 / 管理文件
    |                         |
    +-----------+-------------+
                v
          校验 / 修复 / 版本
                |
                v
          用户确认发布
                |
                v
         继续对话和迭代
```

产品最终交付的不是一次模型回答，而是一个**归属于用户、能继续开发的软件项目**。

## 2. 要解决的核心问题

- **[从想法到实现的门槛]** 用户可能只有目标，没有完整 PRD、架构或代码。系统需要帮助补全必要信息并直接形成可运行结果，而不是要求用户先学会传统软件开发流程。

- **[AI 过程不可理解]** 单次 Prompt 生成很难判断模型理解了什么、为什么这样实现、失败发生在哪一步。多智能体必须通过明确职责和可查看产物，让规划、实现和校验可以被理解。

- **[生成结果难以继续]** 一次性代码或截图不能自然承接后续修改。Project 必须保存代码文件、当前状态、历史版本、错误和发布状态，使用户可以在同一项目中继续对话和开发。

- **[用户失去代码控制]** Vibe Coding 不能等于把代码藏起来。用户需要查看、编辑、下载和管理自己的代码文件，并知道每次修改对应哪个版本。

- **[生成与上线混在一起]** AI 完成一次修改不代表用户同意上线。构建版本与线上版本必须分开，发布、更新和下线由用户明确决定。

## 3. 核心产品能力

### 3.1 多智能体协作

- **[单一入口]** 用户主要与 Lead 对话，不需要先理解内部角色、模式和流程。
- **[专业分工]** Product Manager、Architect、Engineer、Data Analyst 分别处理需求、结构、实现和验证；角色价值由交付结果证明，不靠头像或对话表演。
- **[按任务演进]** 简单任务应走短路径，复杂任务再调用更多角色、工具和返工；多智能体的目的不是增加 Agent 数量，而是降低不同类型的不确定性。
- **[用户介入]** 用户可以查看、修改和确认关键方案；系统只在范围、预算、破坏性操作和发布等真实风险处停下。

### 3.2 Vibe Coding 工作区

- **[自然语言开发]** 用户通过对话创建、解释、修改和修复项目，不要求先定位代码文件或写出实现步骤。
- **[实时预览]** 生成结果可以直接运行并在桌面、移动尺寸下检查，而不是以静态截图代表完成。
- **[视觉与代码双入口]** 普通用户可以选择界面元素修改内容和样式；需要精确控制时，可以查看和编辑项目代码文件。
- **[文件管理]** Project 展示源码、生成产物和媒体文件，支持查看、刷新、选择和后续扩展的增删改操作。
- **[针对当前对象继续对话]** 用户可以基于当前页面、元素、文件、错误或版本继续要求 Agent 修改，不必每次重新描述整个项目。

### 3.3 Project、代码与版本

- **[Project 是核心资产]** 一个 Project 对应一份持续存在的软件项目，而不是一段 Chat。需求、代码、Agent 产物、版本和发布状态都归入 Project。
- **[代码仓库归属]** 每个 Project 默认绑定代码仓库，用户对源码拥有可见、可编辑和可带走的控制权；平台内部版本必须能追溯到 Git commit。
- **[修改形成版本]** Agent Build、用户 Edit、代码保存、自动修复和 Restore 都形成新版本，不覆盖历史。
- **[恢复不改历史]** 恢复旧版本会创建新的当前版本；过去的版本和操作记录继续保留。
- **[发布指针独立]** 用户可以发布最新版本或指定版本；新草稿、新构建和 Restore 不应偷偷改变线上结果。

### 3.4 从构建到在线产品

- **[质量闭环]** 构建结果需要经过可解释的校验；错误应能定位、修复和重新验证，而不是只提示“生成失败”。
- **[公开访问]** 用户可以把确认过的版本发布为稳定链接，并在后续修改后明确 Update。
- **[后续能力]** 数据库、认证、支付、域名、分析、SEO、第三方连接和作品分享属于从“应用原型”走向“在线产品”的长期能力，但必须围绕同一个 Project 和发布版本工作。
- **[持续迭代]** 发布不是终点。用户可以回到项目继续对话、修改、验证并发布下一版本。

## 4. 与 Atoms 的关系

根据公开参考资料，Atoms 的产品主链路是：

`输入需求与资料 → Agent 编排 → 预览和编辑 → 数据与外部服务 → 校验和修复 → 版本与发布 → 分享和增长`

Another Atom 保留这条产品主干，但采用自己的品牌、交互、Contract 和工程实现，不复制 Atoms 的界面、私有 Prompt 或未公开基础设施。

| 维度 | 共同目标 | Another Atom 的明确取向 |
| --- | --- | --- |
| 产品形态 | 从意图到在线产品的 Agent 工作台 | 默认以单一 Lead 作为用户入口，内部组织方式由系统负责 |
| 多智能体 | 专业角色协作完成复杂任务 | 角色必须交付可查看产物，避免退化成多个 Agent 互相聊天 |
| Vibe Coding | 自然语言驱动创建和修改 | 对话、视觉编辑和代码文件编辑共同服务同一个 Project |
| 项目工作区 | Chat、Viewer、Edit 和文件联动 | Project 从创建开始拥有代码仓库、版本和发布状态 |
| 质量与恢复 | 错误可检查、修复，历史可恢复 | 校验结果、失败原因、版本和恢复都作为持久事实保存 |
| 发布 | 版本可以发布、更新和下线 | Agent 不能自动发布；线上指针由用户明确控制 |

## 5. 相对 Atoms 值得讨论和优化的点

以下内容是 Another Atom 的产品取舍，不代表 Atoms 没有对应内部实现；公开资料不足的地方不做推断。

### 5.1 一个 Lead，还是让用户先选模式和 Agent

- **当前倾向：** 默认只提供一个 Lead。Lead 判断是直接回答、简单修改还是组织团队，用户仍可查看实际调用的角色并覆盖关键判断。
- **优势：** 降低首次使用门槛，用户先表达目标，不需要先理解 Engineer、Team、Race 等内部模式。
- **代价：** 自动路由错误时可能让用户觉得失去控制，也会弱化高级用户主动选择执行成本和速度的能力。
- **待讨论：** 是否在高级设置中保留“快速实现 / 完整团队 / 多方案比较”等显式策略，但不把模式选择放在主入口之前。

### 5.2 固定审批，还是风险驱动确认

- **当前倾向：** 用户明确要求构建，已经授权一次受控工作；正常工作自动继续，只在范围适配、额外预算、破坏性代码操作和线上变化时确认。
- **优势：** 缩短 Golden Path，避免用户反复确认没有新增信息的步骤。
- **代价：** 用户失去每次构建前强制停下来编辑计划的窗口。
- **补偿方式：** 方案和中间产物始终可检查；用户可以暂停、修改、恢复或创建新版本。

### 5.3 代码仓库是后接能力，还是 Project 的默认组成

- **当前倾向：** 一个 Project 从创建开始就对应一个代码仓库；GitHub/GitLab 是未来可连接的 remote，而不是代码存在的前提。
- **优势：** 即使没有第三方授权，代码、版本和恢复也有稳定归属；平台版本能直接映射 commit。
- **代价：** 平台需要承担仓库持久化、隔离、备份和导出责任；接入 remote 后还需要处理双向同步和冲突。
- **待讨论：** 用户自有 remote 是自动镜像、手动 Push/Pull，还是以平台仓库为主、remote 只做导出。

### 5.4 Remix 解决上下文污染，还是在原 Project 内修复连续性

- **公开参考现象：** Atoms 建议连续多轮无法修复时，从当前版本 Remix 到新 Chat，以减少旧上下文干扰。
- **Another Atom 倾向：** 优先依赖结构化 Contract、版本和失败证据，在原 Project 中从明确基线继续；只有产品方向真正分叉时才 Remix。
- **优势：** 保留完整项目归属和版本历史，减少“为修复问题而复制项目”。
- **代价：** Runtime 必须正确处理 Context 失效、失败恢复和版本基线，工程要求更高。

### 5.5 先扩展功能广度，还是先证明项目闭环

- **当前倾向：** 先保证需求、生成、代码、校验、版本、发布和继续修改连成真实闭环，再增加 Cloud、支付、增长、作品广场和大量连接器。
- **优势：** 可以判断核心产品是否成立，不让外部服务数量掩盖生成和迭代体验的问题。
- **代价：** 短期功能丰富度会落后于 Atoms，能直接上线的应用类型也更受限制。
- **待讨论：** 闭环稳定后，优先补真实后端能力、代码协作，还是分享与 Remix 生态。

## 6. 版本如何服务整体目标

- **[V1：证明完整闭环]** 用受控 Web Runtime、固定专业团队、项目代码仓库、版本和显式发布，证明用户能从想法走到公开结果，并能继续修改。
- **[V2：提升协作能力]** 在不破坏 Project、代码归属和发布控制的前提下，引入动态任务图、角色子集、局部并行、Tool、返工和仲裁。
- **[后续：走向在线产品平台]** 按用户价值逐步增加后端、数据、第三方服务、协作、分享和增长能力；具体归属由后续版本文档决定。

V1、V2 是实现阶段，不是两个不同产品。任何版本都必须服务同一个长期闭环，不能为了展示新 Agent 或新技术而割裂 Project 历史、代码归属和用户控制。

## 7. 判断产品是否成立

- **[目标保持]** 最终结果是否保留用户原始产品目标和已确认范围。
- **[首次可用]** 用户能否在合理时间内从想法得到可交互结果，而不是停在计划或代码文本。
- **[修改成功]** 用户能否通过对话、视觉编辑或代码文件完成修改，并形成可恢复版本。
- **[过程可理解]** 系统暂停或失败时，用户是否知道当前阶段、原因、影响和下一步。
- **[代码可掌控]** 用户是否能查看、管理和带走项目代码，版本是否能追溯。
- **[发布可控制]** 线上版本是否只在用户明确操作后变化，历史版本能否恢复。
- **[项目可继续]** 用户再次进入项目时，是否能基于已有需求、代码和版本继续推进，而不是重新开始。

产品成功的最终标准不是生成了多少页面、调用了多少 Agent，而是：**用户是否真正得到并持续掌控了一个可以运行的软件项目。**
